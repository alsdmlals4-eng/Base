# UI 아트 감사 영역

각 영역은 독립적으로 감사하되 앞 영역의 변경이 뒤 영역에 미치는 영향을 기록한다. 패턴의 존재 자체가 결함은 아니다.

## A. 기계적 장식과 불필요한 효과

- 반복되는 글로우, 그림자, 유리 효과, 그라디언트가 정보 전달이나 세계관에 기여하는지 확인한다.
- 장식이 상호작용 상태와 경쟁하거나 모든 표면을 같은 중요도로 만드는지 확인한다.
- 의도적 장식은 무엇을 강조하고 어떤 상황에서 사라지는지 설명할 수 있어야 한다.

## B. 레이아웃·컨테이너·이미지 구조

- Godot의 `Container`와 anchor/offset, Web의 flow/grid/flex가 콘텐츠 변화와 해상도 변화에 견디는지 확인한다.
- 절대 좌표와 고정 크기는 필요한 연출인지, 임시 정렬인지 구분한다.
- 이미지의 역할, 자르기 방식, 비율, 대체 표현, 클릭 영역을 확인한다.

## C. 간격·밀도·리듬

- 요소 사이 간격이 관계를 드러내고 반복 가능한 간격 체계를 이루는지 확인한다.
- 작은 화면, 긴 번역 문자열, 입력 방식 변화에서 밀도가 무너지지 않는지 확인한다.
- 숫자 하나가 특이하다는 이유만으로 바꾸지 말고 시각 리듬과 구조상의 이유를 찾는다.

## D. 타이포그래피·정보 계층

- 제목, 본문, 레이블, 상태, 도움말이 크기·굵기·행간·위치로 구분되는지 확인한다.
- 과도한 자간, 기울임, 외곽선이 읽기를 방해하거나 장식처럼 반복되는지 확인한다.
- 한글, 영문, 숫자, 특수문자와 폰트 대체 상황을 실제 렌더로 확인한다.

## E. 색상·대비·상태 표현

- 색상이 브랜드와 분위기뿐 아니라 상태, 선택, 위험, 비활성, 포커스를 명확히 구분하는지 확인한다.
- 텍스트와 핵심 UI 구성요소의 대비를 실제 배경에서 확인한다.
- 색상만으로 상태를 전달하지 않는지, 키보드 포커스와 비활성 상태가 보이는지 확인한다.

WCAG 대비 기준은 판단의 하한 근거로 사용하되 게임 연출과 일시적 장식에는 적용 범위를 구분한다. 실패 판정은 실제 전경·배경 조합을 계산하거나 렌더에서 확인한 경우에만 내린다.
