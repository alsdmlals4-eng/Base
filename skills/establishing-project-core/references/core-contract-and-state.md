# 코어 확정 상세 계약

## 계약 필드

- 정체성: 한 문장 약속, 대상 플레이어와 플레이 상황, 차별 원리, 바뀌면 다른 프로젝트가 되는 경계.
- 핵심 경험: 반복 행동·판단·선택·즉시 피드백·다음 행동 동기·반드시 지켜야 할 감정과 판타지.
- 시스템: 코어 루프, 주요 시스템 입력·출력, 상태·가치 변화, 판매·활용·전투·성장 등 결과 사용처, 콘텐츠 연결 인터페이스.
- 기술: 반드시 보존할 데이터·호환성·공통 계약과 대체 가능한 구현을 분리.
- 보호 경계: `INVARIANT / CHANGEABLE / REQUIRES_REAPPROVAL / OUT_OF_SCOPE`.

## 상태와 승인 증거

```text
CORE_SEED
→ CORE_PROPOSED
→ CORE_STRESS_TESTED
→ CORE_CONFIRMED | CORE_REVISE | CORE_REJECTED
→ CORE_RECORDED
```

`CORE_CONFIRMED`에는 승인 주체, 승인한 계약 버전, 승인 시점, 수정·기각 항목과 이유가 필요하다. AI의 추론, 과거 문서, 현재 구현만으로 승인 상태를 만들지 않는다.

## 스트레스 테스트

- 제거·대체·극단적 축소 후에도 핵심 행동과 플레이어 약속이 남는가?
- 콘텐츠가 크게 늘어나도 중심 순환과 판단이 흐려지지 않는가?
- 기술 구현을 교체해도 제품 정체성과 데이터 계약이 유지되는가?
- 제작·기술·법적 제약과 PoC 반증에서도 성립하는가?
- 코어 기능, `MVP_SUPPORT`, 표현 외피가 서로 섞이지 않았는가?

## 기록·잠금

승인된 코어만 등록된 책임 원본, 개발 게이트, 적대적 검토 기준, Active Context의 현재 상태·금지 방향에 연결한다. 같은 내용을 여러 문서에 독립 원본으로 복제하지 않는다.

## 재개 조건

- 사용자가 프로젝트 방향 변경을 명시했다.
- PoC·플레이테스트가 핵심 가설을 반복 반증했다.
- 제작·기술·법적 제약으로 핵심 경험이 성립하지 않는다.
- 현재 구현과 코어가 장기간 충돌해 정체성이 이미 변했다.
- 기존 코어 문장끼리 모순된다.

재확정 전에는 기존 코어와 새 후보, 영향·마이그레이션·중단 기준을 분리한다.

## 실패 경계

사용자 승인 없이 `CORE_CONFIRMED` 또는 `CORE_RECORDED`로 표시하거나, 기능 목록 전체·장르명·홍보 문구·임시 코드 구조·UI·대사·스킨·분량을 불변 코어로 고정하거나, PoC 반증을 무시하거나, 일반 기능 변경으로 코어를 암묵적으로 바꾸면 실패다.
