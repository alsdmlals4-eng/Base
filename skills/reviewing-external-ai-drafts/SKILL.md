# Reviewing External AI Drafts

## Core principle

외부 산출물은 참고 입력이다. 실제 저장소, 승인된 기획, 테스트와 출처를 독립적으로 확인하기 전에는 정확성·완료·안전성을 인정하지 않는다.

## Required inputs

- 작업 패키지와 승인된 목표.
- 기준 브랜치·시작 커밋.
- 외부 AI 결과와 변경 diff.
- 읽기·쓰기 allowlist와 보호 경로.
- 완료 기준과 검증 방법.

## Review order

```text
출처와 범위
→ 현행 책임 원본
→ 사실·가정·제안 분리
→ diff와 참조 영향
→ 데이터·저장·호환성
→ 테스트·실행 결과
→ 최소 반영안
```

## Process

1. 결과가 요청 범위와 산출물 형식을 지켰는지 확인한다.
2. 새 파일을 만들기 전에 같은 책임의 현행 문서·코드가 있는지 찾는다.
3. 구현 사실, 외부 근거, 추정, 제안을 분리한다.
4. 삭제·이동·이름 변경은 모든 참조와 대체 경로를 확인한다.
5. 중복 문구는 짧게 만드는 것보다 책임 원본 하나로 통합하는 것을 우선한다.
6. 코드·데이터 변경은 실제 호출, 스키마, 저장, 오류 경로를 확인한다.
7. 대표 정상 사례, 경계 사례, 회귀 사례를 검증한다.
8. 외부 AI가 만든 설명보다 실제 diff와 명령 결과를 우선한다.
9. 승인할 부분, 수정할 부분, 폐기할 부분을 구분한다.
10. 실제 반영은 필요한 최소 변경으로 수행하고 다시 검증한다.

## Review dimensions

| 영역 | 확인 질문 |
|---|---|
| 의도 | 사용자가 해결하려는 문제가 유지되는가? |
| 범위 | 요청하지 않은 기능·리팩터링이 섞였는가? |
| 사실 | 문서와 보고가 실제 파일·실행 상태와 일치하는가? |
| 구조 | 새 책임 원본과 기존 원본이 중복되는가? |
| 데이터 | ID·스키마·저장 호환성이 유지되는가? |
| 품질 | 완료 기준이 관찰 가능하고 테스트됐는가? |
| 출처 | 외부 사실·벤치마킹·인용을 확인할 수 있는가? |
| 보안 | 비밀값·권한·외부 전송 위험이 없는가? |

## Output contract

```md
# 외부 AI 산출물 검수

## 승인 가능
## 수정 후 승인
## 폐기
## 확인한 기준 문서와 파일
## 실행한 검증
## 미검증·위험
## 실제 반영할 최소 diff
## worktree·브랜치 후속 처리
```

## Automatic rejection

- 실제 파일을 읽지 않고 완료를 주장한다.
- 출처 없는 최신 사실을 확정한다.
- 프로젝트 고유 규칙을 Base 공용 규칙으로 일반화한다.
- 기존 사용자 변경을 덮어쓴다.
- 테스트 없이 저장 형식·공개 인터페이스를 바꾼다.
- 가짜 경로·명령·함수·데이터 필드를 만든다.
- 생성형 이미지의 가짜 문자·로고를 최종 UI로 채택한다.

## Validation scenarios

1. 문서 초안이 기존 기준 문서와 70% 중복되면 새 파일을 승인하지 않고 기존 원본에 필요한 차이만 반영한다.
2. 코드 패치가 테스트를 추가했더라도 원래 증상을 재현하지 못하면 버그 수정 완료로 승인하지 않는다.
3. 아트 프롬프트 사례는 모델에서 한 번 성공한 결과만으로 보편적 스킬로 승격하지 않고 사례 또는 가설로 기록한다.
