# 게임 프로젝트 개발 게이트 방법

- 상태: 공용 방법
- 목적: 작업 단위의 준비·구현·검증·문서화·완료 판단과 제품 단계의 Greenlight를 분리해, 근거 없는 착수와 완료 선언을 막는다.
- 적용 대상: 기획, 코드, 데이터, UI·UX, 아트, 사운드, QA, 분석, 프로덕션과 통합검수 작업

## 1. 게이트의 두 축

게임 프로젝트에는 서로 다른 두 종류의 게이트가 필요하다.

1. **작업 실행 게이트**: 하나의 Issue·Goal·Plan·PR이 안전하게 시작되고 끝났는지 판단한다.
2. **제품 마일스톤 게이트**: 프로젝트가 Concept에서 Production·Alpha·Beta·Release Candidate로 넘어갈 준비가 됐는지 판단한다.

두 축을 섞지 않는다. 한 기능의 Definition of Done을 통과했다고 프로젝트 전체가 Vertical Slice를 통과한 것은 아니다.

**게이트(Gate)**: 다음 단계로 넘어가기 전에 완료 조건과 증거를 검토하는 승인 지점이다.

**Greenlight(그린라이트)**: 게이트 증거를 바탕으로 다음 제작 단계에 공식 진입하도록 승인하는 결정이다.

## 2. 작업 실행 게이트

```text
요청 접수
→ 컨텍스트·책임 원본 확인
→ 문제·목적·사용자 경험 정의
→ 상태와 범위 분류
→ Definition of Ready
→ 계획·승인
→ Implementation Gate
→ Verification Gate
→ Documentation Gate
→ Integration·Completion Gate
→ 인수인계·학습 환류
```

### 2.1 Intake·Context Gate

다음을 찾기 전에는 해결책을 확정하지 않는다.

- 최신 사용자 지시
- 프로젝트 `AGENTS.md`
- `BASE_RULES_VERSION.md`와 적용 Base 기준
- Documentation Map
- Active Context·Handoff
- 관련 분야 본책과 프로젝트 스킬
- Roadmap·Issue·Goal·Plan
- 실제 코드·데이터·자산·테스트
- `[백업]`, `[보류]`, `[미검증]`에 남은 관련 정보

출력:

```yaml
problem:
user_or_player_value:
primary_discipline:
affected_disciplines:
change_type:
current_truth_sources:
protected_decisions_and_paths:
unknowns:
```

### 2.2 Definition of Ready

**Definition of Ready(착수 준비 기준)**: 구현·제작을 시작하기 전에 충족해야 하는 공통 조건이다.

필수 조건:

- 해결할 문제와 목적이 한 문장으로 설명된다.
- 플레이어·사용자가 얻을 경험이 명확하다.
- 포함 범위와 제외 범위가 분리돼 있다.
- 주 책임 분야와 영향 분야가 지정돼 있다.
- 선행 조건과 의존성이 해결됐거나 명시돼 있다.
- 기존 기능·데이터·인터페이스·승인 자산의 보호 범위가 확인됐다.
- 저장·호환성·이관 위험이 기록돼 있다.
- 완료 기준이 `조건 → 행동 → 관찰 결과`로 작성돼 있다.
- 자동·수동·사용자 검수 방법이 있다.
- 적용할 Base 공용 스킬과 프로젝트 전용 스킬이 연결돼 있다.
- `[보류]` 항목을 재개하는 작업이면 재개 승인이 기록돼 있다.

준비되지 않은 작업은 구현자에게 넘기지 않고 조사·기획·결정 작업으로 분리한다.

### 2.3 Planning·Approval Gate

L2 이상 작업은 실제 저장소 조사 결과에 근거한 Plan을 가진다.

Plan 필수 항목:

- 현재 구조와 호출 흐름
- 변경할 파일과 변경하지 않을 파일
- 상태·데이터 소유자
- 이미지·사운드·UI 영향
- 마이그레이션과 호환성
- 실패·취소·폴백
- Acceptance Criteria
- 검증 순서
- 문서·스킬·Manifest 갱신 대상
- 롤백 방법

**Acceptance Criteria(인수 기준)**: 사용자가 결과를 승인할 수 있도록 관찰 가능한 조건으로 작성한 기준이다.

사용자의 구현 승인 전에는 제품 파일을 대규모로 수정하지 않는다. 명백한 오탈자와 복구 가능한 작은 링크 수정은 예외로 둘 수 있다.

### 2.4 Implementation Gate

- 승인 범위만 변경한다.
- 기능 추가와 대규모 리팩터링을 가능한 한 분리한다.
- 기존 저장 데이터, 공개 인터페이스와 사용자 흐름을 보호한다.
- `[보류]` 내용을 별도 재개 결정 없이 구현하지 않는다.
- 가장 작은 검증 가능한 변경 단위를 사용한다.
- 사용자 변경을 임의로 되돌리지 않는다.
- 이미지가 이미 승인돼 있으면 별도 지시 없이 대체 시안을 만들지 않는다.
- 코드·자산·문서 생성 사실을 검증 완료로 간주하지 않는다.

### 2.5 Verification Gate

최소 검증 순서:

```text
포맷·문법·정적 검사
→ 관련 자동 테스트
→ 핵심 정상 경로
→ 실패·취소·중복 입력·누락 데이터
→ 저장·불러오기·호환성
→ 실제 화면·플레이·오디오·성능
→ diff와 승인 범위 대조
→ 사용자 수동 검수 절차
```

실행하지 않은 검증은 `[미검증]`으로 기록한다. 도구가 없거나 환경 제약으로 실행하지 못한 항목을 통과로 표시하지 않는다.

### 2.6 Documentation Gate

방향, 수치, 용어, 기능, 구현 상태, 이미지, 사운드, 검증 또는 일정이 바뀌면 같은 작업에서 다음 영향을 확인한다.

- 분야별 활성 본책
- Roadmap와 마일스톤 게이트
- Documentation Map
- Active Context·Handoff
- Decision Log·Changelog
- 관련 프로젝트 스킬과 학습 기록
- Visual Source of Truth·Asset Manifest
- 테스트·검증 문서
- 사용자 열람 PDF와 Publication Manifest
- README·Issue·Goal·Plan
- Base 기준 버전과 후속 동기화

문서 전문을 여러 곳에 복사하지 않고, 각 책임 원본과 필요한 차이만 갱신한다.

### 2.7 Integration·Completion Gate

**Definition of Done(완료 기준)**: 작업 결과가 완료됐다고 판단하는 공통 조건이다.

- 승인 범위가 실제 파일에 반영됐다.
- Acceptance Criteria가 증거와 함께 판정됐다.
- 자동·수동 검증 결과가 기록됐다.
- 관련 본책·상태·Manifest·PDF 파생본이 최신이다.
- 미검증, 불일치와 남은 위험이 분리됐다.
- 새 작업자가 결과와 다음 작업을 찾을 수 있다.
- PR 자동 검사와 필수 리뷰가 통과했다.
- 제거·이동 파일의 잔여 참조가 없다.
- 프로젝트 고유 교훈과 공용화 가능한 교훈이 분리됐다.

완료 보고는 `구현 완료`, `검증 완료`, `사용자 확인 대기`를 구분한다.

## 3. 제품 마일스톤 게이트

| 단계 | 핵심 질문 | 대표 증거 | 다음 단계 승인 조건 |
|---|---|---|---|
| Concept | 누구에게 어떤 게임 약속을 제공하는가? | 방향서, 대상 플레이어, 금지 방향 | 핵심 경험과 문제 정의 승인 |
| Prototype | 핵심 재미·조작·기술 가설이 성립하는가? | 폐기 가능한 시제품, 관찰 기록 | 가설별 성공·실패 판정 가능 |
| Graybox | 공간·동선·규칙·가독성이 성립하는가? | 임시 자산 플레이, 규모 측정 | 핵심 경로와 전투·상호작용 검증 |
| First Playable | 핵심 루프를 처음부터 끝까지 완주할 수 있는가? | 플레이 가능한 빌드, 정상 경로 | 핵심 루프 단절 없음 |
| Vertical Slice | 최종 목표 품질과 제작 파이프라인을 좁은 범위에서 증명했는가? | 대표 플레이 구간, 목표 아트·사운드·UX·성능 | 품질 기준과 반복 제작 비용 검증 |
| Production | 검증된 방식으로 콘텐츠를 확장할 수 있는가? | 생산 계획, 자산·데이터 파이프라인 | 범위·일정·예산·인력 Greenlight |
| Alpha | 주요 기능이 연결돼 전체 진행을 검증할 수 있는가? | 전체 기능 경로, 통합 빌드 | 주요 기능 연결과 치명적 차단 해소 |
| Feature Complete | 계획된 주요 기능이 모두 존재하는가? | 기능 추적표 | 신규 핵심 기능 추가 종료 |
| Content Complete | 출시 범위 콘텐츠가 모두 제작됐는가? | 콘텐츠 추적표, 현지화·자산 상태 | 누락 콘텐츠 없음 |
| Beta | 품질·밸런스·최적화·호환성에 집중할 수 있는가? | 회귀·성능·호환성 리포트 | 출시 차단 버그 관리 가능 |
| Release Candidate | 치명적 문제가 없다면 출시할 수 있는가? | 후보 빌드, 릴리스 체크리스트 | 출시 게이트 통과 |

### 보조 잠금 지점

- **Content Lock**: 승인된 예외 외에는 신규 콘텐츠 추가를 중단하는 시점.
- **Code Freeze**: 승인된 출시 차단 버그 수정 외의 코드 변경을 제한하는 시점.

잠금은 생산 편의를 위한 규칙이며, 치명적 안전·데이터 손상 문제를 무시하는 근거가 아니다.

## 4. Vertical Slice 게이트 주석

**Vertical Slice(버티컬 슬라이스)**: 최종 게임의 핵심 시스템·아트·사운드·UX·성능과 제작 파이프라인을 좁은 범위에서 목표 품질로 증명하는 플레이 가능한 구간이다.

단순 기능 데모, Prototype 또는 최소 기능 모음과 구분한다.

필수 증거 예:

- 대표 핵심 루프가 처음부터 끝까지 완주됨
- 실제 목표 UI·아트·애니메이션·VFX·사운드 사용
- 실제 데이터·저장·실패·복귀 경로 연결
- 성능·가독성·접근성 기준 측정
- 콘텐츠 한 단위를 반복 생산하는 절차와 비용 확인
- 정상 경로뿐 아니라 선택적 하이라이트의 미보유·실패 경로도 완주 가능

세부 설계는 `skills/designing-vertical-slices/SKILL.md`와 `templates/planning/VERTICAL_SLICE_PLAN.md`를 따른다.

## 5. 게이트 기록 형식

```md
## [게이트명]
- 대상 단계·작업:
- 진입 조건:
- 종료 기준:
- Quality Bar:
- 필수 증거:
- 통과:
- 실패:
- 미검증:
- 승인자:
- 승인일·기준 커밋:
- 다음 단계:
- 남은 위험:
```

**Quality Bar(품질 기준선)**: 결과가 다음 단계로 넘어가기 위해 만족해야 하는 최소 품질 수준이다.

## 6. 실패 조건

- 체크리스트를 채웠다는 사실만으로 게이트 통과
- 문서 또는 PDF 존재를 구현·검증 증거로 사용
- Prototype 결과를 Vertical Slice 품질로 과장
- 기능 완료와 Feature Complete를 혼동
- 콘텐츠 제작 완료와 Content Complete를 추정
- 테스트하지 않은 플랫폼·해상도·저장 호환성을 통과 처리
- 미확정 예산과 일정을 임의 수치로 채움
- 게이트 실패 원인과 재진입 조건을 기록하지 않음
