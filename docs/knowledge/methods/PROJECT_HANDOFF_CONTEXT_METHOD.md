# 프로젝트 인수인계·컨텍스트 설계 방법

- 상태: 공용 방법
- 목적: 새 작업자나 AI가 대화 기록 없이도 프로젝트의 현재 방향, 책임 문서, 다음 행동과 금지 범위를 정확히 이해하도록 한다.

## 1. 인수인계 문서의 역할

인수인계 문서는 전체 기획서를 다시 복사하는 문서가 아니다. 다음 질문에 가장 빠르게 답하는 **라우터이자 실행 컨텍스트**다.

1. 이 프로젝트는 무엇을 만들고 있는가?
2. 지금 어디까지 결정·구현·검증되었는가?
3. 무엇이 절대 바뀌면 안 되는가?
4. 어떤 문서가 각 주제의 책임 원본인가?
5. 다음 작업은 무엇이며 어떤 순서로 진행하는가?
6. 아직 결정하지 않은 것은 무엇인가?
7. 구현 전에 무엇을 확인하고 승인받아야 하는가?

## 2. 문서 상태를 분리한다

모든 핵심 정보는 다음 상태 중 하나로 표시한다.

| 상태 | 의미 |
|---|---|
| 승인 | 구조와 방향이 결정됨 |
| PoC 가설 | 첫 구현에 사용할 수 있으나 검증 후 조정 가능 |
| 미확정 | 구현 전에 사용자 또는 책임자의 결정 필요 |
| 구현 | 실제 파일과 실행 결과로 존재함 |
| 검증 | 테스트·플레이·측정으로 기대 동작을 확인함 |
| 폐기 | 더 이상 사용하지 않으며 활성 문서에서 제거됨 |

`승인`과 `구현`, `구현`과 `검증`을 같은 의미로 쓰지 않는다.

## 3. 권장 문서 계층

```text
README
→ HANDOFF_CONTEXT 또는 ACTIVE_CONTEXT
→ DOCUMENTATION_MAP
→ 전체 기획서
→ 주제별 승인 책임 문서
→ ROADMAP
→ 현재 Issue / Goal / 승인 제안서
→ 실제 파일과 테스트
```

- `README`: 프로젝트 한 문장, 현재 상태, 최초 읽기 순서.
- `HANDOFF_CONTEXT`: 방향, 핵심 결정, 현재 단계, 다음 작업, 위험을 압축.
- `DOCUMENTATION_MAP`: 주제별 책임 원본 위치.
- 전체 기획서: 시스템 간 관계와 전체 경험.
- 승인 문서: 세부 규칙과 불변 조건.
- 로드맵: 시간 순서와 단계별 완료 기준.
- Issue/Goal: 현재 실행 범위.

## 4. 인수인계 컨텍스트 필수 구성

```md

## Current State
- 현재 단계:
- 구현 여부:
- 다음 승인 게이트:

## Product Promise
- 한 문장 소개:
- 핵심 사용자 경험:
- 핵심 루프:

## Non-negotiable Decisions
- 변경 시 사용자 승인 필요한 구조:

## Data and Ownership
- 책임 원본:
- 런타임 상태 소유:
- 중복 금지 데이터:

## Presentation Contract
- 화면 가독성:
- 아트·모션·오디오 핵심 원칙:

## Current Execution Order
1.
2.
3.

## Pending Decisions
- 구현 전 필수:
- 플레이테스트 뒤 조정:

- 작업 유형별 문서 경로:

## Do Not Do
- 폐기된 안:
- 범위 밖 작업:
```

## 5. 좋은 컨텍스트의 작성 규칙

- 현재 사실을 먼저 쓰고 역사적 배경은 링크로 보낸다.
- 같은 규칙의 전문을 여러 문서에 복제하지 않는다.
- 숫자는 단위, 적용 대상, 상태를 함께 적는다.
- 경로와 Issue 번호를 정확히 기록한다.
- “적절히”, “좋게”, “최적화” 대신 관찰 가능한 기준을 쓴다.
- 현재 구현이 없는 경우 `구현 전`이라고 명시한다.
- 최신 사용자 결정이 이전 문서와 충돌하면 책임 원본을 먼저 갱신한다.
- 외부 참고자료는 프로젝트 규칙을 덮어쓰지 않는다고 명시한다.

## 6. 작업 유형별 읽기 최소화

모든 작업자가 모든 문서를 읽게 하지 않는다.

| 작업 | 최소 문서 |
|---|---|
| 제품 방향 | Handoff, 전체 기획서, 관련 승인 문서 |
| 데이터 구조 | Handoff, 데이터 책임 문서, 기술 구조, 현재 Issue |
| UI·아트 | Handoff, UI/아트 책임 문서, 화면 기준, 실제 자산 |
| 애니메이션 | Handoff, 모션 계약, 전투 판정 데이터, 대표 스프라이트 |
| 밸런스 | 전체 기획서, 수치 책임 문서, 시뮬레이션 결과 |
| 구현 | AGENTS, Handoff, 현재 Issue/Goal, 승인 제안서, 실제 파일 |
| 검수 | 완료 기준, 테스트 체크리스트, 실제 실행 결과 |

## 7. 콜드 스타트 인수인계 테스트

문서를 작성한 사람의 설명 없이 새로운 작업자에게 다음 질문을 준다.

1. 프로젝트의 핵심 재미는 무엇인가?
2. 지금 코드를 수정해도 되는가?
3. 다음 작업은 무엇인가?
4. 가장 중요한 불변 조건 세 가지는 무엇인가?
5. 데이터 원본과 화면 자산은 어디에 있는가?
6. 미확정 수치와 승인된 구조를 구분할 수 있는가?
7. 폐기된 안을 다시 구현할 위험이 없는가?

10분 안에 정확히 답하지 못하면 읽기 순서, 책임 문서 링크, 상태 표기를 보강한다.

## 8. 흔한 실패

- 전체 기획서를 짧게 요약하면서 핵심 예외를 누락.
- 과거 계획과 현재 계획이 동시에 활성 문서에 존재.
- Issue에는 최신 정보가 있지만 로드맵과 GDD는 오래된 상태.
- 프로젝트 이름, 경로, 데이터 구조가 예전 명칭을 사용.
- 구현하지 않은 기능을 완료된 것처럼 기록.
- 외부 참고 저장소의 규칙을 프로젝트 승인보다 우선.
- 다음 작업이 “개발 진행”처럼 너무 넓게 적힘.

## 9. 완료 기준

- 한 개의 시작 문서에서 모든 책임 원본으로 이동할 수 있다.
- 최신 사용자 결정이 GDD, 로드맵, 활성 컨텍스트, Issue에 일관되게 반영된다.
- 승인·PoC·미확정·구현·검증 상태가 구분된다.
- 현재 실행 순서와 승인 게이트가 번호로 정리된다.
- 새 작업자가 콜드 스타트 질문을 10분 안에 답한다.
- 중복 문서와 폐기된 안이 활성 읽기 경로에 없다.
