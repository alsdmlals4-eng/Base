# 프로젝트 인수인계·컨텍스트 설계 방법

- 상태: 공용 방법
- 목적: 새 채팅, 새 AI, 새 작업자가 대화 기록이나 작성자의 추가 설명 없이도 프로젝트의 현재 방향, 책임 문서, 다음 행동과 금지 범위를 정확히 이해하도록 한다.

## 0. 지속성 계약

프로젝트 문서는 작업 결과를 설명하는 부록이 아니라 다음 작업을 가능하게 하는 실행 기반이다.

항상 최신 상태를 유지해야 하는 책임 원본:

- 프로젝트 방향과 핵심 경험을 설명하는 기획서
- 현재 단계·우선순위·선행 조건·종료 기준을 설명하는 로드맵
- 반복 작업의 입력·절차·산출물·검증을 설명하는 Base skill과 프로젝트 skill extension
- 질문별 현행 책임 원본을 연결하는 Documentation Map
- 현재 상태·최근 결정·다음 작업을 연결하는 Active Context 또는 Handoff

방향, 수치, 용어, 범위, 구현 상태, 파일 경로 또는 작업 절차가 바뀌면 같은 작업 안에서 영향받는 책임 원본을 갱신한다. 인수인계 시점에 한꺼번에 복구하려 하지 않는다.

기획서만 읽어도 다음을 이해할 수 있어야 한다.

1. 무엇을 만들고 있는가?
2. 플레이어에게 어떤 경험을 제공하려는가?
3. 현재 어떤 방향과 범위가 승인됐는가?
4. 무엇은 미확정·보류·금지인가?
5. 더 자세한 규칙과 실제 파일은 어디에서 확인하는가?

세부 구현을 기획서에 모두 복사하지 않는다. 기획서는 방향과 판단 기준을 책임지고, 정확한 데이터·파일·테스트는 명시된 참조 원본으로 연결한다.

## 1. 인수인계 문서의 역할

인수인계 문서는 전체 기획서를 다시 복사하는 문서가 아니다. 다음 질문에 가장 빠르게 답하는 **라우터이자 실행 컨텍스트**다.

1. 이 프로젝트는 무엇을 만들고 있는가?
2. 지금 어디까지 결정·구현·검증되었는가?
3. 무엇이 절대 바뀌면 안 되는가?
4. 어떤 문서가 각 주제의 책임 원본인가?
5. 다음 작업은 무엇이며 어떤 순서로 진행하는가?
6. 아직 결정하지 않은 것은 무엇인가?
7. 구현 전에 무엇을 확인하고 승인받아야 하는가?
8. 어떤 Base skill과 프로젝트 skill extension을 사용해야 하는가?
9. 이번 작업에서 Base로 환류해야 할 공용 학습 데이터는 무엇인가?

## 2. 문서 상태를 분리한다

모든 핵심 정보는 다음 상태 중 하나로 표시한다.

| 상태 | 의미 |
|---|---|
| 승인 | 구조와 방향이 결정됨 |
| PoC 가설 | 첫 구현에 사용할 수 있으나 검증 후 조정 가능 |
| 미확정 | 구현 전에 사용자 또는 책임자의 결정 필요 |
| 구현 | 실제 파일과 실행 결과로 존재함 |
| 검증 | 테스트·플레이·측정으로 기대 동작을 확인함 |
| 폐기 | 더 이상 사용하지 않으며 활성 문서에서 제거됨 |

`승인`과 `구현`, `구현`과 `검증`을 같은 의미로 쓰지 않는다.

## 3. 권장 문서 계층

```text
README
→ AGENTS
→ BASE_RULES_VERSION과 Base 로컬 사본
→ HANDOFF_CONTEXT 또는 ACTIVE_CONTEXT
→ DOCUMENTATION_MAP
→ 프로젝트 방향·전체 기획서
→ 주제별 승인 책임 문서
→ ROADMAP
→ 현재 Issue / Goal / 승인 제안서
→ 프로젝트 skill extension
→ 실제 파일과 테스트
```

- `README`: 프로젝트 한 문장, 현재 상태, 최초 읽기 순서.
- `AGENTS`: 프로젝트 최상위 규칙과 금지 범위.
- `BASE_RULES_VERSION`: 적용 중인 공용 학습 데이터 버전.
- `HANDOFF_CONTEXT`: 방향, 핵심 결정, 현재 단계, 다음 작업, 위험을 압축.
- `DOCUMENTATION_MAP`: 질문별 책임 원본 위치.
- 전체 기획서: 시스템 간 관계, 전체 경험, 방향과 범위.
- 승인 문서: 세부 규칙과 불변 조건.
- 로드맵: 시간 순서, 선행 조건, 우선순위와 단계별 완료 기준.
- Issue/Goal: 현재 실행 범위.
- 프로젝트 skill extension: Base skill을 이 프로젝트의 경로·데이터·검증에 연결.

## 5. 좋은 컨텍스트의 작성 규칙

- 현재 사실을 먼저 쓰고 역사적 배경은 링크로 보낸다.
- 같은 규칙의 전문을 여러 문서에 복제하지 않는다.
- 숫자는 단위, 적용 대상, 상태를 함께 적는다.
- 경로와 Issue 번호를 정확히 기록한다.
- “적절히”, “좋게”, “최적화” 대신 관찰 가능한 기준을 쓴다.
- 현재 구현이 없는 경우 `구현 전`이라고 명시한다.
- 최신 사용자 결정이 이전 문서와 충돌하면 책임 원본을 먼저 갱신한다.
- 외부 참고자료는 프로젝트 규칙을 덮어쓰지 않는다고 명시한다.
- 로드맵의 단계와 현재 Issue·Goal이 서로 연결되어야 한다.
- 프로젝트 skill extension은 실제 파일·데이터 경로와 프로젝트 전용 검증을 포함한다.
- 기획서는 작업 방향과 판단 기준을 책임지고, 실제 구현 상태를 추정하지 않는다.

## 6. 작업 중 갱신 규칙

다음 변경이 발생하면 작업 종료 전에 관련 책임 원본을 갱신한다.

| 변경 | 갱신 대상 |
|---|---|
| 프로젝트 방향·핵심 경험 | 방향서·전체 기획서·Active Context |
| 시스템 규칙·수치·용어 | 분야별 책임 기획서·데이터 문서·Issue |
| 우선순위·선행 조건·마일스톤 | Roadmap·Active Context |
| 반복 작업 절차·검증 | 프로젝트 skill extension·필요 시 Base skill |
| 구현·테스트 상태 | Active Context·Handoff·테스트 기록 |
| 책임 문서 경로 | Documentation Map·README·참조 문서 |
| 공용 교훈·실패 방지 | Base method·skill·template·case |

문서 최신화는 별도의 선택적 정리 작업이 아니라 완료 조건이다.

## 7. 작업 유형별 읽기 최소화

모든 작업자가 모든 문서를 읽게 하지 않는다. 대신 현재 작업에 적용되는 공용·전용 책임 원본을 모두 읽게 한다.

| 작업 | 최소 문서 |
|---|---|
| 제품 방향 | AGENTS, Handoff, 전체 기획서, Roadmap, 관련 승인 문서 |
| 데이터 구조 | AGENTS, Handoff, 데이터 책임 문서, 기술 구조, 현재 Issue |
| UI·아트 | AGENTS, Handoff, UI/아트 책임 문서, 관련 Base method·skill·case, 실제 자산 |
| 애니메이션 | Handoff, 모션 계약, 전투 판정 데이터, 대표 스프라이트 |
| 밸런스 | 전체 기획서, 수치 책임 문서, Roadmap, 시뮬레이션 결과 |
| 구현 | AGENTS, Base version·관련 skill, Handoff, 현재 Issue/Goal, 승인 제안서, 실제 파일 |
| 검수 | 완료 기준, 테스트 체크리스트, 실제 실행 결과, 관련 skill 실패 기준 |
| 인수인계 | 전체 책임 문서 지도, Roadmap, Active Context, 현재 Issue, Base 환류 상태 |

## 8. 콜드 스타트 인수인계 테스트

문서를 작성한 사람의 설명 없이 새로운 작업자 또는 새 채팅에 다음 질문을 준다.

1. 프로젝트의 핵심 재미와 플레이어 약속은 무엇인가?
2. 현재 단계와 가장 높은 우선순위는 무엇인가?
3. 지금 코드를 수정해도 되는가?
4. 다음 작업과 선행 조건은 무엇인가?
5. 가장 중요한 불변 조건 세 가지는 무엇인가?
6. 각 주제의 책임 기획서와 실제 데이터 원본은 어디에 있는가?
7. 미확정 수치와 승인된 구조를 구분할 수 있는가?
8. 폐기된 안을 다시 구현할 위험이 없는가?
9. 적용할 Base skill과 프로젝트 extension은 무엇인가?
10. 완료를 어떻게 검증하고 어떤 공용 학습 데이터를 환류해야 하는가?

10분 안에 정확히 답하지 못하면 읽기 순서, 책임 문서 링크, 상태 표기, 로드맵 또는 스킬 연결을 보강한다.

## 9. 흔한 실패

- 전체 기획서를 짧게 요약하면서 핵심 경험과 예외를 누락.
- 과거 계획과 현재 계획이 동시에 활성 문서에 존재.
- Issue에는 최신 정보가 있지만 로드맵과 GDD는 오래된 상태.
- 프로젝트 이름, 경로, 데이터 구조가 예전 명칭을 사용.
- 구현하지 않은 기능을 완료된 것처럼 기록.
- 외부 참고 저장소의 규칙을 프로젝트 승인보다 우선.
- 다음 작업이 “개발 진행”처럼 너무 넓게 적힘.
- 프로젝트 스킬이 실제 파일·데이터 구조와 연결되지 않음.
- Handoff에 모든 내용을 복사해 기획서와 충돌함.
- 작업 완료 후 공용 교훈과 실패 사례를 Base에 환류하지 않음.

## 10. 완료 기준

- 한 개의 시작 문서에서 모든 책임 원본으로 이동할 수 있다.
- 기획서만으로 핵심 방향, 플레이어 가치, 범위와 금지 방향을 이해할 수 있다.
- 최신 사용자 결정이 기획서, 로드맵, 활성 컨텍스트, Issue에 일관되게 반영된다.
- 승인·PoC·미확정·구현·검증 상태가 구분된다.
- 현재 실행 순서, 선행 조건과 승인 게이트가 번호로 정리된다.
- Base skill과 프로젝트 extension이 실제 작업 구조와 연결된다.
- 새 작업자가 콜드 스타트 질문을 10분 안에 답한다.
- 중복 문서와 폐기된 안이 활성 읽기 경로에 없다.
- 프로젝트 전용 기록과 Base 공용 학습 데이터·case가 구분된다.
