# AI 아트 프롬프트·디자인 기술 관리 방법

- 상태: 공용 방법
- 목적: 이미지 생성·편집 노하우를 일회성 긴 프롬프트가 아니라 재사용 가능한 디자인 기술, 프롬프트 패턴, QA 근거로 관리한다.

## 1. 기본 관점

아트 프롬프트는 결과 이미지 설명만이 아니라 다음 계약을 가져야 한다.

```text
사용 목적
→ 플레이어·사용자에게 먼저 전달할 정보
→ 유지할 정체성
→ 변경할 시각 축
→ 구도·정보 위계
→ 출력 규격
→ 실패 기준과 검증
```

같은 프롬프트가 한 번 성공했다는 이유만으로 공용 기술로 확정하지 않는다. 모델, 버전, 원본 이미지, 비율, 언어가 달라지면 결과가 달라질 수 있으므로 상태와 검증 조건을 기록한다.

## 2. 기술 상태

| 상태 | 의미 |
|---|---|
| `관찰` | 사례에서 가능성을 확인했지만 재현하지 않음 |
| `가설` | 적용 가치가 있어 시험할 프롬프트 패턴 |
| `패턴` | 같은 조건에서 반복 재현됨 |
| `검증` | 여러 자산·상황에서 QA를 통과함 |
| `제외` | 현재 모델·비용·권리·품질 조건에 부적합 |
| `폐기` | 더 안정적인 기술로 대체됨 |

## 3. 디자인 기술 카드 데이터

### 공통 필드

| 필드 | 책임 |
|---|---|
| `technique_id` | 프로젝트 내부 식별자. Base 사례에는 선택 사항 |
| `name` | 사람이 이해할 기술명 |
| `category` | UI, UX, 캐릭터, 표정, 포즈, 레이아웃, 생성, 편집, 후처리 |
| `goal` | 해결하는 문제 |
| `user_value` | 사용자·플레이어에게 주는 가치 |
| `use_when` | 적용 조건 |
| `avoid_when` | 사용하지 않을 조건 |
| `inputs` | 원본 이미지, 디자인 카드, 문구, 화면 규격 등 |
| `model_compatibility` | 확인한 모델·서비스·버전·날짜 |
| `prompt_pattern` | 재사용 가능한 기본 구조 |
| `prompt_cases` | 성공·실패·수정 사례 |
| `control_terms` | AU, 포즈, 카메라, 재질, 레이아웃 등 제어어 |
| `editable_points` | 안전하게 변경할 축 |
| `protected_elements` | 유지해야 할 정체성·구조 |
| `risks` | 흔한 실패와 비용 |
| `qa` | 관찰 가능한 검수 기준 |
| `evidence` | 실제 테스트·출처·비교 결과 |
| `scope` | Base 공용 원리와 프로젝트 전용 값 구분 |
| `status` | 관찰·가설·패턴·검증·제외·폐기 |

## 4. 프롬프트 모듈

권장 순서:

```text
# Goal
# Asset Context
# Preserve
# Change
# Composition
# Style and Material
# Information Layout
# Output
# Avoid
# QA
```

- 안정적으로 반복할 규칙과 예시는 앞부분에 둔다.
- 매 요청마다 바뀌는 캐릭터명·색·표정·문구는 뒤쪽 변수로 분리한다.
- 프롬프트 예시는 하나의 좋은 사례와 대표 실패 수정 사례를 우선한다.
- 모델이 구조화 출력을 지원하면 기술 카드 메타데이터를 JSON 또는 고정 Markdown으로 회수한다.

## 5. 표정 제어

FACS Action Unit은 표정의 관찰 가능한 움직임을 설명하는 어휘다. 이미지 모델의 공통 공식 명령 체계가 아니다.

권장 순서:

```text
감정 목적
→ 자연어 표정 설명
→ 좌우·강도·시선·머리 방향
→ FACS AU 보조 표기
→ 유지 요소와 오류 방지
```

예시:

```text
원본 캐릭터의 얼굴 비율, 안경, 앞머리, 의상과 채색을 유지한다.
오른쪽 눈만 자연스럽게 윙크하고 입꼬리는 약하게 올린다.
보조 표기: AU46R, AU12B.
닫힌 눈의 속눈썹과 안경테가 겹치지 않게 한다.
```

- 단독 AU 명령은 빠른 탐색용으로 사용할 수 있다.
- 좌우 지정과 강도 A~E 표기는 모델이 이해하지 못할 수 있으므로 자연어를 함께 쓴다.
- 레퍼런스 그리드의 비표준 번호는 모델별 별칭으로 분리한다.

## 6. 캐릭터 프로모션 포스터

포스터는 일러스트와 정보 구조를 함께 설계한다.

```text
메인 캐릭터
+ 세계·배경
+ 이름·엠블럼
+ 특징 설명 2~3개
+ 세로·가로 카피
+ 표정·측면 인셋
+ 타이틀·날짜·하단 정보
```

제작 원칙:

- 캐릭터 실루엣과 얼굴을 정보 모듈이 가리지 않는다.
- 키 컬러, 이름, 특징, 인셋을 변수화해 다른 캐릭터에 재사용한다.
- 이미지 생성 단계의 문자와 로고는 배치 시안으로 취급한다.
- 최종 타이포그래피는 편집 가능한 레이어 또는 UI로 분리한다.
- 인셋 표정은 캐릭터의 실제 감정·상태 키와 연결한다.
- 세로 포스터, 캐릭터 상세 페이지, 이벤트 배너로 재배치할 수 있는 슬롯 구조를 유지한다.

## 7. Base와 프로젝트 경계

Base:

- 기술 카드 스키마.
- 표정·포스터 프롬프트 구조.
- 실패 기준과 검증법.
- 일반화된 사례와 비표준 코드 경고.

프로젝트:

- 실제 캐릭터 외형·이름·세계관·키 컬러.
- 사용하는 모델·계정·비용·출력 경로.
- 승인된 프롬프트와 원본 이미지.
- 프로젝트 전용 AU 조합·포스터 문구·화면 규격.
- 실제 QA 결과와 자산 manifest.

## 8. 검증 순서

```text
원본 정체성 비교
→ 변경 축만 확인
→ 얼굴·손·소품 오류
→ 실제 크롭과 축소
→ UI·텍스트 레이어 결합
→ 상태·캐릭터 간 일관성
→ 현지화·접근성
→ 재생성 비용과 재현성
```

## 9. 완료 기준

- 기술의 목적, 사용 조건, 비사용 조건이 명확하다.
- 프롬프트에서 유지 요소와 변경 요소가 분리된다.
- 모델 호환성과 검증 상태가 기록된다.
- 실제 화면과 후처리 경로가 정의된다.
- 한 번의 성공 결과를 보편적 사실로 표현하지 않는다.
- 프로젝트 전용 값 없이도 다른 프로젝트가 같은 검증 절차를 적용할 수 있다.
