# 분야별 프로젝트 스킬 학습·진화 방법

- 상태: 공용 방법
- 목적: 공용 작업 규칙은 foundation에 한 번만 두고, 각 제작 분야는 독립 실행 가능한 프로젝트 스킬을 통해 반복 작업의 품질과 인수인계를 개선한다.
- 적용 대상: AI 협업 또는 반복 가능한 제작 절차를 가진 게임 프로젝트

## 1. 핵심 원칙

스킬은 조언 모음이 아니라 다음을 고정하는 **반복 실행 계약**이다.

```text
사용 조건
→ 사용하지 않는 조건
→ 필수 입력
→ 먼저 읽을 책임 원본
→ 작업 절차
→ 산출물
→ 완료 기준
→ 검증
→ 실패 조건
→ 학습·갱신
```

프로젝트의 세계관, 실제 수치, 기능명, 파일 경로와 승인 자산은 프로젝트 스킬이 책임한다. 여러 프로젝트에 재사용 가능한 판단법과 검증 원리만 Base로 환류한다.

**항상 학습한다**는 뜻은 스킬 본문을 매 작업마다 무조건 수정한다는 뜻이 아니다.

```text
스킬 호출
→ 결과·실패·예외 기록
→ 변경 필요성 판정
→ 필요할 때만 스킬 계약 갱신
→ 반복 검증
```

모든 의미 있는 호출은 Learning Log에 남긴다. 변경할 근거가 없으면 `변경 없음`과 이유를 기록한다.

## 2. 스킬 계층

### 2.1 Foundation·공용 스킬

여러 분야에서 반복 사용하는 절차는 `skills/foundation/` 또는 프로젝트가 정한 공용 영역에 둔다.

권장 책임:

- 요청 접수와 작업 단계 분류
- 컨텍스트·책임 원본 라우팅
- 영향도 분석
- 개발 게이트 판정
- Decision Record 작성
- 문서 수명주기와 Documentation Map 갱신
- 검증·완료 선언
- PDF·파생본 발행
- 컨텍스트 압축과 Handoff
- 외부 AI 초안·diff 검수
- 프로젝트 교훈 분리와 Base 환류 판단

공용 절차를 각 분야 스킬에 장문 복사하지 않는다. 분야 스킬은 foundation 스킬 경로와 해당 작업에 필요한 차이만 기록한다.

### 2.2 분야별 프로젝트 스킬

각 분야는 독립 진입 가능한 스킬을 가진다. 프로젝트 규모에 따라 여러 사람이 같은 스킬을 사용하거나 한 사람이 여러 스킬을 담당할 수 있다.

| 분야 | 권장 프로젝트 스킬 책임 |
|---|---|
| 설정·내러티브 | 정사·용어·시나리오·캐릭터 관계·대사 일관성 |
| 게임 디자인 | 핵심 루프·전투·경제·진행·밸런스·시스템 상호작용 |
| UX·UI·접근성 | 정보 구조·화면 흐름·입력·피드백·온보딩·접근성 |
| 개발·엔지니어링 | 아키텍처·상태 소유·데이터·AI·저장·성능·빌드 |
| 테크니컬 아트·파이프라인 | 자산 규격·Import·피벗·애니메이션 계약·도구·예산 |
| 아트 | 시각 언어·캐릭터·환경·건물·UI 그래픽·VFX·애니메이션 |
| 사운드 | BGM·SFX·음성·이벤트 연결·믹싱·반복 방지 |
| QA | 자동·수동·회귀·시각·오디오·성능·호환성 검증 |
| 프로덕션·PM | 범위·일정·의존성·위험·예산 상태·마일스톤 |
| 분석·유저리서치 | 벤치마킹·SWOT·플레이테스트·텔레메트리·가설 판정 |
| 통합검수 | 문서·구현·자산·검증·일정·스킬·PDF 일치성과 게이트 판정 |

분야 안에서 반복 빈도와 품질 기준이 크게 다른 작업은 하위 스킬로 분리할 수 있다. 예: 게임 디자인의 `밸런스 조정`, 아트의 `캐릭터 제작`, UX의 `플레이테스트 설계`.

## 3. 선택적 호출 구조

프로젝트는 사람용 `PROJECT_SKILL_MAP.md`와 기계 판독용 `SKILL_REGISTRY.json`을 함께 둔다.

- `PROJECT_SKILL_MAP.md`: 분야 책임, 관계, 설명, 콜드 스타트 안내
- `SKILL_REGISTRY.json`: 경로, 상태, trigger tags, 사용·비사용 조건, Learning Log와 검토 기준

기본 정책:

```json
{
  "load_all_skills": false,
  "default_selection": "none",
  "require_trigger_match": true,
  "max_primary_discipline_skills": 1,
  "max_foundation_skills": 3
}
```

작업 시작 시 `routing-project-work-by-discipline`로 다음을 판정한다.

```text
요청·변경 유형
→ 주 책임 분야 1개
→ 영향 분야
→ 필요한 foundation 스킬
→ 필요한 분야 진입 스킬
→ 후속 게이트에서만 호출할 스킬
```

다음은 금지한다.

- 전체 skills 폴더 기본 로드
- trigger와 무관한 관성적 호출
- 같은 책임의 중복 스킬 동시 호출
- 보류·백업·제거 후보 스킬 호출
- 후속 검증·발행·인수인계 스킬을 작업 시작부터 모두 호출

## 4. 프로젝트 스킬 최소 계약

각 스킬에는 다음 항목이 있어야 한다.

```yaml
name:
skill_id:
discipline:
purpose:
use_when:
do_not_use_when:
trigger_tags:
load_by_default: false
required_inputs:
read_first:
foundation_dependencies:
project_rules:
process:
outputs:
definition_of_ready:
definition_of_done:
validation:
failure_conditions:
related_skills:
learning_log:
review_triggers:
last_reviewed_at:
last_reviewed_commit:
knowledge_state:
```

### 필수 해석

- `use_when`: 호출해야 하는 구체적 조건이다.
- `do_not_use_when`: 비슷해 보이지만 호출하지 않을 조건이다.
- `trigger_tags`: Skill Registry가 선택에 사용하는 작업 유형 태그다.
- `load_by_default`: 현행 스킬도 기본값은 `false`다.
- `read_first`: 프로젝트 분야 본책, 실제 파일과 검증 경로를 연결한다.
- `foundation_dependencies`: 공용 규칙을 복사하지 않고 참조한다.
- `project_rules`: 프로젝트 고유 명칭·수치·경로·승인 자산을 둔다.
- `validation`: 존재하는 실제 명령과 수동 검수만 기록한다.
- `review_triggers`: 어떤 변경이 스킬 재검토를 요구하는지 적는다.

## 5. 스킬 학습 순환

```text
스킬 적용
→ 실제 작업
→ 결과와 실패 검증
→ Learning Log 기록
→ 프로젝트 고유 정보와 재사용 원리 분리
→ 스킬 변경 필요성 판정
→ 필요한 경우 프로젝트 스킬 갱신
→ 반복 검증
→ 패턴 판정
→ Base 환류 후보 검토
```

학습 상태:

| 상태 | 의미 | 사용 방식 |
|---|---|---|
| 관찰 | 한 번 발견된 현상 | 강제 규칙으로 사용하지 않음 |
| 가설 | 해결 가능성이 있으나 추가 검증 필요 | 적용 조건과 실패 기준 기록 |
| 패턴 | 여러 작업에서 반복된 경향 | 프로젝트 스킬 권장 절차로 사용 가능 |
| 검증 | 여러 조건에서 재현된 안정적 방법 | 프로젝트 공통 규칙으로 채택 가능 |
| 승격 후보 | 여러 프로젝트에 재사용 가치가 있음 | Base Method·Skill·Template·Test 검토 |

한 번 성공한 방법을 즉시 공용 강제 규칙으로 만들지 않는다.

## 6. 항상 기록할 실행 결과

스킬 사용 후 최소 기록:

```md
## 실행 기록
- 작업·PR:
- 적용 스킬 ID·기준 커밋:
- 호출 트리거:
- 입력과 범위:
- 실제 산출물:
- 실행한 검증:
- 결과: 성공/부분 성공/실패/미검증
- 성공한 조건:
- 실패·예외·재현 조건:
- 사용자 피드백:
- 불필요하게 호출한 스킬:
- 누락된 스킬·검증:
- 스킬 변경 필요: 예/아니오
- 변경하지 않는 이유:
- 지식 상태: 관찰/가설/패턴/검증/승격 후보
- 프로젝트 전용으로 유지할 내용:
- Base 환류 후보:
- 다음 검토 트리거:
```

실패 로그는 삭제하지 않는다. 장문 도구 로그 전체 대신 재현 조건, 원인, 수정한 계약과 재발 방지 기준을 보존한다.

## 7. 자동·수동 검수

자동 검수 적합:

- Skill Registry JSON 문법
- 중복 skill ID
- 활성 스킬 경로 존재
- 사용·비사용 조건과 trigger tags 존재
- `load_by_default=false`
- Learning Log 경로 존재
- 분야 진입 스킬 등록
- 보류·백업 상태의 기본 호출 제외
- `[기획서]`가 저장소 루트에 존재

수동·통합검수 필요:

- 스킬 책임이 실제 작업과 맞는지
- 호출 스킬 집합이 과도하지 않은지
- 학습 기록이 의미 있는 개선으로 이어졌는지
- 분야 스킬과 foundation 책임이 중복되지 않는지
- 프로젝트 고유 정보가 Base에 섞이지 않았는지

## 8. 검토 주기와 트리거

다음 중 하나가 발생하면 스킬을 검토한다.

- 동일 실패가 반복됨
- 사용자 피드백이 기존 완료 기준과 다름
- 실제 파일·도구·경로가 변경됨
- 새 예외·폴백이 필요함
- 검증 명령·Quality Bar가 변경됨
- 여러 분야에 같은 절차가 복제됨
- 새 작업 유형에 대응하는 스킬이 없음
- 새 채팅이 필요한 스킬을 찾지 못함
- 90일 이상 사용됐지만 검토 기록이 없음

정기 검토는 `verifying-game-project-operating-system` 또는 `evolving-project-discipline-skills`를 사용한다.

## 9. 스킬 통합·분리 기준

### 통합할 때

- 입력·절차·산출물·검증이 대부분 같다.
- 한 스킬이 다른 스킬의 얇은 별칭에 불과하다.
- 같은 공용 절차가 여러 파일에 반복된다.

공통 부분을 foundation으로 올리고 분야 차이만 남긴다.

### 분리할 때

- 품질 기준과 실패 조건이 다르다.
- 별도 책임 원본·도구·검증 경로가 있다.
- 변경 빈도나 담당자가 다르다.
- 하나의 스킬이 너무 많은 종류의 산출물을 책임해 콜드 스타트가 느려진다.

## 10. 기본 컨텍스트 효율

- 작업 시작 시 전체 스킬 저장소를 읽지 않는다.
- `SKILL_REGISTRY.json`에서 trigger가 일치하는 후보를 찾고 `PROJECT_SKILL_MAP.md`로 책임을 확인한다.
- `[백업]`, `[보류]` 스킬과 오래된 실행 로그는 기본 읽기에서 제외한다.
- 스킬 본문에는 현재 실행에 필요한 계약만 두고 긴 사례는 case·Learning Log로 연결한다.
- 동일 프로젝트 소개를 모든 스킬에 복사하지 않고 루트 `[기획서]/00_프로젝트_허브/START_HERE.md`와 분야 본책을 참조한다.

## 11. Base 환류 경계

Base로 환류 가능:

- 반복 검증된 판단 순서
- 프로젝트에 독립적인 실패 방지 규칙
- 재사용 가능한 체크리스트·템플릿·회귀 테스트
- 도구에 종속되지 않는 입력·출력·검증 계약
- 적용하지 말아야 할 조건

프로젝트에 유지:

- 세계관·캐릭터·명칭
- 게임 규칙과 실제 수치
- 파일·데이터·자산 경로
- 승인 이미지와 프롬프트 원본
- 구현·테스트 현황
- 특정 팀·계정·예산·일정

## 12. 완료 검수 질문

- 각 분야에서 반복 작업을 시작할 현행 진입 스킬이 있는가?
- `SKILL_REGISTRY.json`과 `PROJECT_SKILL_MAP.md`가 일치하는가?
- 스킬이 실제 본책·파일·테스트를 가리키는가?
- 사용·비사용 조건으로 필요한 경우에만 호출되는가?
- 공용 절차가 분야 스킬마다 복제되지 않았는가?
- 모든 의미 있는 호출에 Learning Log 기록이 있는가?
- 변경 근거가 없을 때 불필요하게 스킬 본문을 수정하지 않았는가?
- 프로젝트 고유 내용이 Base 공용 스킬에 섞이지 않았는가?
- 미검증 가설이 강제 규칙으로 승격되지 않았는가?
- 실패가 다음 실행 계약이나 회귀 테스트를 개선했는가?
- 새 AI가 필요한 스킬만 2분 안에 찾을 수 있는가?
