# 분야별 기획서 PDF 발행 방법

- 상태: 공용 방법
- 목적: 분야별 Markdown 책임 원본과 승인된 이미지·실제 캡처를 사용해, 해당 분야의 목적부터 현재 상태·전체 작업 과정·검증·다음 작업까지 한눈에 볼 수 있는 최신 PDF 통합본을 만든다.
- 적용 대상: 사용자·리드·외부 협업자가 읽는 분야별 기획서 PDF, 고정 배포본과 검토 패키지

## 1. PDF의 역할

PDF는 단순 요약본이 아니다. 해당 분야의 활성 본책과 필요한 부록을 읽기 좋은 순서로 조합한 **배포용 최신 통합본**이다.

- Markdown·데이터·Manifest가 편집 가능한 책임 원본이다.
- PDF는 책임 원본에서 생성되는 읽기 전용 파생본이다.
- PDF를 수동으로 따로 수정하지 않는다.
- PDF만 최신이고 원본 책임 문서가 오래된 상태를 만들지 않는다.
- PDF 존재를 구현·검증 완료 증거로 사용하지 않는다.

## 2. 필수 내용 계약

각 분야 PDF는 필요한 범위에서 다음 전체 흐름을 포함한다.

```text
분야 목적과 프로젝트 내 역할
→ 핵심 사용자·플레이어 가치
→ 승인된 방향과 금지 방향
→ 현재 개발 단계와 다음 게이트
→ 분야 전체 작업 흐름
→ 단계별 입력·판단·산출물
→ 분야 책임 문서와 프로젝트 스킬
→ 실제 코드·데이터·자산 경로
→ 승인된 이미지·UI·도표·다이어그램
→ 현재 구현·제작 상태
→ 검증 상태와 실패 조건
→ 위험·미확정·보류
→ 다음 작업과 완료 기준
→ 부록·출처·기준 커밋
```

공통 운영 규칙 전문을 모든 분야 PDF에 반복하지 않는다. 개발 게이트·공용 Workflow는 짧은 요약과 공용 원본 경로를 제공하고, 분야별 차이와 적용 결과를 상세히 보여준다.

## 3. 첫 화면·가독성 구조

권장 순서:

1. 표지: 프로젝트, 분야, 문서 역할, 기준 커밋·빌드, 생성일
2. 한눈에 보기: 현재 단계, 상태, 다음 게이트, 핵심 위험, 다음 작업
3. 목차
4. 분야 목적·Quality Bar·금지 방향
5. 전체 프로세스 다이어그램
6. 승인 결정과 책임 경계
7. 상세 기획·제작 흐름
8. 승인 이미지·실제 캡처 비교
9. 구현·검증·미검증 상태
10. Roadmap·Issue·스킬 연결
11. 보류·위험·다음 작업
12. 부록 색인·출처·변경 이력

각 장은 `왜 → 무엇 → 어떻게 → 현재 상태 → 검증 → 다음 작업` 순서로 읽히게 한다.

## 4. 이미지 포함 규칙

PDF에는 현재 설명과 관련된 승인 자료만 포함한다.

- `DIRECTION_APPROVED`, `PRODUCTION_READY`, `IMPLEMENTED`, `VISUALLY_VALIDATED` 상태를 구분한다.
- `REFERENCE`, `PARTIAL_REFERENCE`, `SUPERSEDED`, `MIGRATION_PENDING`은 역할과 제한을 캡션에 표시한다.
- 폐기된 시안을 현행 기준처럼 배치하지 않는다.
- 콘셉트 이미지와 실제 게임 캡처를 나란히 비교할 때 각각의 상태를 표시한다.
- 이미지 한 장 전체가 승인된 것이 아니라면 채택 요소와 비채택 요소를 캡션에 기록한다.
- 원본·캐노니컬 경로와 Asset ID를 캡션 또는 부록에 연결한다.
- 이미지 안의 임시 수치·문구를 승인 기획값으로 해석하지 않는다.
- 기존 승인 이미지가 있으면 별도 지시 없이 새 이미지를 생성하지 않는다.

권장 캡션:

```text
[Asset ID] 자료명 — 상태 / 용도
채택: 실루엣, 정보 계층
비채택: 임시 수치, 레거시 문구
원본: images/canonical/...
실제 캡처: images/captures/...
```

## 5. 접근성과 품질

- 제목 계층과 목차 북마크를 유지한다.
- 본문과 캡션은 확대 없이 읽을 수 있는 크기를 사용한다.
- 색만으로 상태를 구분하지 않고 문자 라벨을 함께 쓴다.
- 표가 페이지 밖으로 잘리지 않게 한다.
- 이미지에 설명 캡션과 대체 설명 원문을 둔다.
- 링크는 PDF에서도 클릭 가능하거나 경로를 읽을 수 있게 한다.
- 페이지 번호와 문서 기준 커밋을 표시한다.
- 가능한 경우 글꼴을 포함하거나 대체 글꼴에서 한글 깨짐을 검사한다.
- PDF 내보내기 후 첫 페이지, 목차, 표, 이미지, 링크와 마지막 페이지를 시각 검수한다.

## 6. Publication Manifest

각 PDF는 `PUBLICATION_MANIFEST.json`에서 원본과 생성 상태를 추적한다.

권장 필드:

```json
{
  "publication_id": "game-design-bible",
  "discipline": "게임 디자인",
  "role": "read_only_derivative",
  "source_files": ["[기획서]/02_게임_디자인/게임_기획서.md"],
  "approved_image_paths": ["images/canonical/example.webp"],
  "output_pdf": "[기획서]/02_게임_디자인/게임_기획서.pdf",
  "source_commit": "",
  "content_sha256": "",
  "generated_at": "",
  "generator": "",
  "status": "NOT_BUILT"
}
```

상태 예:

- `NOT_BUILT`: 생성 파이프라인 미설치 또는 아직 생성하지 않음
- `CURRENT`: 현재 원본·이미지 해시와 일치
- `STALE`: 원본 또는 승인 이미지가 변경됨
- `FAILED`: 생성·검수 실패
- `MISSING_ASSET`: Manifest의 이미지가 없음

## 7. 최신성 검증

PDF를 최신이라고 판단하려면 다음이 모두 필요하다.

- 모든 source file과 승인 이미지가 존재한다.
- PDF가 존재하고 실제 PDF 파일이다.
- Manifest의 source commit 또는 content hash가 현재 입력과 일치한다.
- 원본 Markdown·승인 이미지 변경 PR에서 PDF와 Manifest가 함께 갱신됐다.
- 생성 후 링크·이미지·페이지 잘림을 검수했다.
- PDF의 상태·기준 커밋이 본책과 일치한다.

파일 수정 시간만으로 최신성을 판정하지 않는다. checkout·CI에서 시간 정보가 바뀔 수 있으므로 입력 해시와 Git diff를 사용한다.

## 8. 생성 파이프라인

프로젝트 도구에 맞는 하나의 재현 가능한 명령을 둔다.

예:

```text
python tools/build_discipline_publications.py --all
```

가능한 구현:

- Pandoc + HTML/CSS 또는 LaTeX
- Quarto
- MkDocs·HTML → PDF
- WeasyPrint
- 프로젝트 전용 Python·Node 빌더

도구 선택보다 중요한 조건:

- 같은 입력에서 같은 구조의 결과 생성
- Markdown·이미지 경로를 직접 사용
- 빌드 실패 시 기존 PDF를 최신으로 표시하지 않음
- 생성 로그와 Manifest 갱신
- CI 또는 로컬 검수 명령 존재

DOCX가 필요한 경우에도 Markdown과 Manifest에서 생성하고 독립 원본으로 수동 편집하지 않는다.

## 9. 분야별 포함 예시

| 분야 | PDF에서 특히 보여줄 전체 과정 |
|---|---|
| 설정·내러티브 | 정사 원칙 → 시나리오 → 캐릭터·대사 → 구현·현지화 → 검수 |
| 게임 디자인 | 약속 → 핵심 루프 → 시스템·밸런스 → 데이터·UI 계약 → 플레이 검증 |
| UX·UI | 사용자 과업 → 정보 구조 → 와이어프레임 → 시각 적용 → 사용성·접근성 QA |
| 개발 | 요구 → 아키텍처 → Scene·데이터 → 구현 → 테스트·성능·빌드 |
| 테크니컬 아트 | 원본 제작 → 규격 → Import → 엔진 적용 → 성능·시각 QA |
| 아트 | 아트 방향 → 자산 계열 → 제작 규격 → 승인 이미지 → 실제 캡처 비교 |
| 사운드 | 감정·정보 목적 → 음원 제작 → 이벤트 연결 → 믹싱 → 반복·접근성 QA |
| QA | 위험 → 테스트 설계 → 실행 → 버그 등급 → 회귀·릴리스 판정 |
| 프로덕션 | 범위 → 일정·의존성 → 마일스톤 → 위험·예산 상태 → Greenlight |
| 분석 | 질문 → 근거 → 조사·계측 → 해석 → 채택·제외 결정 |
| 통합검수 | 요구 → 책임 원본 → 구현·자산 → 검증 → 불일치 → 게이트 판정 |

## 10. 실패 조건

- PDF가 본책보다 오래됐지만 최신으로 표시됨
- 요약만 있고 분야의 전체 작업 흐름·현재 상태·다음 작업이 없음
- 승인 이미지가 빠지거나 폐기 이미지가 현행 기준처럼 포함됨
- 이미지 캡션에 상태·채택 범위·원본 경로가 없음
- PDF와 Markdown을 둘 다 독립 원본으로 수정
- 공용 설명을 모든 분야 PDF에 장문 복제
- 실제 구현·검증하지 않은 내용을 완료처럼 표현
- 생성 실패 후 이전 PDF를 그대로 배포하면서 기준 커밋을 갱신함
