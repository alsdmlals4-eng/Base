# 게임 애니메이션·전투 연출 설계 방법

- 상태: 공용 방법
- 목적: 애니메이션을 장식이 아니라 상태·판정·역할을 전달하는 인터페이스로 설계한다.

## 1. 연출 우선순위

```text
판정과 상태의 명확성
→ 역할과 무게 전달
→ 타격감과 보상감
→ 화려함
```

모션이 멋있어도 공격 시점, 이동 방향, 위험 범위가 읽히지 않으면 실패다.

## 2. 먼저 확인할 입력

- 카메라 거리와 실제 유닛 화면 크기.
- 최대 동시 유닛 수와 겹침 정도.
- 공격속도·이동속도 변화 범위.
- 근접, 투사체, 마법, 지원, 비행, 대형 역할.
- 시뮬레이션 시간이 일시정지·감속·히트 스톱과 어떻게 분리되는지.
- 전투 데이터가 판정을 소유하는지, 애니메이션 이벤트가 판정을 소유하는지.
- 좌우 미러링, 방향 수, 아틀라스 규격.
- 스프라이트·리깅·3D 등 제작 방식.

## 3. 공통 상태 계약

대부분의 전투 유닛은 다음 상태를 검토한다.

```text
deploy
idle
move
attack_basic
skill
hit_light
hit_heavy
controlled
death
capture_or_interact
victory
defeat
```

특수 역할은 별도 상태를 추가한다.

- 돌진: 준비, 가속, 충돌, 방향 회복.
- 비행: 상승, 순항, 급강하, 공중 회복.
- 공성: 구조물 전용 공격.
- 잠입: 진입, 이동, 출현.
- 지속 시전: 시작, 유지, 종료 또는 취소.

상태 이름과 전이 조건은 코드·데이터·자산에서 일관되게 사용한다.

## 4. 공격 타이밍

모든 공격은 세 구간으로 나눈다.

```text
준비 Anticipation
→ 판정 Active
→ 회복 Recovery
```

- 준비는 방향, 범위, 무게, 회피 가능성을 알린다.
- 판정은 무기 접촉, 투사체 발사, 마법 방출과 일치한다.
- 회복은 공격 후 빈틈과 역할의 무게를 표현한다.
- 판정 전 취소와 판정 후 취소의 결과를 구분한다.
- 판정 오차 목표는 한 애니메이션 프레임 이내로 둔다.

## 5. 전투 데이터와 애니메이션 분리

권장 책임:

```text
전투 데이터
- 피해량
- 공격 주기
- 판정 범위
- 타깃 조건
- 취소 가능 조건

애니메이션 데이터
- 상태 ID
- 재생 시간 또는 FPS
- 판정 이벤트 프레임
- 투사체 생성 프레임
- VFX·오디오 이벤트
- 카메라 충격 이벤트
```

스프라이트 시트가 피해량이나 실제 범위를 결정하지 않는다. 애니메이션 이벤트는 전투 판정과 표현을 동기화하되 규칙 원본을 중복 소유하지 않는다.

## 6. 공용 데이터와 시각 변형 재사용

같은 역할과 전투 규칙을 여러 진영·스킨에서 공유할 때 다음 구조를 우선 검토한다.

```text
공용 Archetype / Combat / Animation Timing
+
진영·스킨별 Visual Set
```

공유 가능한 것:

- 능력치와 스킬 규칙.
- 상태 ID와 전이.
- 프레임 수·재생 시간·판정 이벤트 위치.
- 충돌·투사체·타기팅 데이터.

분리 가능한 것:

- 스프라이트·모델·리깅 자산.
- 초상화와 아이콘.
- 팔레트와 재질.
- 필요할 경우 VFX·오디오 표현.

시각 세트가 같은 프레임 레이아웃과 이벤트 계약을 지키면 별도 병종 코드와 데이터 복제를 피할 수 있다. 반대로 체형·무기·공격 궤적이 크게 다르면 같은 타이밍을 억지로 적용하지 말고 공용 계약을 재평가한다.

## 7. 이동과 루트 모션

- 전략·시뮬레이션 게임에서는 코드가 위치와 경로를 소유하는 방식을 기본으로 검토한다.
- 애니메이션은 보폭과 무게를 표현하되 위치 원본이 되지 않는다.
- 이동속도 변화에 따라 재생속도만 과도하게 올리면 발이 떨리므로 보폭·루프·속도 범위를 함께 설계한다.
- 기병, 대형, 비행은 일반 인간형 이동 루프를 단순 확대하지 않는다.

## 8. 피격과 사망

가벼운 연속 피격:

- 짧은 플래시, 미세 반동, 작은 파편을 사용한다.
- 공격 판정이 계속 끊기지 않게 내부 재생 간격을 둔다.

강한 피격:

- 돌진, 밀쳐내기, 큰 단일 피해, 방어 파괴처럼 의미 있는 상황에 사용한다.
- 화면 전체 흔들림보다 해당 위치의 반응을 우선한다.

사망:

- 판정 즉시 공격·충돌·점령 기여를 중지한다.
- 시체가 이동로와 상호작용을 막지 않게 한다.
- 대형·보스 사망은 붕괴 방향과 통로 상태를 명확히 표시한다.

## 9. 승리·패배의 단계

모든 처치에 승리 모션을 재생하지 않는다.

| 사건 | 권장 반응 |
|---|---|
| 개별 처치 | 공격 회복 뒤 다음 목표 탐색 |
| 작은 교전 종료 | 짧은 호흡·무기 정리 |
| 거점·목표 확보 | 참여자만 짧은 확인 동작 |
| 중요 구조물 파괴 | 흐름을 멈추지 않는 짧은 반응 |
| 스테이지 승리 | 병종별 승리 모션과 결과 UI 연결 |
| 스테이지 패배 | 핵심 목표 붕괴와 전선 정지 중심 |

승리 연출은 보상감을 주되 다음 조작과 결과 확인을 늦추지 않는다.

## 10. 다수 유닛 동기화

- 동일 병종의 대기·이동 루프는 결정론적 초기 오프셋을 사용할 수 있다.
- 공격 판정 시간과 실제 공격 주기는 무작위로 흐리지 않는다.
- 같은 시드와 입력에서 오프셋도 재현되게 한다.
- 무작위 흔들림으로 생동감을 만들지 말고 의미 있는 보조 동작을 제한적으로 사용한다.

## 11. 카메라·히트 스톱·화면 흔들림

- 일반 공격에는 기본적으로 사용하지 않거나 매우 약하게 사용한다.
- 강한 충돌, 보스, 대형 구조물 붕괴에만 제한한다.
- 여러 유닛이 동시에 공격할 때 게임 전체가 반복적으로 멈추지 않게 한다.
- 다른 전선이나 UI 판단을 방해하는 자동 카메라 이동을 피한다.
- 접근성 옵션에서 흔들림과 플래시를 줄일 수 있는 구조를 고려한다.

## 12. 제작 순서

```text
역할별 키포즈
→ 대기·이동
→ 기본 공격의 준비·판정·회복
→ 피격·사망
→ 핵심 스킬
→ 특수 이동·상호작용
→ 승리·패배
→ VFX·오디오·카메라 동기화
→ 실제 화면 크기 검증
```

대표 체형과 역할을 먼저 검증한 뒤 전체 병종으로 확장한다.

## 13. 검증 기준

- 실제 플레이 줌에서 이동만 보고 역할을 구분할 수 있다.
- 판정과 접촉·발사 프레임 오차가 목표 범위 안이다.
- 무기와 진행·공격 방향이 일치한다.
- 같은 병종 여러 기가 겹쳐도 공격 방향이 읽힌다.
- 연속 피격으로 애니메이션이 계속 끊기지 않는다.
- 대형 공격은 준비·판정·회복을 구분할 수 있다.
- 스킨·진영 변형이 공용 프레임·이벤트 계약을 지킨다.
- 승리 연출이 결과 UI와 다음 입력을 지연하지 않는다.
- 저해상도와 최대 객체 수에서도 VFX가 핵심 정보를 가리지 않는다.

## 14. 실패 기준

- 모든 병종에 같은 걷기·공격 모션을 무기만 바꿔 적용.
- 실제 피해가 접촉보다 먼저 또는 한참 뒤 발생.
- 빠른 공격속도에서 준비·회복이 사라져 무게와 회피 가능성이 무너짐.
- 매 타격마다 큰 흔들림과 히트 스톱 사용.
- 스킨별 자산이 공용 이벤트 프레임을 지키지 않음.
- 순간이동·잠입·돌진이 같은 연출로 보임.
- 승리 모션이 전투 중 반복되어 흐름을 막음.
