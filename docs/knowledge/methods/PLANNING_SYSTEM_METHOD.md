# 프로젝트 기획 시스템 설계 방법

- 상태: 공용 방법
- 목적: 아이디어·기능·아트·연출을 개별 목록으로 관리하지 않고, 사용자 가치에서 검증까지 이어지는 책임 문서 체계를 만든다.

## 1. 기획의 여덟 층

```text
문제와 목적
→ 사용자 약속
→ 핵심 행동과 판단
→ 감정과 즉시 반응
→ 규칙과 상태 변화
→ 콘텐츠와 표현 방식
→ 구현 경계와 위험
→ 검증과 학습
```

| 층 | 반드시 답할 질문 |
|---|---|
| 문제와 목적 | 누구의 어떤 문제를 왜 지금 해결하는가? |
| 사용자 약속 | 사용자가 무엇을 기대하고 실제로 무엇을 받는가? |
| 행동과 판단 | 사용자가 반복해서 무엇을 보고 선택·조작하는가? |
| 감정과 반응 | 행동 직후 무엇을 느끼고 어떻게 알아차리는가? |
| 규칙과 상태 | 조건·입력·판정·결과·기록은 무엇인가? |
| 콘텐츠와 표현 | 같은 규칙을 어떤 장면·대사·아트·연출로 경험하는가? |
| 경계와 위험 | 무엇이 상태를 소유하고 무엇을 침범하지 않는가? |
| 검증과 학습 | 어떤 관찰 결과로 성공·실패·재설계를 판단하는가? |

상위 층이 불명확하면 하위 기능·자산 제작을 시작하지 않는다.

## 2. 책임 문서 계층

```text
현재 상태
→ 프로젝트 방향
→ 전체 기획서
→ 분야별 책임 문서
→ 실행 로드맵
→ Issue·Goal·구현 지시
→ 테스트·QA
→ 사례·회고
```

- **현재 상태:** 구현, 승인 계획, 진행 중, 백업을 구분한다.
- **프로젝트 방향:** 사용자 약속, 핵심 경험, 불변 조건, 금지 방향을 책임진다.
- **전체 기획서:** 시스템과 콘텐츠 사이의 관계를 설명한다.
- **분야별 책임 문서:** 서사, 아트, 연출, 데이터, 경제처럼 한 주제의 승인 원본이다.
- **로드맵:** 단계 의존성, 진입·종료 기준, 검증을 책임진다.
- **Issue·Goal:** 현재 실행 범위와 수정·금지 범위를 적는다.
- **QA·사례:** 결과와 재사용 가능한 교훈을 보존한다.

같은 규칙의 전문을 여러 문서에 복사하지 않는다.

## 3. 상태 표기

| 상태 | 의미 |
|---|---|
| 구현 확인 | 현재 코드·데이터·실행으로 확인됨 |
| 승인 계획 | 방향과 완료 기준은 승인됐으나 구현 전 |
| 진행 중 | 부분 구현 또는 미완료 검증 |
| 가설 | 조사·PoC·사용자 테스트 필요 |
| 백업 | 현재 작업의 기본 읽기 대상이 아닌 과거 근거 |

문서와 전달 패키지가 존재한다는 사실을 구현 완료 증거로 사용하지 않는다.

## 4. 한 주제 한 원본

- 현재 사실은 상태 문서.
- 제품·게임의 약속은 방향 기획서.
- 시스템 관계는 전체 기획서.
- 아트·서사·연출 기준은 분야별 문서.
- 단계와 순서는 로드맵.
- 실제 값과 ID는 코드·데이터.
- 완료 근거는 테스트·QA.
- 과거 이유는 사례·Git 이력.

다른 문서는 원본 경로와 작업에 필요한 차이만 적는다.

## 5. 기획 결정 카드

```md
## 결정
- 질문:
- 현재 문제:
- 사용자 가치:
- 선택한 방향:
- 검토한 대안:
- 적용 이유:
- 제외 이유:
- 영향 범위:
- 위험:
- 검증 방법:
- 상태:
```

중요한 결정의 이유가 없으면 같은 논의를 반복하거나 표면만 유지하게 된다.

## 6. 수직 범위 자르기

큰 기능은 파일·메뉴가 아니라 사용자 경험의 수직 단위로 자른다.

```text
진입 조건
→ 사용자 행동
→ 시스템 반응
→ 결과와 기록
→ 저장·복귀
→ 검증
```

피할 분할:

- 데이터 구조만 만들고 화면·결과 없음.
- UI만 만들고 실제 상태 연결 없음.
- 전체 콘텐츠를 작성했지만 실행 가능한 한 장면이 없음.
- 모든 아트를 제작한 뒤 사용 장면을 결정함.

## 7. 우선순위

1. 핵심 경험을 이해하거나 공정하게 만드는가?
2. 첫 사용의 혼란을 줄이는가?
3. 반복 행동과 즉시 반응을 강화하는가?
4. 기존 상태·데이터를 재사용해 작게 검증 가능한가?
5. 캐릭터·브랜드·세계에 대한 애착을 강화하는가?
6. 접근성과 제작 효율을 개선하는가?
7. 분량만 늘리고 핵심을 흐리지는 않는가?

## 8. 위험 분류

| 위험 | 예시 | 대응 |
|---|---|---|
| 의미 | 핵심 경험이 다른 장르로 바뀜 | 불변 조건·금지 방향 확인 |
| 상태 | 저장·진행·경제·결제 변경 | 별도 승인, 이관·회귀 계획 |
| 정보 | 비밀·정답·미확정 사실 누설 | 공개 단계와 역할 분리 |
| UI | 핵심 행동이 가려짐 | 시선 우선순위·최소 해상도 검증 |
| 생산 | 자산 수와 변형이 과도함 | 티어·폴백·재사용 계획 |
| 인수 | 새 담당자가 이유와 상태를 이해 못함 | 책임 원본·읽기 순서·handoff |
| 조사 | 사례 표면 복제·출처 오해 | 관찰·해석·결정 분리 |

## 9. 완료 기준 작성

완료 기준은 기능명이 아니라 `조건 → 행동 → 관찰 결과`로 쓴다.

나쁜 예:

- 표정 시스템 구현.
- 관계 개선.
- 연출 강화.

좋은 예:

- 존재하지 않는 표정 키를 요청해도 기본 표정으로 진행되며 입력과 상태가 유지된다.
- 선택 후 다음 장면의 말투와 정보 공유가 달라지고 저장 후에도 유지된다.
- 모션 감소에서 흔들림이 사라져도 경고가 텍스트와 아이콘으로 전달된다.

## 10. 리뷰 체크

- [ ] 문제와 사용자 가치가 한 문장으로 설명된다.
- [ ] 구현 사실과 승인 계획이 구분돼 있다.
- [ ] 사용자가 실제로 하는 행동이 적혀 있다.
- [ ] 행동 직후 반응과 기록이 있다.
- [ ] 상태 소유자와 표현 계층이 분리돼 있다.
- [ ] 포함·제외 범위가 있다.
- [ ] 실패·취소·복귀·중복 입력·저장 예외가 있다.
- [ ] 최소 수직 PoC가 있다.
- [ ] 완료 기준이 관찰 가능하다.
- [ ] 새 담당자가 실제 파일 위치로 이동할 수 있다.
