# 게임 프로젝트 저장소 운영체계 방법

- 상태: 공용 방법
- 목적: 게임의 방향, 구현, 이미지, 사운드, 검증과 일정을 분야별 활성 책임 원본으로 연결하고 새 채팅·새 AI가 저장소만으로 작업을 재개하게 한다.
- 적용 대상: 개인 개발, 소규모 팀, AI 협업 팀, 프리프로덕션부터 출시 준비 단계까지의 게임 프로젝트

## 1. 운영체계가 해결하는 문제

문서가 많아도 다음 문제가 있으면 프로젝트는 인수인계 가능한 상태가 아니다.

- 기획은 채팅에 있고 실제 파일과 연결되지 않는다.
- 같은 규칙이 여러 문서에 복제돼 서로 다른 값이 남는다.
- 구현 완료와 기획 승인, 테스트 완료가 같은 상태처럼 기록된다.
- UI 행동, 시각 기준, 엔진 구현과 QA가 서로 다른 파일에 흩어져 전체 경험을 복원하기 어렵다.
- 새 이미지가 들어올 때 기존 승인 기준과 교체 관계가 기록되지 않는다.
- 새 채팅마다 긴 프롬프트와 과거 대화를 다시 설명해야 한다.
- 코드 변경 뒤 관련 기획서와 검증 문서가 오래된 상태로 남는다.

운영체계의 목표는 문서를 많이 만드는 것이 아니라 다음 연결을 유지하는 것이다.

```text
사용자 방향
→ 분야별 책임 원본
→ 실행 Roadmap·Issue·Plan
→ 코드·데이터·자산
→ 테스트·플레이·캡처 증거
→ 상태 갱신과 다음 작업
```

## 2. 프로젝트 운영 계층

### 2.1 프로젝트 허브

프로젝트 허브는 부서가 아니라 사람과 AI가 가장 먼저 읽는 통제 계층이다.

필수 책임:

- 최초 읽기 순서
- 현재 개발 단계와 다음 게이트
- 활성 책임 원본 지도
- 최신 방향과 구현·검증 상태
- 중요한 결정과 변경 이력
- 변경 유형별 문서 갱신 매트릭스
- 전문 용어
- 이미지·자산 상태 인덱스
- GPT·Codex·GitHub 작업 흐름

권장 위치:

```text
[기획서]/00_프로젝트_허브/
```

### 2.2 제작·지원 분야

다음 11개 책임을 기본으로 한다. 팀 규모가 작으면 한 사람이 여러 분야를 맡거나 문서를 통합할 수 있지만 책임 자체를 삭제하지 않는다.

| 분야 | 핵심 질문 | 주요 책임 |
|---|---|---|
| 설정·내러티브 | 이 세계에서 왜 일어나는가? | 세계관 정사, 시나리오, 캐릭터, 용어, 대사 |
| 게임 디자인 | 플레이어가 무엇을 보고 판단하고 반복하는가? | 핵심 루프, 전투, 경제, 성장, 콘텐츠, 밸런스 |
| UX·UI·접근성 | 플레이어가 어떻게 이해하고 조작하는가? | 정보 구조, 화면 흐름, 입력, 피드백, 온보딩, 접근성 |
| 개발·엔지니어링 | 어떤 구조가 상태와 결과를 소유하는가? | 엔진, Scene, 코드, 데이터, AI, 저장, 성능, 빌드 |
| 테크니컬 아트·파이프라인 | 자산이 어떻게 일관된 품질로 엔진에 들어가는가? | 규격, Import, 피벗, 애니메이션 계약, 도구, 자산 예산 |
| 아트 | 무엇을 어떤 시각 언어와 품질로 보여주는가? | 캐릭터, 환경, 건물, UI 그래픽, 애니메이션, VFX |
| 사운드 | 무엇을 언제 어떤 우선순위로 들려주는가? | BGM, SFX, 음성, 이벤트 연결, 믹싱, 반복 방지 |
| QA | 의도한 기능이 실제로 올바르게 작동하는가? | 자동·수동·회귀·성능·시각·오디오 테스트 |
| 프로덕션·PM | 언제, 누가, 무엇을 완료해야 하는가? | 마일스톤, 일정, 의존성, 위험, 범위, 예산 상태 |
| 분석·유저리서치 | 실제 사용자는 어떻게 행동하고 이해하는가? | 벤치마킹, SWOT, 플레이테스트, 텔레메트리, 개선안 |
| 통합검수 | 모든 결과가 서로 일치하고 다음 단계로 넘어갈 수 있는가? | 문서·구현·자산·검증·일정의 일관성, 릴리스 준비도 |

**텔레메트리(Telemetry)**: 플레이 중 발생한 행동과 시스템 상태를 수집해 사용 패턴과 문제를 분석하는 계측 데이터다.

### 2.3 분야를 독립시키는 기준

다음 중 두 가지 이상이면 독립 본책 또는 독립 분야를 검토한다.

- 서로 다른 전문 판단과 품질 기준이 있다.
- 변경 빈도와 담당자가 다르다.
- 다른 분야가 전문을 복사하지 않고 참조해야 한다.
- 독립적인 제작 파이프라인이나 검증이 있다.
- 결과가 늦게 발견되면 비용이 크게 증가한다.

규모가 작아 통합할 때도 문서 내부 장과 책임 표로 경계를 유지한다.

## 3. 책임 원본 구조

### 3.1 한 질문 한 원본

**SSOT(Single Source of Truth)**: 같은 질문에 대해 현재 공식 답을 소유하는 책임 원본을 하나만 두는 원칙이다.

```text
프로젝트 전체 현재 상태 → Active Context
프로젝트 비전·약속 → 프로젝트 방향서
분야의 방향·품질·현황 → 분야별 본책
세부 수치·계약 → 데이터·기술 부록 또는 실제 데이터
개발 순서 → Roadmap
현재 실행 범위 → Issue·Goal·Plan
완료 증거 → 테스트·QA·캡처
중요한 이유 → Decision Log
과거 상태 → Git 이력
```

다른 문서는 원본 링크, 필요한 요약과 해당 작업의 차이만 기록한다.

### 3.2 본책과 부록

본책은 분야 전체의 방향과 현재 상태를 설명하는 활성 진입점이다. 모든 수치와 세부 계약을 본책에 복제하지 않는다.

본책이 책임지는 것:

- 분야 목적과 사용자 가치
- 확정 방향과 금지 방향
- 현재 구현·검증 상태
- 분야별 품질 기준
- 다른 분야와의 입력·출력 계약
- 최신 자료와 실제 캡처
- 위험, 미확정, 다음 작업
- 세부 부록 색인

부록이 책임지는 것:

- 고유 수치표
- 데이터 스키마
- 화면별 명세
- 자산 계약
- 테스트 케이스
- 조사 보고서
- 과거 결정을 이해하는 데 필요한 근거

## 4. 공통 문서 상태

| 상태 | 의미 | 필요한 증거 |
|---|---|---|
| 확정 | 방향·규칙이 승인됨 | 사용자·책임자 결정 또는 Decision Record |
| 구현 | 실제 파일에 존재함 | 코드·Scene·데이터·자산 경로 |
| 검증 | 실제 동작이 확인됨 | 테스트 결과·플레이 기록·캡처 |
| 진행 중 | 부분 구현 또는 검증 | 완료되지 않은 범위와 남은 항목 |
| 가설 | PoC·조사·사용자 테스트 필요 | 검증 질문과 성공·실패 기준 |
| 미확정 | 아직 결정되지 않음 | 결정이 필요한 질문 |
| 보류 | 활성 범위에서 의도적으로 제외 | 재개 조건 |
| 대체됨 | 새 원본으로 교체됨 | 새 책임 원본 경로 |
| 불일치 | 원본과 실제 결과가 다름 | 차이, 영향, 수정 계획 |

상태는 문서 전체뿐 아니라 중요한 섹션·자산·기능 단위에도 적용할 수 있다.

## 5. 사용자 가독성 계약

각 분야 본책의 첫 화면은 상태 대시보드처럼 읽혀야 한다.

필수 상단 정보:

```yaml
문서명:
분야:
주 책임:
상태:
마지막 편집일:
마지막 검토일:
기준 커밋:
관련 구현 기준:
관련 검증 기준:
최신 자료:
현재 위험:
다음 작업:
```

권장 본문 순서:

1. 한눈에 보기
2. 목차
3. 분야 목표와 품질 기준
4. 현재 상태
5. 확정 사항
6. 구현 상태
7. 검증 상태
8. 상세 기획
9. 다른 분야와의 계약
10. 최신 이미지·캡처·자료
11. 위험과 미확정
12. 다음 작업
13. 완료 기준
14. 부록 색인
15. 변경 이력

Markdown을 편집 가능한 책임 원본으로 사용한다. PDF·HTML·DOCX는 Markdown과 이미지에서 생성하는 열람본으로 관리하고 별도 수정하지 않는다.

## 6. 이미지·시각자료 운영

### 6.1 상태와 용도를 분리한다

| 상태 | 의미 |
|---|---|
| REFERENCE | 아이디어·부분 요소 참고 |
| DIRECTION_APPROVED | 시각 방향 승인 |
| PRODUCTION_READY | 제작 규격 확정 |
| IMPLEMENTED | 게임에 적용됨 |
| VISUALLY_VALIDATED | 실제 화면에서 검증됨 |
| PARTIAL_REFERENCE | 일부 요소만 계속 참고 |
| SUPERSEDED | 새 기준으로 교체됨 |
| MIGRATION_PENDING | 인덱스에는 있으나 파일 이전·저장 미완료 |

이미지 한 장 전체를 승인하지 않는다. 참고할 요소와 참고하지 않을 요소를 명시한다.

### 6.2 캐노니컬 경로

현재 공식 이미지에는 항목별 고정 경로 하나만 사용한다.

```text
images/canonical/battlefield_overview.webp
```

새 승인본은 같은 경로를 교체하고 이전 상태는 Git 이력으로 보존한다. `v2`, `final`, `latest`, 날짜별 활성 복제본은 사용하지 않는다.

### 6.3 Visual DNA

**Visual DNA**: 같은 계열의 이미지가 일관된 시각 언어로 재현되도록 비율, 실루엣, 팔레트, 카메라, 재질, 광원과 후처리 규칙을 압축한 카드다.

필수 항목:

- 목적과 사용 화면
- 승인 레퍼런스
- 실루엣·비율
- 카메라·구도
- 해상도·픽셀 밀도
- 팔레트·명암
- 재질·외곽선
- 허용 변형과 금지 요소
- 생성 이미지 프롬프트 핵심
- 후처리와 엔진 Import
- 실제 게임 캡처와 차이

### 6.4 골든 스크린샷

**골든 스크린샷(Golden Screenshot)**: 특정 화면의 승인된 실제 게임 기준 캡처로, 이후 변경에서 의도하지 않은 시각 회귀를 비교하는 기준이다.

콘셉트 이미지, 승인 방향, 현재 구현 캡처와 골든 스크린샷을 같은 상태로 취급하지 않는다.

## 7. 변경 영향과 추적성

모든 L1 이상 작업은 주 책임 분야, 영향 분야와 변경 유형을 선언한다.

```yaml
primary_discipline:
affected_disciplines:
change_type:
design_sources:
implementation_paths:
asset_paths:
test_paths:
required_evidence:
```

**추적성 매트릭스(Traceability Matrix)**: 요구·결정·구현·테스트·증거가 어디에서 서로 연결되는지 보여주는 표다.

권장 연결:

```text
결정 ID
→ 분야별 본책 섹션
→ Issue·Plan
→ 코드·데이터·자산
→ 테스트 케이스
→ 결과·캡처
```

## 8. 개발 단계와 게이트

**게이트(Gate)**: 다음 단계로 넘어가기 전에 완료 조건과 증거를 검토하는 승인 지점이다.

| 단계 | 주석 |
|---|---|
| Concept | 게임 약속과 대상 플레이어를 확정하는 단계 |
| Prototype | 재미·조작·기술 가설을 빠르게 검증하는 폐기 가능한 시제품 |
| Graybox | 임시 도형과 최소 자산으로 공간·규칙·동선을 검증하는 단계 |
| First Playable | 핵심 루프를 처음부터 끝까지 처음 플레이할 수 있는 상태 |
| Vertical Slice | 최종 목표 품질과 제작 파이프라인을 좁은 범위에서 증명한 플레이 구간 |
| Production | 검증된 파이프라인으로 콘텐츠를 본격 제작하는 단계 |
| Alpha | 주요 기능이 연결돼 전체 진행을 검증하는 단계 |
| Feature Complete | 계획된 주요 기능이 모두 들어간 상태 |
| Content Complete | 출시 범위 콘텐츠 제작이 완료된 상태 |
| Beta | 신규 기능보다 버그·밸런스·최적화에 집중하는 단계 |
| Release Candidate | 치명적 문제가 없다면 출시할 수 있는 후보 빌드 |

각 분야 본책에는 현재 단계와 다음 게이트의 분야별 완료 기준을 기록한다.

## 9. GPT·Codex·GitHub 역할

### GPT

- 방향과 의사결정 구조화
- 분야·영향도 판정
- 기획서와 사용자 열람 구조 작성
- 분석·벤치마킹
- Codex 작업 패키지 작성
- 통합검수

GPT는 대화를 최종 책임 원본으로 사용하지 않는다. 승인된 결정은 프로젝트 저장소에 반영한다.

### Codex Plan Mode

- 실제 저장소 읽기
- 기획과 구현 상태 비교
- 예상 변경 파일·데이터 책임·위험·검증 제안
- 관련 본책 갱신 계획 포함

### Codex 구현

- 승인된 Plan 범위만 변경
- 코드·데이터·자산·테스트·문서 동기화
- 실행 결과와 미검증 항목 보고

### GitHub

- 활성 문서와 이력의 책임 원본
- Issue·PR 작업 계약
- CODEOWNERS와 리뷰
- GitHub Actions 검사
- 브랜치 보호와 릴리스 증거

## 10. 자동화 가능한 것과 불가능한 것

자동 검사 적합:

- 깨진 링크와 누락 이미지
- 필수 문서·메타데이터
- 금지된 활성 버전 파일명
- Manifest 중복
- 변경 유형별 본책 갱신 누락
- Markdown 열람본 생성 실패
- 코드·데이터 자동 테스트
- 이미지 형식·크기·Import 규격

사람·AI 통합검수가 필요한 것:

- 실제 재미와 명확성
- 이미지가 프로젝트 방향과 맞는지
- 상충하는 기획의 의미 판단
- 버티컬 슬라이스 품질 기준 충족
- 예산·일정·출시 승인

## 11. 설치와 이관 원칙

기존 프로젝트를 정리할 때 바로 이동·삭제하지 않는다.

```text
저장소 감사
→ 문서·이미지·코드·테스트 인벤토리
→ 주 책임 분야·영향 분야 분류
→ 고유 결정·수치·계약 확인
→ 새 본책과 부록으로 승계 계획
→ 사용자 승인
→ 경로·참조·자동화 설치
→ 통합검수
→ 완전히 흡수된 중복만 제거
```

기존 승인 이미지가 있으면 새 시안을 만들지 않는다. 상태와 경로만 정리한다.

## 12. 콜드 스타트 완료 기준

새 작업자가 10분 안에 다음을 답할 수 있어야 한다.

1. 게임의 핵심 약속과 대상 플레이어는 누구인가?
2. 현재 개발 단계와 다음 게이트는 무엇인가?
3. 현재 최우선 작업과 선행 조건은 무엇인가?
4. 무엇이 확정·구현·검증·미확정인가?
5. 분야별 현행 책임 원본은 어디인가?
6. 최신 이미지와 실제 게임 캡처는 어디인가?
7. 변경하면 안 되는 방향과 보호 경로는 무엇인가?
8. 관련 코드·데이터·테스트는 어디인가?
9. 작업 종료 시 무엇을 갱신해야 하는가?

답할 수 없으면 운영체계 설치가 완료된 것이 아니다.
