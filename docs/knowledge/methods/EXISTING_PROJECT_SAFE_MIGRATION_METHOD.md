# 기존 게임 프로젝트 안전 마이그레이션 방법

- 상태: 공용 방법
- 목적: 이미 운영 중인 프로젝트의 고유 결정·구현·승인 자료·작업 이력을 보존하면서 문서, 스킬, 템플릿과 작업 구조를 단계적으로 재배치한다.
- 적용 대상: Base 기준을 기존 프로젝트에 분화·적용하는 구조 개편, 문서 통합, 폴더 이동과 책임 원본 재정의

## 1. 최우선 원칙

```text
기존 내용 보존
→ 현재 책임과 참조 관계 파악
→ 중복·충돌·누락 분석
→ 이동·통합·재배치 계획
→ 사용자 검토
→ 승인된 범위만 변경
→ 변경 전후 보존·참조 검증
```

Base 구조를 그대로 복제하거나 프로젝트를 강제로 개명하지 않는다. 최신 사용자 지시와 프로젝트 현행 책임 원본이 Base 공용 예시보다 우선한다.

## 2. 절대 보존 대상

다음은 구조가 좋지 않거나 중복돼 보여도 조사 없이 삭제·축약·변경하지 않는다.

- 사용자 승인 기획과 결정
- 세계관, 설정, 용어, 수치와 예외
- 현재 구현 상태와 실제 경로
- 승인 이미지·UI·다이어그램·프롬프트
- 테스트 결과, 실패 사례와 미검증 기록
- Roadmap·Issue·Goal·Plan·PR 진행 이력
- Active Context·Handoff의 미완료 작업
- `[보류]`, `[확인 필요]`, `[미검증]`
- 외부 문서·코드·자동화에서 참조되는 파일
- 출처·승인 근거·감사 기록

삭제 전에는 해당 정보가 다른 책임 원본 또는 Git 이력만으로 안전하게 복구 가능한지 확인한다.

## 3. 변경 권한 단계

### 단계 A — Audit only

첫 단계에서는 다음만 작성한다.

- 현재 구조 분석
- 현행 책임 문서 지도
- 파일·이미지·스킬·PDF 인벤토리
- 중복·충돌·누락 목록
- 이동·통합·재배치 제안
- 제거 후보와 보존 근거
- 변경 전후 예상 구조
- 위험·검증·롤백 방법

명백한 오탈자와 복구 가능한 끊어진 내부 링크 외에는 실제 구조를 바꾸지 않는다.

### 단계 B — Governance foundation

사용자 승인 후 시작 문서, Documentation Map, 상태 규칙, 갱신 매트릭스와 프로젝트 스킬 지도를 설치한다. 기존 원본은 우선 연결하고 대규모 이동·삭제하지 않는다.

### 단계 C — Approved migration

승인된 파일 처리표에 따라 내용 승계, 참조 변경, 이동과 통합을 수행한다.

### 단계 D — Cleanup·Enforcement

보존 대조와 링크·콜드 스타트 검증이 끝난 후 완전히 흡수된 중복만 제거한다. PR 검사와 브랜치 보호는 실제 활성 상태를 별도로 검증한다.

## 4. 최초 조사 범위

- 프로젝트 루트와 주요 폴더
- `AGENTS.md`, README, START_HERE
- Documentation Map
- Active Context·Handoff
- Roadmap, Issue·Goal·Plan·PR
- 분야별 기획서와 부록
- 분야별 프로젝트 스킬과 foundation 스킬
- 템플릿과 자동화
- PDF·HTML·DOCX 파생본과 Publication Manifest
- 승인 이미지·자산과 Visual Source of Truth
- 테스트·검증 문서와 실제 명령
- 코드·데이터·설정이 참조하는 문서·파일 경로
- 최근 커밋과 이동·삭제 이력

`[백업]`, `[보류]`, archive, hold는 기본 활성 읽기에서 제외하되 고유 정보와 참조 감사에는 포함한다.

## 5. 파일 인벤토리 계약

각 파일·자산·스킬에 대해 다음을 기록한다.

| 항목 | 내용 |
|---|---|
| 현재 경로 | 실제 저장소 경로 |
| 현재 역할 | 이 파일이 답하는 질문과 산출물 |
| 참조 위치 | 문서·코드·Issue·PR·PDF·자동화의 참조 |
| 고유 내용 | 다른 파일에 없는 문장·표·이미지·결정·예외 |
| 중복 내용 | 다른 원본과 겹치는 범위 |
| 상태 | 현행·보조·백업·보류·제거 후보 |
| 제안 | 유지·수정·통합·이동·이름 변경 |
| 위험 | 변경 시 누락·링크·호환성 위험 |
| 검증 | 보존과 참조를 확인할 방법 |

한 파일 안에 여러 분야 책임이 섞여 있으면 요소 단위로 인벤토리한다.

## 6. 책임 원본 판정

한 주제에 여러 파일이 있다고 즉시 하나를 삭제하지 않는다.

구분:

- 현행 책임 원본
- 현행 원본을 보조하는 자료
- 승인 기록·출처 자료
- 이전 버전
- 아직 승계되지 않은 고유 정보
- 향후 작업을 위한 `[보류]`
- 실제로 완전히 중복된 파일

하나라도 고유 문장, 표, 수치, 이미지, 예외, 결정 이유 또는 미검증 항목이 있으면 완전 중복으로 판정하지 않는다.

## 7. 통합 절차

1. 통합 대상의 고유 정보를 모두 추출한다.
2. 충돌하는 문장·수치·상태를 나란히 표시한다.
3. 최신 사용자 결정과 실제 구현·검증을 기준으로 현행 내용을 판정한다.
4. 확정할 수 없는 차이는 `[확인 필요]`로 유지한다.
5. 새 책임 원본에 고유 정보가 모두 반영됐는지 대조한다.
6. 표, 이미지, 링크, 파일 경로, 예외와 상태를 확인한다.
7. 기존 파일 참조를 새 원본으로 갱신한다.
8. 검증 전에는 원본을 삭제하지 않는다.
9. 검증 후 보존 필요성을 판정한다.

문장을 줄이는 과정에서 의미, 제약, 예외, 승인 상태, 구현 상태와 검증 결과를 제거하지 않는다.

## 8. 수명주기 영역

### `[현행]`

현재 프로젝트가 실제로 참조하는 책임 원본과 실행 스킬이다. 기본 컨텍스트는 이 영역과 현재 작업에 필요한 실제 파일만 읽는다.

### `[백업]`

다음처럼 Git 이력만으로 충분하지 않은 보존 사유가 있을 때만 사용한다.

- 사용자가 별도 보존을 요청함
- 외부 원본·승인 기록·법무·감사 자료
- 외부 링크가 고정 경로를 요구함
- 비교 검토를 위해 완전한 과거 패키지가 필요함

단순 이전 버전은 별도 백업 파일을 만들지 않고 Git 이력으로 보존한다. `[백업]`은 기본 읽기에서 제외하고 Documentation Map에 보존 이유와 접근 경로만 둔다.

### `[보류]`

현재 구현·활성 범위가 아니지만 향후 검토 가치가 있는 내용이다.

각 항목 필수 정보:

```yaml
hold_reason:
resume_condition:
related_owner_document:
prerequisites:
not_in_current_scope: true
last_reviewed:
```

별도 재개 승인 전에는 구현하지 않는다.

### `[제거 후보]`

완전히 흡수됐을 가능성이 있지만 아직 삭제 승인을 받지 않은 파일이다. 삭제 조건·참조 상태·복구 방법을 기록한다.

## 9. 파일 이동·이름 변경

이동 전 검색:

- Markdown 링크와 이미지 경로
- README·START_HERE·Documentation Map·AGENTS
- 스킬·템플릿·Manifest
- 코드 주석·설정·스크립트·테스트
- Issue·PR·Goal·Plan
- PDF 링크·캡션·Publication Manifest
- 외부 링크와 자동화

모든 참조를 수정할 수 없으면 이동하지 않고 위험으로 보고한다. 가능한 경우 한 PR에서 파일 이동과 참조 수정을 함께 수행한다.

## 10. 삭제 조건

다음이 모두 충족된 경우에만 삭제 후보가 실제 삭제 단계로 간다.

- 모든 고유 정보가 현행 책임 원본에 반영됨
- 승인 자료와 출처가 별도 보존됨
- 활성·보조·외부 참조가 없음
- `[보류]`, `[확인 필요]`, `[미검증]` 내용이 없음
- 사용자가 보존을 요청한 파일이 아님
- Git 이력 또는 별도 백업으로 복구 가능
- 삭제 후 Documentation Map과 읽기 순서가 정상
- 변경 전후 보존 대조가 통과
- 사용자 승인 또는 승인된 이관표가 있음

확신할 수 없으면 삭제하지 않고 `[제거 후보]`로 유지한다.

## 11. 변경 전후 보존 검증

비교 항목:

- 전체 파일 수와 처리 유형별 수
- 문서별 제목과 주요 섹션
- 승인 결정 ID와 위치
- 고유 수치·데이터 계약
- 이미지·자산·프롬프트와 참조
- `[보류]`, `[확인 필요]`, `[미검증]`
- Roadmap 미완료 작업
- 분야별 스킬 목록
- Documentation Map 경로
- 내부 링크와 코드·데이터 참조
- PDF 원본·승인 이미지·Publication Manifest

필수 대조표:

| 기존 내용 | 기존 위치 | 변경 후 위치 | 보존 여부 | 검증 결과 |
|---|---|---|---|---|
|  |  |  |  |  |

변경 전에는 있었지만 변경 후 찾을 수 없는 내용이 하나라도 있으면 완료로 처리하지 않는다.

## 12. 콜드 스타트 검증

새 AI가 저장소만 보고 다음을 답할 수 있어야 한다.

- 프로젝트 목적과 핵심 경험
- 현재 구현·검증 상태
- 다음 우선 작업과 선행 조건
- 변경하면 안 되는 내용
- 분야별 책임 문서와 프로젝트 스킬
- 구현 전후 검증 방법
- `[보류]`, `[확인 필요]`, `[미검증]` 위치
- 승인 이미지와 최신 PDF
- Base 공용 기준과 프로젝트 전용 확장 경계

답하지 못하면 마이그레이션이 완료된 것이 아니다.

## 14. 실패 조건

- 실제 파일을 읽지 않고 구조를 추정
- 감사 없이 폴더부터 생성
- Base 구조에 맞추기 위한 강제 개명
- 여러 문서를 합친 뒤 고유 정보 대조 없이 원본 삭제
- `[보류]`를 폐기하거나 현행 기능처럼 설명
- 승인 이미지·프롬프트 제거 또는 임의 교체
- 파일 이동 후 코드·Issue·PDF 참조를 확인하지 않음
- 파일 수 감소를 성공 지표로 사용
- 실제 런타임 검증 없이 문서 정리만으로 제품 품질 완료 선언
