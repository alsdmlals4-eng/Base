# 대화·이벤트 프레젠테이션 설계 방법

- 상태: 공용 방법
- 목적: 대화 UI, 캐릭터 배치, 표정, 컷인, 화면 전환, 음향을 정보 위계와 감정 변화에 맞춰 설계하고, 표현 계층이 게임·제품 상태를 중복 소유하지 않게 한다.

## 1. 연출의 역할

연출은 다음 네 역할 중 하나 이상을 수행해야 한다.

1. **방향 제시:** 지금 무엇을 보고 어떤 행동을 해야 하는지 알려준다.
2. **중요도 표시:** 여러 정보 중 핵심을 구분한다.
3. **감정 강화:** 인물·상황·결과의 정서를 더 잘 느끼게 한다.
4. **기억 형성:** 능력, 관계, 규칙, 성취의 결정적 순간을 남긴다.

역할이 없는 효과는 줄이거나 제거한다.

## 2. 한 장면의 시선 흐름

한 장면에는 하나의 첫 시선과 하나의 중심 질문을 둔다.

```text
공간과 현재 상태
→ 주요 대상 또는 화자
→ 현재 행동·대사·질문
→ 사용자 선택·조작
→ 보조 기록·메뉴
```

화면을 만들기 전에 다음을 적는다.

- 첫 1초에 보여야 할 것.
- 3초 안에 이해할 것.
- 선택 전에 읽어야 할 정보.
- 선택 후 즉시 변해야 할 요소.
- 상세 정보가 열릴 위치.

## 3. 이벤트 비트

```text
공간 제시
→ 대상 진입 또는 변화
→ 핵심 정보·대화
→ 작은 행동·소품 변화
→ 사용자 선택·조작
→ 즉시 시각·음향 반응
→ 결과·기록
→ 다음 질문 또는 여운
```

모든 장면에 모든 비트가 필요한 것은 아니다. 목적이 없는 비트는 합치거나 삭제한다.

## 4. 인물 배치와 카메라

### 질문

- 정보 확인인가, 감정 확인인가?
- 누가 주도하고 누가 반응하는가?
- 입력 전에 공간 정보를 보여줘야 하는가?
- 가까운 컷이 실제 새 정보를 제공하는가?

### 인원별 원칙

- **1인:** 공간·소품·행동을 함께 보여준다.
- **2인:** 주도자와 반응자의 거리·시선으로 관계를 표현한다.
- **3인:** 주 화자 1명과 반응자 2명으로 우선순위를 둔다.
- **4인 이상:** 모두를 작게 세우기보다 컷 전환·그룹 반응으로 나눈다.

거리와 가림은 신뢰, 압박, 보호, 단절, 비밀 중 어떤 의미인지 장면 맥락으로 구분한다.

## 5. 대화 UI

### 기본 구성

- 배경과 인물이 대사창보다 먼저 읽힌다.
- 이름표는 화자 구분을 담당한다.
- 대사창은 얼굴·손·핵심 소품을 과도하게 가리지 않는다.
- 선택지는 직전 대사와 인과를 유지하는 위치에 일관되게 둔다.
- 시스템 안내와 캐릭터 대사는 시각적으로 구분한다.
- 보조 기록·로그·메뉴는 현재 질문보다 강한 대비를 사용하지 않는다.

### 텍스트 원칙

- 긴 문장과 현지화 확장을 고려한다.
- 타이핑 효과는 끌 수 있어야 한다.
- 자동 진행은 선택, 경고, 중요한 기록 갱신에서 멈춘다.
- 흔들림·왜곡·색 변화는 핵심 단어 일부에 제한한다.
- 생성 이미지 안의 텍스트를 의미 전달 수단으로 사용하지 않는다.

## 6. 표정 의미 키

표정은 파일 번호보다 의미 키로 요청한다.

예시:

- `neutral`
- `focus`
- `alert`
- `anger`
- `concern`
- `soft`
- `awkward`
- `hurt`
- `resolve`
- `special_state`

### 사용 규칙

- 대사와 같은 감정을 반복 설명하지 않는다.
- 말과 속마음이 다를 때 가장 가치가 높다.
- 표정 변화 빈도를 제한한다.
- 모든 인물이 모든 키를 가질 필요는 없다.
- 요청 키가 없으면 가까운 감정군 또는 기본 표정으로 폴백한다.
- 자산 부재가 대사·입력·진행을 막지 않는다.

## 7. 컷인 체계

| 유형 | 목적 |
|---|---|
| 기능·능력 발동 | 누가 어떤 방식으로 기여했는지 표시 |
| 위험 경고 | 즉시 주의할 변화와 행동 중단 요구 |
| 규칙·가설 확인 | 사용자의 해석이 맞았음을 현장에서 입증 |
| 관계 반응 | 선택 기억과 감정 전환 강조 |
| 특수 상태 | 오염, 오류, 변형, 감응 등 기본 규칙 이탈 |
| 기록·성취 갱신 | 사용자가 만든 결과가 남았음을 표시 |

### 제한

- 일반 대사마다 사용하지 않는다.
- 같은 유형을 짧은 간격으로 반복하지 않는다.
- 선택 입력 중 포커스를 빼앗지 않는다.
- 컷인이 없어도 텍스트·아이콘·상태 변화로 의미를 이해할 수 있어야 한다.
- 장르와 맞지 않는 과도한 축하·충격 연출을 피한다.

## 8. 전환과 타이밍

전환은 다음 중 하나를 알릴 때 사용한다.

- 장소 변화.
- 시간 변화.
- 정보 단계 변화.
- 감정 단계 변화.
- 사용자 통제의 시작·종료.

장식 목적만 있으면 줄인다.

권장 범위의 예:

- 일반 UI 반응: 입력 직후.
- 짧은 캐릭터 반응: 약 0.2~0.8초.
- 컷인: 약 0.8~2초.
- 핵심 문구: 자동 소거 금지 또는 충분한 확인 시간.

정확한 값은 프로젝트에서 실제 플레이로 검증한다.

## 9. 음향과 시각 신호

- 중요한 상태는 소리만으로 전달하지 않는다.
- 신호음·환경음·장비음은 반복 가능성과 피로를 함께 본다.
- 성공·실패 음향은 결과의 성격과 장르 톤에 맞춘다.
- 공포·충격 효과는 빈도와 음량을 제한한다.
- 최초 안내는 충분히, 반복 안내는 짧게 제공한다.

## 10. 접근성 폴백

필수 검토:

- 모션 감소.
- 점멸 제한.
- 색상 독립 아이콘·문구.
- 텍스트 우선 정보.
- 읽기 시간 보장.
- 키보드 포커스와 닫기·뒤로가기 일관성.
- 자막과 음향 설명.

모션 감소 시:

- 카메라 줌을 페이드·강조 테두리로 대체.
- 화면 흔들림을 문구·아이콘·색 변화로 대체.
- 이동 애니메이션을 축소하거나 즉시 배치.
- 핵심 의미와 입력 순서는 동일하게 유지.

## 11. 상태 소유 경계

프레젠테이션 계층이 할 수 있는 것:

- 현재 상태에 맞는 표정·자산 선택.
- 컷인·전환·음향 재생.
- 재생 중 입력 잠금과 해제.
- 텍스트·아이콘·VFX 표현.
- 자산 폴백과 개발 로그 기록.

프레젠테이션 계층이 해서는 안 되는 것:

- 결과 판정.
- 보상 지급.
- 관계 변화 결정.
- 퀘스트·사건 완료 처리.
- 저장 버전 변경.
- 대사 분기 조건 중복 정의.

도메인 로직이 결과를 결정하고, 프레젠테이션은 그 결과를 표현한다.

## 12. 목업과 실제 구현

목업은 시선 우선순위와 미감 탐색에 사용한다. 다음은 실제 화면에서 다시 검증한다.

- 텍스트 길이와 현지화.
- 해상도와 화면비.
- 마우스·키보드·패드 입력.
- 포커스 순서.
- 자산 크롭.
- 실제 데이터 최대 개수.
- 저장·복귀 중간 상태.
- 옵션과 접근성.

픽셀 위치보다 구조와 역할을 보존한다.

## 13. 구현 절차

1. 현재 화면과 상태 소유자를 감사한다.
2. 한 장면의 첫 시선·중심 질문·입력 순서를 적는다.
3. 기본 텍스트·UI만으로 흐름을 완성한다.
4. 표정 의미 키와 폴백을 연결한다.
5. 가장 가치가 큰 컷인 1종을 추가한다.
6. 접근성 폴백을 같은 단계에서 만든다.
7. 실제 해상도·입력·복귀를 검증한다.
8. 안정된 패턴만 다른 장면으로 확장한다.

## 14. 완료 기준

- [ ] 첫 시선과 중심 질문이 명확하다.
- [ ] 현재 화자·주요 대상·선택이 혼동되지 않는다.
- [ ] 표정·컷인이 의미 있는 순간에만 사용된다.
- [ ] 연출을 꺼도 핵심 정보를 이해한다.
- [ ] 자산 누락 시 기본 폴백으로 진행된다.
- [ ] 컷인·오버레이·메뉴의 입력 포커스가 충돌하지 않는다.
- [ ] 긴 텍스트와 최소 해상도에서 잘림이 없다.
- [ ] 음향·색상만으로 상태를 전달하지 않는다.
- [ ] 프레젠테이션 계층이 결과와 저장 상태를 소유하지 않는다.
- [ ] 실제 사용자가 중요한 순간을 더 잘 기억하는지 확인했다.
