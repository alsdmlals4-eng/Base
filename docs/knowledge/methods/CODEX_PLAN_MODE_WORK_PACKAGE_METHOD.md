# Codex Plan Mode 작업 패키지 설계 방법

- 상태: **채택**
- 적용 범위: 새 Codex 채팅에서 기존 프로젝트 문서를 바탕으로 구현 전 Plan Mode 제안서를 작성하게 하는 작업
- 목적: 이전 대화가 없는 Codex가 제품 방향, 책임 원본, 실제 저장소 상태와 승인 게이트를 혼동하지 않고 좋은 구현 계획을 제출하게 한다.

---

## 1. 세 산출물을 구분한다

Codex 협업에서는 다음 세 문서를 같은 것으로 취급하지 않는다.

### 작업 요청서 Work Order

사용자·기획자가 Codex에 전달하는 입력이다.

- 무엇을 조사할지.
- 어떤 문서를 읽을지.
- 바꾸면 안 되는 불변 조건.
- 검토할 선택과 위험.
- 산출물 형식.
- 구현 금지 범위.

### Plan Mode 제안서

Codex가 실제 저장소와 근거를 조사한 뒤 제출하는 출력이다.

- 현재 상태와 근거.
- 추천 구조와 대안.
- 예상 파일·상태 소유·데이터 경계.
- 단계별 구현 계획.
- 검증과 위험.
- 사용자 결정 요청.

### 구현 결과

사용자가 Plan Mode 제안서를 승인한 뒤 별도 실행에서 만드는 코드·데이터·Scene·테스트와 PR이다.

```text
작업 요청서
≠ Plan Mode 제안서
≠ 구현 결과
```

이 구분이 없으면 기획자가 적은 기술 가설이 Codex 검토 없이 확정되거나, Plan Mode 요청이 구현 승인으로 오인된다.

---

## 2. 작업 패키지의 목표

좋은 작업 패키지는 새 채팅의 Codex가 10~20분 안에 다음을 답할 수 있게 한다.

1. 제품 또는 기능의 목적은 무엇인가.
2. 현재 실제 구현 상태는 무엇인가.
3. 어떤 문서가 책임 원본인가.
4. 무엇은 승인된 불변 조건인가.
5. 무엇은 추천 가설이며 다시 검토해야 하는가.
6. 이번 채팅에서 무엇을 만들면 안 되는가.
7. 최종 Plan Mode 제안서에 무엇을 포함해야 하는가.
8. 사용자가 무엇을 승인해야 다음 단계로 갈 수 있는가.

---

## 3. 필수 구성

### A. 복사 가능한 시작 프롬프트

첫 부분에는 새 Codex 채팅에 그대로 붙여 넣을 수 있는 프롬프트를 둔다.

포함할 것:

- 저장소와 Issue/Goal.
- Plan Mode·읽기 전용이라는 선언.
- 구현 파일·브랜치·PR 금지.
- 읽기 순서.
- 실제 저장소 조사 지시.
- 참고 저장소와 우선순위.
- 최종 산출물 형식.
- 마지막 상태 문구.

프롬프트는 “좋은 계획을 세워줘”처럼 추상적으로 쓰지 않는다.

### B. 제품 또는 기능 컨텍스트

작업자가 기술만 보고 제품 의도를 놓치지 않도록 다음을 짧게 제공한다.

- 한 문장 약속.
- 핵심 사용자 흐름.
- 이번 작업이 사용자 가치에 연결되는 이유.
- 이번 단계가 구현하는 것과 구현하지 않는 것.

### C. 승인 불변 조건

임의로 바꾸면 안 되는 구조를 목록화한다.

예:

- 제품 플랫폼과 입력.
- 핵심 데이터 소유.
- 콘텐츠 수와 관계.
- 월드·전장·화면 구조.
- 보안·저장·호환 규칙.
- 승인 게이트.

불변 조건과 수치 가설을 같은 목록에 섞지 않는다.

### D. 실제 상태 조사 목록

Codex가 문서만 믿지 않고 저장소를 확인하게 한다.

- 핵심 파일 존재 여부.
- 기존 코드·Scene·데이터·테스트.
- 현재 실행 명령.
- 문서와 실제 파일의 충돌.
- 보호 경로와 마이그레이션 위험.
- 참고 저장소의 채택·제외 요소.

검색 결과가 없다는 이유만으로 파일이 없다고 단정하지 않도록 트리와 알려진 경로 확인을 요구한다.

### E. 검토할 작업 묶음

기능명이 아니라 책임 단위로 나눈다.

예:

- 엔진·환경.
- Scene·상태 소유.
- 데이터·직렬화.
- UI·Signal.
- 시간·결정론.
- 아트·애니메이션 계약.
- 자동·수동 검증.
- 문서·인수인계.

각 묶음에는 질문 형태의 검토 항목을 둔다.

### F. 사전 추천안

기획 측에서 생각한 추천이 있다면 숨기지 말고 표로 제공한다.

단, 다음 문구를 명시한다.

> 사전 추천은 검증 대상이며 Codex의 최종 Plan Mode 제안서가 아니다.

Codex가 채택·수정·기각 이유를 적게 한다. 추천안이 이미 승인된 것처럼 쓰지 않는다.

### G. 산출물 요구

다음을 구체적으로 요구한다.

- Scene 또는 구성 트리.
- 예상 파일 경로와 책임.
- 상태 소유와 데이터 흐름.
- 단계별 작업과 관찰 가능한 결과.
- 자동·수동 검증.
- 위험과 대응.
- 사용자 결정 요청.
- 승인 게이트.

### H. 포함·제외 범위

제외 범위에는 특히 다음을 적는다.

- 실제 파일 수정.
- 브랜치·커밋·PR.
- 인접 시스템의 선행 구현.
- 최종 자산과 콘텐츠.
- 승인되지 않은 의존성.
- 범용 프레임워크 확장.

### I. 중단 조건

Codex가 질문 없이 추정하면 위험한 상황을 미리 적는다.

- 책임 원본 충돌.
- 예상 밖 기존 구현.
- 승인 불변 조건 변경 필요.
- 공식 도구 버전 불일치.
- 범위 밖 구현이 필수로 보이는 경우.
- 외부 의존성이 필요한 경우.

---

## 4. 읽기 순서 설계

권장 순서:

```text
최신 사용자 지시
→ 프로젝트 작업 규칙
→ 현재 상태·인수인계
→ 문서 라우터
→ 현재 Work Order
→ 전체 기획
→ 관련 승인 책임 문서
→ Roadmap·Issue·Goal
→ 실제 파일·테스트
→ 사전 기술 추천
→ 참고 저장소·외부 근거
```

사전 기술 추천을 승인 책임 문서보다 먼저 읽히면 기술안이 제품 의도를 덮을 수 있다.

모든 과거 문서를 읽게 하지 않는다. 문서 라우터가 필요한 책임 원본만 선택하게 한다.

---

## 5. 좋은 요청과 나쁜 요청

### 나쁜 요청

```text
기획서 보고 Godot 프로젝트를 잘 설계해줘.
```

문제:

- 구현인지 계획인지 불명확.
- 어떤 문서가 최신인지 모름.
- 제품 불변 조건 없음.
- 실제 저장소 조사 요구 없음.
- 완료 기준과 승인 게이트 없음.

### 좋은 요청

```text
Issue #12를 Codex Plan Mode로 조사하세요.
이번 실행에서는 구현 파일과 브랜치를 만들지 마세요.
HANDOFF → DOCUMENTATION_MAP → Work Order → 관련 APPROVED → Issue/Goal → 실제 파일 순으로 읽으세요.
실제 저장소의 구현 유무를 확인하고, 사전 추천안을 비판적으로 검토한 뒤 PROPOSAL_WORKFLOW 형식으로 예상 파일·책임·단계·검증·위험·사용자 결정을 제출하세요.
```

---

## 6. 사전 기술 추천을 다루는 방법

기획자가 미리 만든 기술안은 유용하지만 다음 위험이 있다.

- 실제 코드와 맞지 않음.
- 최신 공식 버전과 다름.
- 범위가 너무 큼.
- 구현 비용과 테스트 부담이 숨겨짐.
- 한 프로젝트 사례를 그대로 복사함.

따라서 Work Order는 Codex에 다음을 요구한다.

1. 실제 파일과 공식 근거 확인.
2. 추천안의 채택·수정·기각 판단.
3. 선택하지 않은 대안의 이유.
4. 범위 축소 가능성.
5. 다음 단계에 넘길 인터페이스만 남길 항목.

---

## 7. Plan Mode 출력 품질 체크

- 실제 파일과 근거가 있다.
- 추천안 하나가 중심이고 대안이 과도하지 않다.
- 상태 소유와 데이터 책임이 중복되지 않는다.
- 예상 경로를 실제 저장소 확인 없이 사실처럼 쓰지 않는다.
- 구현 단계마다 화면·로그·테스트에서 확인할 결과가 있다.
- 자동 테스트와 수동 검수가 분리돼 있다.
- 구현하지 않은 것을 PASS로 보고하지 않는다.
- 위험이 추상적이지 않고 대응 방법이 있다.
- 사용자 결정 요청이 너무 많지 않다.
- 마지막에 승인 전 구현 금지를 명시한다.

---

## 8. 프로젝트 문서 연결 규칙

프로젝트에는 다음만 둔다.

- 현재 Work Order.
- 프로젝트 전용 불변 조건.
- 실제 파일·Issue·Goal.
- 사전 추천과 최종 Codex 제안.

Base에는 다음을 둔다.

- Work Order 작성 방법.
- 범용 템플릿.
- 반복되는 실패 사례.
- 검수 체크리스트.

프로젝트 Work Order 전체를 Base에 복제하지 않는다. 프로젝트에서 검증된 교훈만 사례 또는 방법으로 승격한다.
