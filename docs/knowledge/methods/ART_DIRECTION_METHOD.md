# 아트 디렉션·제작 규격 설계 방법

- 상태: 공용 방법
- 목적: 스타일 키워드만 제시하는 수준을 넘어, 실제 게임 화면에서 읽히고 반복 제작 가능한 아트 계약을 만든다.

## 1. 아트 기획의 기본 순서

```text
플레이 화면과 카메라
→ 정보 우선순위
→ 실루엣과 형태 언어
→ 크기·비율·팔레트·재질
→ 상태·등급·진영 차이
→ 제작 규격과 파일 구조
→ 실제 화면 축소 검증
```

아트 스타일을 정할 때 장르 이미지나 분위기 문장부터 시작하더라도 최종 문서는 반드시 화면 크기, 자산 크기, 기능 구분, 상태 표현과 검수 기준으로 내려와야 한다.

## 2. 먼저 확인할 입력

- 플랫폼과 기준 해상도.
- 카메라 거리와 확대·축소 범위.
- 한 화면 최대 유닛·건물·VFX 수.
- 플레이어가 1초 안에 구분해야 하는 정보.
- 진영, 역할, 등급, 상태의 수.
- 실제 조작 중 UI가 차지하는 영역.
- 픽셀, 벡터, 3D, 2D 컷아웃 등 제작 방식.
- 팀 규모, 제작 시간, 외주 여부.
- 생성형 콘셉트 사용 범위와 최종 자산 후처리 규칙.

입력이 없으면 임의의 고해상도 콘셉트 아트가 아니라 가장 위험한 화면 가독성 가설부터 확인한다.

## 3. 형태 언어를 기능과 연결한다

좋은 형태 언어는 “둥근 진영”, “어두운 적”처럼 추상적인 구분만 하지 않는다.

| 정보 | 형태로 표현할 항목 |
|---|---|
| 병종 역할 | 무기 길이, 자세, 몸통 폭, 이동 중심 |
| 진영 | 직선·곡선·대칭·비대칭, 소재, 부속 구조 |
| 등급 | 실루엣 부속, 장비 복잡도, 문양, 짧은 연출 |
| 상태 | 파손, 건설, 비활성, 강화, 위험 전조 |
| 지상·비행 | 그림자, 높이 연결, 날개·부유 구조 |
| 구조물 기능 | 지붕, 입구, 부속 설비, 사거리 표시 |

색상만으로 역할이나 진영을 구분하지 않는다. 흑백 실루엣에서도 핵심 역할이 남아야 한다.

## 4. 화면 크기 계약

각 자산은 콘셉트 원화 크기가 아니라 실제 플레이 크기로 평가한다.

예시 항목:

```text
기준 출력 해상도
내부 논리 해상도
일반 유닛 화면 높이
대형 유닛 화면 높이
건물 최대 화면 크기
최소 전략 줌
최대 동시 겹침 수
```

- 작은 전략 게임은 24~64px 구간에서 무기·자세·몸통 덩어리가 먼저 읽혀야 한다.
- 크기 차이는 역할 정보와 충돌하지 않게 한다.
- 상위 등급을 단순 확대만으로 표현하면 대형 병종의 의미가 무너진다.
- UI 프레임, 체력바, 전조 VFX를 함께 올린 상태에서 검증한다.

## 5. 팔레트와 재질 계약

- 핵심 팔레트와 보조 팔레트를 분리한다.
- 동일 재질은 자산마다 다른 명암 규칙을 사용하지 않는다.
- 강한 발광은 상시 기본값이 아니라 스킬·등급·위험 순간에 제한한다.
- 배경과 캐릭터의 명도·채도 우선순위를 정한다.
- 금속, 천, 피부, 갑각, 마법, 나무, 돌처럼 반복 재질의 하이라이트 위치와 단계 수를 통일한다.
- 색각 이상과 흑백 테스트를 통과하도록 형태·문양·프레임을 함께 사용한다.

## 6. 등급과 성장 표현

등급 상승은 다음 순서로 확장한다.

```text
기본 역할 실루엣 유지
→ 부속 1개 추가
→ 무기·장비의 기능 강조
→ 고유 문양·짧은 연출
→ 전용 키포즈 또는 상태 표현
```

- 일반 등급부터 최종 보스처럼 보이지 않게 한다.
- 상위 등급은 장식이 아니라 역할의 강화로 읽혀야 한다.
- 등급 VFX를 제거해도 실루엣 차이가 남아야 한다.
- 같은 등급 여러 기가 모였을 때 화면이 빛 덩어리가 되지 않게 한다.

## 7. 건물과 환경 제작 계약

건물은 최소 다음 상태를 고려한다.

```text
계획 또는 배치 미리보기
건설 중
활성
업그레이드 중
비활성
파손
폐허
```

- 기능은 지붕, 입구, 부속 설비, 굴뚝, 깃발, 사격부처럼 큰 구조로 구분한다.
- 작은 장식이나 간판 문자만으로 기능을 설명하지 않는다.
- 길, 이동 영역, 상호작용 지점과 시각적 충돌이 없어야 한다.
- 환경은 전투 정보를 돋보이게 하며 캐릭터보다 강한 대비를 상시 사용하지 않는다.

## 8. 생성형 콘셉트 사용

허용:

- 스타일 후보 비교.
- 실루엣·재질·조명 무드 탐색.
- 건물 성장 구조와 병종 장비 아이디어.

필수 후처리:

- 손, 관절, 무기 방향 검수.
- 가짜 문자 제거.
- 팔레트와 광원 통일.
- 실제 픽셀 또는 제작 그리드 재작성.
- 실제 게임 크기 축소 테스트.
- 기존 IP와 유사한 고유 디자인 제거.

금지:

- 생성 이미지를 수정 없이 최종 자산으로 사용.
- 출처와 권리를 확인하지 않은 스타일 모방.
- 미세 장식만 보고 게임 화면 가독성을 판단.

## 9. 아트 제작 산출물

아트 책임 문서는 최소 다음을 포함한다.

- 한 문장 스타일 정의.
- 기준 해상도와 화면 크기 표.
- 진영·병종·등급 형태 언어.
- 팔레트·재질·광원 규칙.
- 유닛과 건물 상태 목록.
- 애니메이션 프레임 예산.
- VFX와 UI의 겹침 제한.
- 파일명, 아틀라스, 방향 수, 피벗 후보.
- 제작 우선순위.
- 실제 게임 삽입 검증 절차.
- 자동 탈락 조건.

## 10. 검수 순서

```text
흑백 실루엣
→ 실제 크기 축소
→ 같은 병종 다수 겹침
→ 배경과 HUD 위 배치
→ 상태·등급 비교
→ 애니메이션 키포즈
→ VFX·조명 추가
→ 저해상도·색각·성능 검증
```

먼저 화려한 일러스트를 완성한 뒤 게임 화면에 맞추려 하지 않는다.

## 11. 자동 탈락 예시

- 역할을 색상 외에는 구분할 수 없음.
- 무기 방향과 공격 방향이 다름.
- 실제 화면 크기에서 얼굴이나 미세 장식만 남음.
- 상시 발광이 위험 전조와 등급 연출을 가림.
- 건물 상태가 바뀌면 원래 기능을 알아볼 수 없음.
- 같은 병종 여러 기가 겹치면 하나의 덩어리로 보임.
- 생성형 가짜 문자나 잘못된 관절이 남아 있음.
- 아트 문서에 실제 화면 검증 방법이 없음.

## 12. 완료 기준

- 핵심 역할을 실제 플레이 크기에서 1초 안에 구분한다.
- 진영과 등급이 색상 없이도 일정 수준 구분된다.
- 자산 제작자가 추가 질문 없이 기본·상태·등급 변형을 만들 수 있다.
- UI, VFX, 배경과 함께 놓아도 전투 정보가 유지된다.
- 예산과 일정에 맞는 프레임·방향·변형 수가 명시된다.
- 검수 실패 시 어떤 단계로 되돌아갈지 정해져 있다.
