# 기획 조사·벤치마킹·근거 관리 방법

- 상태: 공용 방법
- 목적: 자료를 많이 모으는 조사보다, 현재 의사결정에 필요한 근거를 빠르게 찾고 적용·제외 결론까지 남긴다.

## 1. 조사 시작 질문

조사 전에 다음을 한 문장으로 작성한다.

```text
우리는 ___을 결정하기 위해 ___한 사례와 근거가 필요하다.
```

함께 적을 항목:

- 결정할 내용.
- 현재 가설.
- 대상 사용자·플랫폼·장르.
- 비용·일정·기술 제약.
- 필요한 최신성.
- 조사 종료 조건.
- 결과를 반영할 책임 문서.

질문 없이 자료부터 모으면 흥미로운 사례는 늘어나지만 결론이 흐려진다.

## 2. 자료 종류를 구분한다

| 자료 | 적합한 용도 |
|---|---|
| 공식 문서·개발자 발표 | 실제 기능, 기술 구조, 의도 확인 |
| 게임·제품 직접 사용 | 입력 흐름, 반응, 피로, 가독성 확인 |
| 패치노트·변경 이력 | 실패·개선 방향과 운영 변화 확인 |
| 사용자 리뷰·포럼 | 반복 불만, 기대, 오해 지점 수집 |
| 영상·스크린샷 | 화면 구성과 동작 관찰, 실제 판정 확인에는 한계 |
| 연구·전문 서적 | 인지, 접근성, UX, 시각 원리의 일반 근거 |
| 에셋·툴 판매 페이지 | 제작 범위, 기능 패키징, 시장 표현 참고 |
| 공개 저장소 | 구조, 테스트, 문서화, 자동화 사례 확인 |

한 종류의 자료만으로 결론을 확정하지 않는다. 공식 설명과 실제 사용 경험이 다를 수 있다.

## 3. 출처 우선순위

```text
1차 출처와 직접 관찰
→ 신뢰도 높은 분석·연구
→ 다수 사용자 반응
→ 단일 후기·요약·2차 인용
```

- 변경 가능성이 있는 정보는 확인 날짜를 남긴다.
- 검색 결과 요약만 보고 결론을 쓰지 않는다.
- 정확한 수치와 규칙은 원문을 확인한다.
- 출처가 없거나 재현하지 못한 주장은 `가설`로 표시한다.

## 4. 벤치마킹 분해

사례를 통째로 “좋다”라고 평가하지 않고 다음 축으로 나눈다.

- 핵심 사용자 행동.
- 판단해야 하는 정보.
- 입력 단계와 마찰.
- 즉시 피드백.
- 실패와 복구.
- 반복 플레이 피로.
- 화면 가독성.
- 제작 비용.
- 기술·성능 제약.
- 수익·운영 구조가 경험에 미치는 영향.
- 현재 프로젝트와 다른 전제.

## 5. 기록 표

```md
| 사례 | 관찰 근거 | 해결한 문제 | 강점 | 한계 | 현재 프로젝트 적용 | 검증 |
|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |
```

적용 결론은 다음 중 하나로 표시한다.

- `반드시 반영`
- `PoC 검증`
- `조건부 참고`
- `제외`
- `추가 조사`

## 6. 원리와 표면을 분리한다

예시:

```text
표면: 특정 게임의 원형 미니맵
원리: 플레이어가 짧은 시간 안에 멀리 떨어진 위협을 파악해야 함
```

현재 화면에서 전장이 한눈에 보인다면 미니맵을 복제할 이유가 없다. 원리를 만족하는 더 단순한 방식이 있는지 먼저 찾는다.

또 다른 예시:

```text
표면: 적군 전용 병종 데이터 10개
원리: 진영이 다르게 보이면서 역할과 위협은 비교 가능해야 함
```

같은 전투 규칙을 공유할 수 있다면 공용 병종 데이터와 진영별 시각 세트가 더 적합할 수 있다.

## 7. 비용·권리·윤리 확인

- 코드, 아트, 사운드, 문구, UI 배치를 직접 복제하지 않는다.
- 라이선스와 사용 권한을 확인한다.
- 생성형 도구를 사용한 자료는 최종 자산 여부와 후처리 책임을 구분한다.
- 유료 자료, 로그인 정보, 쿠키, 개인 데이터가 필요한 조사 방법은 권한을 먼저 확인한다.
- 비공개 프로젝트 사례를 Base에 올릴 때는 공개 가능한 수준으로 일반화한다.

## 8. 조사 종료 조건

다음이 준비되면 조사를 멈추고 결정을 작성한다.

- 문제와 제약이 명확하다.
- 서로 다른 접근 2개 이상을 비교했다.
- 채택안과 제외안의 이유가 있다.
- 현재 프로젝트에 맞춘 최소 구현 또는 PoC가 있다.
- 검증 방법이 있다.
- 새로운 사례를 더 찾아도 결론이 크게 바뀌지 않는다.

자료 수 자체를 완료 기준으로 사용하지 않는다.

## 9. 사례로 전환하는 방법

조사가 끝난 뒤 재사용 가치가 있으면 `docs/knowledge/cases/`에 사례를 남긴다.

```text
관찰한 사례
→ 해결한 문제
→ 프로젝트 제약
→ 채택·제외 결정
→ 실제 결과
→ 재사용 가능한 원칙
```

아직 결과가 없다면 `가설` 상태로 남기고 검증 전 보편적 방법으로 승격하지 않는다.

## 10. 정보 갱신

- 외부 도구, 엔진, 플랫폼, 시장 정보는 확인 날짜를 기록한다.
- 링크가 사라져도 의사결정 근거를 이해할 수 있도록 핵심 관찰을 요약한다.
- 오래된 수치와 기능은 최신 원본을 다시 확인한다.
- 새로운 결과가 기존 사례를 뒤집으면 활성 문서를 갱신하고 이전 내용은 Git 이력에 남긴다.

## 11. 완료 보고 형식

```md
## 조사 결론
- 결정할 문제:
- 확인한 근거:
- 반드시 반영:
- PoC 검증:
- 제외:
- 비용·권리 위험:
- 반영할 문서:
- 다음 검증:
```
