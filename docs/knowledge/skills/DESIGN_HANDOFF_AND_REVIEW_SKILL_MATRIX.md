# 기획 인수인계·검수 실무 스킬 매트릭스

- 상태: 공용 스킬 계약
- 목적: 긴 대화와 다수 문서를 실행 가능한 최신 컨텍스트로 정리하고, 새 작업자가 잘못된 가정을 하지 않도록 한다.

## 1. 책임 원본 식별

필요 능력:

- 같은 주제를 다루는 문서를 찾고 최신 책임 원본을 선택.
- 승인, PoC, 미확정, 구현, 검증 상태 구분.
- 폐기된 안과 역사적 기록을 활성 경로에서 제거.
- 파일 경로, Issue, Goal, 실제 코드 관계 확인.

산출물:

- 문서 책임표.
- 중복·충돌 목록.
- 삭제·통합·갱신 계획.

실패:

- 제목의 버전 숫자만 보고 최신본을 판단.
- Issue는 갱신했지만 GDD·로드맵·활성 컨텍스트는 방치.

## 2. 컨텍스트 압축

필요 능력:

- 전체 원문을 복사하지 않고 실행에 필요한 결정만 보존.
- 숫자, 단위, 경로, 상태, 승인 게이트를 정확히 유지.
- 세부 내용은 책임 문서로 링크.
- 역사적 이유와 현재 규칙을 분리.

좋은 컨텍스트:

```text
현재 단계
핵심 경험
불변 조건
데이터 소유
다음 실행 순서
미확정
읽을 문서
금지 범위
```

줄여도 되는 것:

- 반복 설명.
- 폐기된 대안의 장문.
- 이미 책임 문서에 있는 전문.
- 도구 호출 기록.

## 3. 실행 가능한 로드맵

필요 능력:

- 각 Phase의 목적, 선행 조건, 산출물, 완료 기준 작성.
- 큰 단계와 현재 수직 슬라이스 범위 구분.
- 설계·구현·검증·콘텐츠 확장 순서 분리.
- 승인 게이트와 중단 조건 표시.

실패:

- “개발 진행”, “퀄리티 개선”처럼 관찰할 수 없는 단계.
- 모든 시스템을 병렬로 시작.
- 플레이테스트 전 최종 수치를 확정된 것으로 기록.

## 4. 문서 간 동기화

최신 결정이 생기면 최소 다음을 확인한다.

- 전체 기획서.
- 활성 컨텍스트 또는 인수인계 문서.
- 로드맵.
- 주제별 승인 책임 문서.
- 현재 Issue·Goal.
- 문서 라우터.
- 기술·데이터 구조 문서.

모든 문서에 전문을 복사하지 않고 필요한 수준의 요약과 링크만 갱신한다.

## 5. 콜드 스타트 검수

새 작업자가 문서만 보고 다음을 답하게 한다.

- 무엇을 만들고 있는가?
- 지금 구현해도 되는가?
- 다음 작업은 무엇인가?
- 어떤 구조가 승인되었는가?
- 어떤 수치가 조정 가능한가?
- 데이터 원본은 어디인가?
- 어떤 파일을 먼저 읽어야 하는가?
- 절대 다시 구현하면 안 되는 폐기안은 무엇인가?

오답은 작업자의 문제가 아니라 문서 구조의 결함으로 먼저 본다.

## 6. 완료 기준 작성

완료 기준은 다음 형식을 우선한다.

```text
조건
→ 행동
→ 관찰 가능한 결과
```

예시:

- 동일한 데이터 ID와 입력 시 같은 결과가 재현된다.
- 실제 화면 크기에서 역할을 1초 안에 구분한다.
- 미니맵 없이 핵심 목표 상태를 지정 시간 안에 찾는다.
- 누락된 자산 참조가 실행 전에 검출된다.

## 7. 위험 보고

필수 위험 분류:

- 제품 방향 위험.
- 데이터 중복·마이그레이션 위험.
- 성능 위험.
- UI·가독성 위험.
- 자산 제작 비용.
- 외부 의존성과 권리.
- 테스트하지 못한 항목.

위험을 적는 것만으로 끝내지 않고 예방 조치와 검증 방법을 연결한다.

## 8. 완료 기준

- 최신 사용자 결정이 책임 문서·로드맵·Issue에 일치한다.
- 새 작업자가 10분 안에 방향과 다음 행동을 설명한다.
- 승인·가설·미확정·구현·검증 상태가 혼동되지 않는다.
- 중복 데이터와 문서가 제거되거나 명시적 책임 관계를 가진다.
- 모든 다음 작업에 선행 조건과 관찰 가능한 완료 기준이 있다.
- 테스트하지 않은 내용을 완료로 보고하지 않는다.

## 9. 작업 프롬프트에 포함할 항목

```md
- 이번 최신 사용자 결정
- 확인할 책임 문서
- 갱신할 문서와 Issue
- 삭제·통합할 중복 정보
- 보존할 Git 이력·아카이브 규칙
- 콜드 스타트 질문
- 완료 기준
- 미검증·남은 위험
```
