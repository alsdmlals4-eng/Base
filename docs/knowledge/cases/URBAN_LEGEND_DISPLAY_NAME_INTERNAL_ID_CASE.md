# 사례 — 플레이어 표시명과 내부 영속 ID를 분리해 안전하게 리네이밍하기

- 출처 프로젝트: urban-legend / 괴이 기록국
- 상태: 구현·저장 호환 검증
- 주제: 안내 캐릭터의 정체성을 바꾸면서 기존 저장·파일·튜토리얼 키를 보존

## 1. 문제

프로젝트의 안내 캐릭터가 기능성 이름에서 세계관에 맞는 고유 이름으로 바뀌었다. 화면 문구만 바꾸면 일부 구명칭이 남고, 내부 문자열을 일괄 치환하면 다음 위험이 있었다.

- 기존 저장 키와 이벤트 확인 상태가 깨짐.
- 파일·리소스 경로와 코드 참조가 끊김.
- 과거 데이터의 화자 ID를 읽지 못함.
- 문서가 플레이어 명칭과 내부 식별자를 혼용함.

## 2. 채택한 결정

- UI·대사·기획 문서에는 새 플레이어 표시명을 사용.
- 내부 화자 ID, 저장 키, 파일명은 호환 계층으로 유지.
- 표시명 변환을 한 곳에서 책임지게 함.
- 새 문서에는 구명칭을 공식 용어로 정의하지 않음.
- 내부 키 이관은 필요성과 회귀 범위를 별도 작업으로 심사.

## 3. 제외한 접근

- 저장소 전체 문자열 일괄 치환.
- 새 이름과 새 저장 버전을 동시에 도입.
- UI마다 개별 조건문으로 이름 변환.
- 기존 데이터 파일을 즉시 재작성하고 호환 로더를 제거.

## 4. 검증

- 이전 저장을 불러와 안내 확인 상태와 진행이 유지되는지 확인.
- 과거 화자 ID가 새 표시명으로 정상 노출되는지 확인.
- 검색으로 플레이어 화면에 구명칭이 남지 않는지 확인.
- 새 저장과 이전 저장의 왕복에서 동일 이벤트가 중복 재생되지 않는지 확인.

## 5. 결과

캐릭터의 플레이어 정체성을 바꾸면서 영속 데이터와 경로의 안정성을 유지했다. 리네이밍을 콘텐츠 변경과 데이터 마이그레이션으로 분리한 것이 핵심이었다.

## 6. 재사용 가능한 원칙

- 사용자에게 보이는 이름과 영속 식별자는 수명이 다르다.
- 표시 계층과 저장·데이터 계층을 분리한다.
- 이름 변경과 스키마 변경을 한 번에 수행하지 않는다.
- 공식 명칭·내부 키·레거시 별칭을 문서에 명시한다.
- 이관은 검색, 저장 왕복, 이벤트 재발동까지 검증한다.

## 7. 적용 시 주의

- 외부 API·분석·로컬라이제이션 키가 구 ID에 의존할 수 있다.
- 개인정보·법적 명칭·브랜드 변경은 별도 감사와 완전 이관이 필요할 수 있다.
- 내부 키를 영구 유지할지 단계적으로 폐기할지 종료 조건을 정한다.
