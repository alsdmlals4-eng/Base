# 사례 — 생성 이미지와 제품 의미 텍스트를 분리하기

- 출처 프로젝트: urban-legend / 괴이 기록국
- 상태: 자산 제작·엔진 로드 검증
- 주제: 생성형 이미지의 시각 탐색 장점을 사용하면서 문자·UI·현지화 위험을 줄임

## 1. 문제

캐릭터 카드와 장면 이미지를 생성형 도구로 만들 때 이미지 안에 이름, 소속, 능력, 대사를 함께 넣으면 다음 문제가 반복됐다.

- 한글 오탈자와 의미 없는 글자 생성.
- 같은 캐릭터의 카드마다 명칭과 서체가 달라짐.
- 수정·현지화·접근성 대응이 어려움.
- 이미지 해상도와 화면 크롭에 따라 텍스트가 흐려짐.
- 시각 자산과 UI 상태가 한 파일에 결합됨.

## 2. 채택한 결정

- 생성 이미지는 텍스트 없는 캐릭터·배경·행동 컷으로 제작.
- 이름, 소속, 능력, 대사, 버튼은 엔진 UI에서 별도 렌더링.
- 캐릭터 정체성은 레퍼런스 카드와 디자인 명세로 관리.
- 원본 출력과 알파·크롭·색 보정 파생 파일을 분리.
- manifest에 크기, 알파, 참조, 프롬프트 목적, 텍스트 없음, 로드 검증을 기록.
- 실제 화면 슬롯에서 얼굴·실루엣·여백을 검증.

## 3. 제외한 접근

- 생성 이미지 안의 글자를 수작업으로 조금만 수정해 사용.
- 카드 전체를 하나의 이미지로 만들어 UI를 대체.
- 프롬프트만으로 캐릭터 정체성과 자산 규격을 관리.
- 원본을 덮어쓰는 비가역 후처리.

## 4. 검증

- 모든 게임용 이미지에 구운 의미 텍스트가 없는지 확인.
- manifest와 실제 파일의 크기·알파·경로 일치 확인.
- 엔진에서 자산 로드와 투명 가장자리 확인.
- UI 텍스트를 독립적으로 수정·번역할 수 있는지 확인.
- 실제 화면 크롭에서 얼굴·손·핵심 소품이 유지되는지 확인.

## 5. 결과

시각 콘셉트와 캐릭터 자산을 빠르게 확장하면서도 정확한 문구, 레이아웃, 현지화, 접근성을 제품 UI가 책임하게 됐다. 이미지 생성 결과의 문자 품질에 의존하지 않게 됐다.

## 6. 재사용 가능한 원칙

- 이미지 모델은 시각 형태와 분위기를, 제품 UI는 의미 텍스트와 상호작용을 담당한다.
- 캐릭터 정체성은 프롬프트가 아니라 디자인 카드와 manifest로 관리한다.
- 원본 출력과 파생 자산을 분리해 재처리 가능성을 보존한다.
- 생성형 자산은 실제 화면 크기와 로드 조건으로 검수한다.

## 7. 적용 시 주의

- 포스터·간판처럼 이미지 자체의 문자 디자인이 핵심이면 수작업 타이포그래피와 별도 검수가 필요하다.
- 참조 이미지와 스타일 사용 권리, 라이선스, 공개 가능성을 확인한다.
- 생성 결과를 최종 자산으로 쓰는지 콘셉트로만 쓰는지 프로젝트에서 명시한다.
