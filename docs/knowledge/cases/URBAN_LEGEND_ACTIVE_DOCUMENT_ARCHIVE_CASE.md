# 사례 — 현행 책임 문서와 날짜별 백업을 분리해 기본 읽기 비용 줄이기

- 출처 프로젝트: urban-legend / 괴이 기록국
- 상태: 문서 운영 검증
- 주제: 과거 근거를 보존하면서 새 작업자가 오래된 계획을 재검토하지 않게 함

## 1. 문제

장기간 작업 뒤 README, 로드맵, 프로젝트 브리프, 체크리스트, 과거 기획서에 같은 상태가 서로 다르게 남았다.

- 완료된 작업이 아직 다음 작업처럼 표시됨.
- 구명칭과 이전 주인공·시스템 설명이 검색 결과에 노출됨.
- 완료 QA와 과거 승격 후보를 매 작업마다 읽음.
- 어떤 문서가 현재 원본인지 판단하기 어려움.
- 과거 문서를 삭제하면 결정 근거와 외부 링크를 잃을 위험이 있음.

## 2. 채택한 결정

- 현재 구현과 승인 계획을 구분하는 짧은 상태 원본 추가.
- 작업 조건에 따라 필요한 문서만 고르는 문서 라우터 작성.
- 한 주제에 현행 책임 원본 하나만 유지.
- 완료·중복·저사용 정보는 날짜별 archive로 이동.
- 외부 링크 파손 가능성이 큰 경로는 짧은 리디렉션 문서로 유지.
- 백업 인덱스에 원래 경로, 보관 이유, 현행 대체 문서, Git 원문 경로 기록.
- 기본 읽기에서 완료 Goal·QA·벤치마크·archive 제외.

## 3. 제외한 접근

- 모든 과거 문서를 삭제.
- 모든 버전을 활성 폴더에 유지.
- 하나의 거대한 요약 문서에 모든 전문을 복사.
- 버전 번호가 가장 높은 문서를 자동 원본으로 판단.
- 과거 QA를 현재 구현 확인으로 사용.

## 4. 검증

- 새 담당자가 최소 문서만 읽고 현재 구현과 다음 계획을 구분하는지 확인.
- README와 상태 문서에서 분야별 책임 원본으로 이동 가능한지 확인.
- 구명칭·오래된 다음 작업·폐기 경로 검색.
- 백업 파일과 Git 이력에서 정리 전 원문을 찾을 수 있는지 확인.
- 활성 문서 간 현재 버전·로드맵·상태 문구 비교.

## 5. 결과

과거 근거를 잃지 않으면서 기본 읽기 문서 수와 중복 확인을 줄였다. 상태, 방향, 실행, 검증, 역사 기록의 책임이 분리돼 새 작업의 출발점이 명확해졌다.

## 6. 재사용 가능한 원칙

- 역사 보존과 활성 컨텍스트 최소화는 동시에 가능하다.
- 문서 수보다 책임 원본과 라우팅이 중요하다.
- 구현 사실, 승인 계획, 진행 중, 백업 상태를 구분한다.
- 리디렉션은 외부 링크를 보존하지만 현행 원본으로 취급하지 않는다.
- 인수인계 품질은 작성자 설명 없이 새 담당자가 정확히 행동하는지로 검증한다.

## 7. 적용 시 주의

- 규제·계약·감사가 필요한 문서는 Git 이력만으로 충분하지 않을 수 있다.
- 자동화와 외부 링크가 오래된 경로에 의존하면 명시적 마이그레이션이 필요하다.
- archive가 또 다른 미정리 폴더가 되지 않도록 인덱스와 보관 기준을 유지한다.
- 대규모 팀에서는 문서 소유자·승인자·검토 주기를 추가한다.
