# 사례 — 규칙·UI·연출·QA를 같은 의미 단계로 추적하기

- 출처 프로젝트: 십보강호
- 확인 날짜: 2026-07-16
- 작성 상태: 문서 구조 채택·구현 전 검증 필요
- 주제: 같은 판정과 성장 결과를 규칙서, UI 명세, 연출 기획서와 테스트에서 서로 다른 표현으로 왜곡하지 않고 추적 가능하게 유지

## 1. 문제

복합 시스템을 여러 문서로 나누면 각 문서가 자신의 관점에서 같은 사건을 다시 설명하게 된다. 이때 다음 문제가 자주 발생한다.

- 규칙의 처리 순서와 화면의 정보 순서가 달라 결과 원인을 이해하기 어렵다.
- 연출 문서가 도메인 결과를 다시 계산하거나 별도 수치를 만든다.
- UI 명세와 연출 기획서가 화면 흐름·카메라·상태 소유를 중복 책임진다.
- 테스트가 기능 존재만 확인하고 사용자가 결과 이유를 이해했는지는 검증하지 않는다.
- 한 문서를 수정해도 다른 문서의 낡은 설명이 남는다.

## 2. 맥락과 제약

- 규칙, UI, 연출, 아키텍처와 QA는 서로 다른 질문을 책임해야 한다.
- 같은 규칙의 전문을 여러 문서에 복사하지 않는다.
- 화면·애니메이션·음향은 결과를 표현하지만 판정·보상·저장을 결정하지 않는다.
- 연출을 줄이거나 건너뛰어도 동일한 결과와 원인 기록을 확인할 수 있어야 한다.
- 새 작업자는 과거 대화 없이 책임 원본과 검증 경로를 찾아야 한다.

## 3. 검토한 대안

### A. 하나의 거대한 기획서가 전부 책임

장점:

- 검색 위치가 하나다.
- 초기 작성은 빠르다.

단점:

- 규칙·레이아웃·카메라·데이터·QA가 섞인다.
- 수정 영향이 불분명해지고 구현 사실과 승인 계획이 혼동된다.
- 특정 분야 작업자가 필요한 정보만 찾기 어렵다.

### B. 문서를 나누되 같은 전문을 복사

장점:

- 각 문서만 읽어도 문맥을 얻기 쉽다.

단점:

- 한쪽만 고쳐도 충돌이 생긴다.
- 어떤 문장이 현행 원본인지 판별하기 어렵다.
- 테스트가 낡은 복사본을 기준으로 작성될 수 있다.

### C. 질문별 책임 원본을 분리하고 같은 의미 단계와 참조를 공유

장점:

- 각 문서는 자신의 판단만 책임한다.
- 한 판정의 규칙→이벤트→표현→검증 경로를 추적할 수 있다.
- 문서 간 전문 복사 없이도 순서와 용어를 맞출 수 있다.

채택: C.

## 4. 채택한 구조

```text
규칙·도메인 원본
- 조건
- 처리 순서
- 결과
- 원인 목록

↓ 구조화 이벤트·의미 키

UI 원본
- 어떤 정보를 어디에 상시 표시하는가
- 입력·잠금·사용 불가 상태

연출 원본
- 어떤 순서로 강조·재생하는가
- 카메라·VFX·오디오·접근성 폴백

QA 원본
- 사용자가 무엇을 보고 결과 이유를 설명하는가
- 연출 감소·건너뛰기·자산 누락에도 결과가 같은가
```

각 층은 같은 의미 단계를 사용하지만 전문은 복사하지 않는다. 예를 들어 도메인이 `이동 → 공격 성립 확인 → 충돌 판정 → 피해`를 책임하면, UI는 현재 위치와 원인 로그를 보여주고, 연출은 같은 순서로 재생하며, QA는 사용자가 그 순서를 설명할 수 있는지 확인한다.

## 5. 상태 소유 경계

### 도메인이 소유

- 위치·자원·조건·판정·피해·보상·진행·저장.
- 결과를 만든 이유와 변경 전후 상태.

### 프레젠테이션이 소유

- 자산 선택, 강조, 카메라, 애니메이션, 음향, 텍스트와 입력 잠금.
- 자산 부재와 접근성 옵션의 폴백.

### 프레젠테이션이 소유하면 안 되는 것

- 애니메이션 종료 시 피해를 새로 적용.
- 컷인 재생 여부에 따라 보상 변경.
- UI 문구가 실제 난도·관계·저장 상태를 결정.
- 연출용 임시 숫자를 규칙 원본으로 확정.

## 6. 문서 동기화 절차

1. 변경 질문의 책임 원본을 문서 지도에서 찾는다.
2. 규칙·상태 변화가 바뀌면 구조화 결과와 원인 키를 먼저 갱신한다.
3. UI는 표시 위치와 입력 상태만, 연출은 재생 순서와 폴백만 갱신한다.
4. 로드맵과 테스트에 새 완료 기준을 연결한다.
5. Active Context에는 현재 상태와 다음 승인 게이트만 압축한다.
6. 낡은 용어·순서·복사된 전문을 정적 검색한다.
7. 새 작업자가 원본과 검증 경로를 찾는 콜드 스타트 검수를 수행한다.

## 7. 검증 방법

- 한 사건을 골라 규칙 원본, 이벤트 원인, UI 표시, 연출 순서와 테스트가 동일한 의미 단계를 사용하는지 추적한다.
- 연출을 건너뛰거나 자산을 제거해도 결과·보상·저장 상태가 같은지 확인한다.
- 사용자가 로그 없이 성공·실패의 주요 이유를 말하고, 상세 로그에서 계산 원인을 확인하는지 본다.
- UI·VFX·애니메이션 코드가 결과 상태를 직접 변경하지 않는지 정적 검사한다.
- 새 작업자가 10분 안에 현재 방향, 책임 문서, 다음 작업과 검증 경로를 찾는지 확인한다.

## 8. 위험과 실패 징후

- 모든 문서가 같은 표를 복사해 한 문장 수정에도 여러 파일을 고쳐야 한다.
- UI가 규칙에 없는 예외를 만들어 사용 가능 여부를 결정한다.
- 연출용 공방 횟수만큼 실제 판정이 반복된다.
- QA가 화면 존재만 체크하고 결과 이해를 관찰하지 않는다.
- 문서 상태가 `승인 계획`인데 구현 완료로 보고한다.

## 9. 재사용 가능한 원칙

- 한 사건의 처리 순서와 사용자에게 설명하는 순서를 가능한 한 연결한다.
- 문서 분리는 내용 장르가 아니라 질문과 상태 소유 책임으로 한다.
- 문서 사이에는 전문보다 원본 링크, 의미 키와 관찰 가능한 완료 기준을 공유한다.
- 연출은 결과를 더 잘 이해하고 기억하게 하지만 결과 자체를 만들지 않는다.
- 테스트는 내부 계산 일치와 사용자 설명 가능성을 함께 본다.

## 10. 그대로 복사하면 안 되는 요소

- 특정 프로젝트의 전투 단계, 능력명, 수치와 데이터 필드명.
- 특정 엔진의 파일 경로와 클래스 이름.
- 특정 UI 레이아웃과 카메라 지속 시간.

단순 선형 콘텐츠처럼 규칙과 표현의 분리가 거의 없는 경우에는 이 구조를 모두 만들 필요가 없다. 다만 저장·보상·관계·퀘스트처럼 장기 상태가 있는 프로젝트에서는 상태 소유 경계를 생략하지 않는다.
