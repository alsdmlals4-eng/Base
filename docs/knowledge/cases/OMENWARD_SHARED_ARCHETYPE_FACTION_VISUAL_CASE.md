# 사례 — 공용 병종 데이터와 진영별 시각 세트

- 출처 프로젝트: OMENWARD
- 상태: 채택 / 구현 전
- 주제: 동일 역할을 가진 양 진영 유닛의 데이터 중복 제거

## 1. 문제

초기 기획에서는 아군 10병종과 기능상 대응하는 적군 10병종을 별도 데이터·능력·애니메이션 대상으로 다뤘다.

이 방식은 다음 비용을 만들었다.

- 같은 역할과 수치를 두 번 수정해야 함.
- 밸런스가 조금씩 어긋나기 쉬움.
- 애니메이션 상태와 판정 이벤트를 두 벌 관리해야 함.
- 문서에 “대칭”과 “차별화” 규칙이 동시에 존재해 구현자가 별도 클래스를 만들 가능성.
- 자산 수가 10종이 아니라 20종의 독립 병종처럼 증가.

## 2. 제약

- 양 진영은 화면에서 확실히 다르게 보여야 함.
- 적군은 룰렛이 아니라 웨이브로 출격함.
- 역할과 카운터는 같은 기준으로 읽혀야 함.
- 소규모 팀이 데이터·코드·애니메이션 중복을 감당하기 어려움.

## 3. 검토한 대안

### A. 완전 별도 병종 20개

장점:

- 진영별 개성과 AI 차이를 크게 만들 수 있음.

단점:

- 데이터, 스킬, 테스트, 애니메이션, 밸런스 비용이 두 배에 가까워짐.
- 초기 PoC에서 핵심 재미보다 콘텐츠 제작량이 앞섬.

### B. 동일 데이터와 동일 이미지

장점:

- 가장 저렴하고 일관적.

단점:

- 진영 구분과 세계관 몰입이 약함.

### C. 공용 병종 데이터 + 진영별 시각 세트

장점:

- 전투 규칙과 애니메이션 타이밍을 한 번만 관리.
- 아군·적군 비교가 명확.
- 진영 차이는 이미지·실루엣·팔레트에서 유지.
- 적 웨이브는 같은 archetype ID를 참조해 재사용 가능.

채택: C.

## 4. 결정 구조

```text
Shared Unit Archetype
- 능력치
- 역할 태그
- 스킬·패시브
- 이동·타기팅
- 애니메이션 상태와 이벤트 타이밍

Runtime Unit Instance
- archetype_id
- owner/team
- tier/rank
- visual_faction_id

Faction Visual Set
- 스프라이트·모델
- 초상화·아이콘
- 팔레트·재질
- 동일 프레임 레이아웃과 피벗
```

적 웨이브는 별도 `EnemyUnitProfile`이 아니라 공용 `archetype_id`와 소유 진영을 지정한다.

## 5. 중요한 경계

공유한다:

- HP·공격력·사거리·이동속도.
- 스킬과 패시브 규칙.
- 공격 준비·판정·회복 시간.
- 상태 ID와 애니메이션 이벤트.
- 등급·Tier 적용 방식.

분리한다:

- 팀과 적대 관계.
- 출격 방식과 웨이브 구성.
- 화면 이미지·초상화·아이콘.
- 진영 이름과 설명 문구.

첫 구현에서는 이미지 외의 VFX·오디오도 가능하면 공유하고, 실제 가독성 문제가 확인될 때만 진영별 표현 override를 추가한다.

## 6. 검증 방법

- 공용 archetype 하나를 아군·적군으로 각각 생성해 능력치와 공격 결과가 동일한지 확인.
- 같은 공격 이벤트 프레임에 양쪽 이미지가 정확히 접촉·발사하는지 확인.
- 진영별 시각 세트의 상태·프레임 수·피벗이 일치하는지 정적 검증.
- 웨이브 데이터에 별도 적군 스탯 복사본이 없는지 검색.
- 한쪽 archetype 수치를 수정했을 때 양 진영에 즉시 반영되는지 확인.
- 팀·소유권 변경이 시각 세트와 타깃 판정만 바꾸고 전투 데이터는 바꾸지 않는지 확인.

## 7. 재사용 가능한 원칙

- 규칙과 역할이 같고 표현만 다르면 데이터보다 시각 레이어를 분리한다.
- “진영이 다르다”는 이유만으로 별도 클래스·스탯·스킬을 만들지 않는다.
- 공용 애니메이션 이벤트 계약을 만들면 스킨과 진영 변형을 싸게 확장할 수 있다.
- 적 출격 방식과 전투 병종 데이터는 별개의 책임이다.
- 차별화가 필요하면 먼저 편성, 시각, 상황, 소유권에서 만들고 데이터 복제는 마지막에 검토한다.

## 8. 그대로 복사하면 안 되는 조건

다음 경우에는 완전 공유가 부적합할 수 있다.

- 체형과 무기 궤적이 크게 달라 같은 판정 타이밍을 쓸 수 없음.
- 진영별 스킬과 AI가 핵심 재미일 정도로 다름.
- PvP 비대칭성이 제품의 핵심 정체성임.
- 동일 수치가 세계관·가독성·밸런스를 해침.

이 경우에도 공통 인터페이스와 일부 데이터는 공유하고 차이가 필요한 필드만 override하는 방식을 먼저 검토한다.

## 9. 후속 질문

- 이미지 세트가 같은 프레임 수와 피벗을 실제로 지킬 수 있는가?
- 보스와 특수 개체도 공용 archetype 변형으로 충분한가?
- 진영별 VFX·오디오 차이가 필요해지는 시점은 언제인가?
- 시각 세트 누락을 어떤 자동 검사로 막을 것인가?
