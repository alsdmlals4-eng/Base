# 사례 — 표정·컷인·대화 UI가 도메인 상태를 소유하지 않게 분리하기

- 출처 프로젝트: urban-legend / 괴이 기록국
- 상태: 승인 아키텍처·구현 전 검증 필요
- 주제: 프레젠테이션 품질을 높이면서 사건·관계·저장 로직 중복을 방지

## 1. 문제

표정, 컷인, 화면 전환, 대화 UI를 장면별 스크립트에서 직접 처리하면 빠르게 보이지만 다음 문제가 생길 수 있다.

- UI 스크립트가 관계 변화와 보상 지급을 함께 처리함.
- 같은 분기 조건이 대사 데이터와 화면 코드에 중복됨.
- 자산 누락이 장면 진행 실패로 이어짐.
- 연출 비활성·접근성 옵션에서 결과가 달라짐.
- 저장·복귀 중간 상태를 재현하기 어려움.

## 2. 채택한 결정

```text
도메인 로직
- 사건·선택 결과
- 관계·완료 상태
- 보상과 저장

프레젠테이션 요청
- 화자·배치
- 감정 의미 키
- 컷인 유형
- 음향·전환

공용 표현 계층
- 자산 선택과 폴백
- 재생·입력 잠금
- 텍스트·아이콘·VFX
```

- 결과와 상태 변화는 도메인 로직이 결정.
- 서사 데이터는 의미 키와 연출 요청만 제공.
- 공용 표현 계층은 자산을 선택해 재생하고 누락 시 폴백.
- 연출을 끄거나 줄여도 동일한 결과와 저장 상태 유지.

## 3. 제외한 접근

- 컷인 재생 완료 콜백에서 보상 지급.
- 표정 파일명으로 관계 상태 판정.
- 대화 UI가 퀘스트 완료와 저장 버전을 직접 변경.
- 자산이 없을 때 장면 전체를 중단.
- 접근성 옵션에 따라 판정 타이밍이 달라지는 구조.

## 4. 검증 계획

- 표정·컷인 자산을 제거해도 대사·선택·결과가 유지되는지 확인.
- 연출 감소 옵션 전후의 저장 상태와 보상이 동일한지 비교.
- 컷인 중 중복 입력과 메뉴 포커스 충돌이 없는지 확인.
- 저장·불러오기 후 동일 요청에서 결정론적 폴백이 작동하는지 확인.
- UI 코드 검색으로 보상·관계·퀘스트 상태 직접 변경이 없는지 감사.

## 5. 예상 가치

연출 자산과 표현을 확장해도 사건·관계 데이터의 단일 원본을 유지할 수 있다. 자산 제작이 늦거나 접근성 옵션이 활성화돼도 기능 진행을 보존한다.

## 6. 재사용 가능한 원칙

- 프레젠테이션은 상태를 표현하지만 결정하지 않는다.
- 도메인 결과, 의미 기반 연출 요청, 실제 자산 경로를 세 층으로 분리한다.
- 자산 부재와 연출 비활성 상태를 정상 폴백 경로로 설계한다.
- 컷인·전환의 입력 잠금은 일시적 UI 책임이며 진행 결과와 분리한다.

## 7. 적용 시 주의

- 리듬 게임처럼 연출 타이밍 자체가 판정인 장르는 도메인과 프레젠테이션의 경계가 다르다.
- 비동기 자산 로딩에서는 요청 취소와 장면 이탈 처리가 필요하다.
- 실제 구현 결과가 나오기 전까지는 조건부 패턴으로 유지한다.
