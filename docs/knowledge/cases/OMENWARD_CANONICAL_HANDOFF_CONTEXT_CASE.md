# 사례 — 중복 문서를 책임 원본과 인수인계 컨텍스트로 정리하기

- 출처 프로젝트: OMENWARD
- 상태: 문서 운영 검증
- 주제: 다수의 기획 버전·초기 Issue·레거시 명칭을 정리해 새 작업자의 오해를 줄임

## 1. 문제

장기간 기획을 조정하면서 다음이 동시에 남았다.

- 같은 주제의 V1, V2, V3 문서.
- 초기 프로젝트명과 현재 프로젝트명이 섞인 문서.
- 완료된 기획 탐색 Issue와 현재 구현 Issue.
- GDD, 로드맵, Active Context의 서로 다른 상태 설명.
- 실제 책임 원본이 무엇인지 알기 어려운 문서 목록.

이 상태에서는 새 AI나 작업자가 오래된 문서를 먼저 읽고 폐기된 구조를 구현할 위험이 컸다.

## 2. 채택한 결정

- 한 주제당 활성 책임 원본을 하나만 유지.
- 새 결정은 같은 책임 문서를 제자리에서 갱신.
- 일반 이전 버전은 Git 이력으로 보존.
- 외부 납품본이나 큰 방향 전환 비교본만 archive에 저장.
- `DOCUMENTATION_MAP`에서 작업 유형별 책임 문서 라우팅.
- `ACTIVE_CONTEXT`와 별도 handoff 문서에서 현재 방향과 다음 작업 압축.
- 오래된 Issue는 현재 책임 문서 링크와 종료 이유를 남기고 닫음.

## 3. 정리 순서

```text
1. 최신 사용자 결정 수집
2. 주제별 책임 원본 선택
3. 충돌하는 활성 문서 검색
4. 최신 책임 원본 갱신
5. GDD·로드맵·Active Context 동기화
6. Issue·Goal 경로 갱신
7. 중복·레거시 파일 삭제
8. 검색과 콜드 스타트 질문으로 검증
```

## 4. 문서 역할 분리

- 전체 기획서: 시스템 간 관계와 전체 경험.
- 승인 책임 문서: 세부 구조와 불변 조건.
- 로드맵: 개발 순서와 단계별 완료 기준.
- Active/Handoff Context: 현재 상태, 다음 행동, 위험, 읽기 순서.
- Issue/Goal: 현재 실행 범위와 승인 게이트.
- Git 이력: 이전 버전과 변경 이유.

## 5. 장점

- 새 작업자가 최신 방향을 빠르게 찾음.
- 같은 내용을 여러 파일에서 반복 수정하는 비용 감소.
- 폐기된 안의 재구현 위험 감소.
- 구현 전 승인 게이트와 다음 행동이 명확해짐.
- Git 이력이 실제 백업 역할을 하므로 활성 폴더가 가벼워짐.

## 6. 위험

- 삭제 전 책임 원본에 필요한 내용이 모두 이동됐는지 확인해야 함.
- 외부 링크가 오래된 경로를 참조할 수 있음.
- Issue를 닫아도 검색 결과에는 남으므로 종료 설명이 필요함.
- Active Context에 세부 전문을 너무 많이 넣으면 다시 중복 문서가 됨.

## 7. 검증 방법

- 레거시 프로젝트명, 폐기 병종명, 오래된 파일 경로를 저장소에서 검색.
- README에서 시작해 모든 핵심 책임 문서에 도달 가능한지 확인.
- 새 작업자에게 10분 콜드 스타트 질문 수행.
- GDD, 로드맵, Active Context, 현재 Issue의 상태 문구 비교.
- 삭제된 파일이 Git 이력에서 복원 가능한지 확인.

## 8. 재사용 가능한 원칙

- 문서 수가 많아질수록 요약 문서보다 책임 원본과 라우팅 구조가 중요하다.
- 버전 번호가 아니라 최신 승인 결정과 책임 관계가 우선이다.
- Active Context는 전문 저장소가 아니라 실행 캡슐이어야 한다.
- 삭제는 정보 손실이 아니라 활성 정보 구조를 정리하는 작업이다.
- 인수인계 품질은 작성자의 설명 없이 새 작업자가 정확히 행동할 수 있는지로 검증한다.

## 9. 적용 시 주의

- 규제, 계약, 외부 납품, 감사가 필요한 문서는 Git 이력만으로 충분하지 않을 수 있다.
- 오래된 파일 경로에 외부 자동화가 의존하면 리다이렉트나 명시적 마이그레이션이 필요하다.
- 대규모 팀에서는 문서 소유자와 리뷰 절차를 추가해야 한다.
