# 사례 — 공용 Base와 프로젝트 사이의 양방향 학습 순환

- 출처 프로젝트: 십보강호 문서 최적화 작업
- 확인 날짜: 2026-07-16
- 작성 상태: 문서 운영 패턴·구현 검증 불필요
- 주제: 공용 규칙을 프로젝트에 적용하고 프로젝트 교훈을 다시 Base에 반영하는 순환을 한 작업 종료 흐름으로 관리

## 1. 문제

공용 저장소와 프로젝트 저장소를 함께 운영하면 다음 두 실패가 반복될 수 있다.

- Base의 좋은 규칙이 프로젝트 문서에 실제로 반영되지 않는다.
- 프로젝트에서 얻은 교훈이 대화나 프로젝트 파일에만 남아 다른 프로젝트가 재사용하지 못한다.

반대로 모든 프로젝트 문서를 Base에 복사하면 세계관·수치·경로·현재 상태가 섞여 공용 원본이 오염된다.

## 2. 맥락과 제약

- Base는 여러 프로젝트가 공유하는 규칙, 방법, 스킬, 템플릿과 사례의 원본이다.
- 프로젝트는 실제 기획, 수치, 파일 경로, 구현 상태와 테스트 결과를 소유한다.
- 한 작업자가 두 저장소를 순서대로 읽고 수정할 수 있다.
- 기존 책임 문서와 유사한 새 파일을 늘리지 않아야 한다.
- 검증되지 않은 프로젝트 가설을 공용 확정 규칙으로 승격하면 안 된다.

## 3. 검토한 대안

### A. Base를 한 번 복사한 뒤 프로젝트만 관리

장점:

- 일상 작업이 단순하다.

단점:

- Base 개선이 프로젝트에 반영되지 않는다.
- 프로젝트 로컬 사본이 빠르게 오래된다.

### B. 매 작업마다 Base 전체를 읽고 프로젝트에 복제

장점:

- 최신 규칙을 놓치기 어렵다.

단점:

- 컨텍스트 비용이 크다.
- 책임 문서가 중복된다.
- 프로젝트 고유 결정과 공용 원칙이 섞인다.

### C. 문서 지도 기반 선택적 채택과 작업 종료 승격

장점:

- 현재 작업에 필요한 Base 원본만 읽는다.
- 프로젝트 적용 결과와 차이를 명시적으로 기록한다.
- 검증된 일반 교훈만 기존 Base 책임 문서나 사례에 반영한다.
- 양쪽 인수 컨텍스트를 같은 작업에서 최신화한다.

채택: C.

## 4. 순환 구조

```text
1. Base 문서 지도와 현재 작업 관련 원본 확인
2. 프로젝트 AGENTS·Active Context·문서 지도·실제 파일 감사
3. 공용 원칙을 프로젝트 제약에 맞게 구체화
4. 프로젝트 기준 문서·로드맵·테스트·Plan 갱신
5. 구현·플레이·문서 운영 결과 기록
6. 프로젝트 고유 정보와 공용 교훈 분리
7. Base의 기존 책임 문서·사례와 중복 확인
8. 필요한 최소 변경만 Base에 반영
9. 양쪽 문서 지도·Active Context·변경 기록 갱신
10. 미검증과 후속 동기화 보고
```

## 5. 지식 분류

| 내용 | 프로젝트 | Base |
|---|---|---|
| 세계관·캐릭터·고유 용어 | 유지 | 복사하지 않음 |
| 밸런스 수치·확률·경제 | 유지 | 일반 계산법만 검증 후 고려 |
| 실제 파일·엔진·테스트 경로 | 유지 | 템플릿과 검수 원칙만 유지 |
| 반복 가능한 판단 순서 | 적용 기록 | method 후보 |
| 직접 실행 가능한 반복 절차 | 프로젝트 확장 | 반복 검증 후 skill 후보 |
| 구체 문제·대안·결정·결과 | 프로젝트 원본 | 일반화한 case |
| 복사 가능한 산출물 형식 | 프로젝트 사용본 | template |

## 6. 적용 기록의 최소 항목

프로젝트에는 다음을 남긴다.

- 읽은 Base 책임 원본.
- 채택한 공용 원칙.
- 프로젝트가 더 구체적으로 정한 규칙.
- 적용하지 않은 규칙과 이유.
- Base로 되돌려 보낸 교훈.
- 아직 승격하지 않은 후보와 필요한 검증.
- Base 기준 버전·커밋 동기화 필요 여부.

Base에는 프로젝트 전체 문서를 복제하지 않고 문제, 제약, 대안, 결정, 결과, 재사용 원칙과 복사 금지 요소만 남긴다.

## 7. 중복 방지 규칙

- 같은 책임의 Base method·skill·case가 있으면 새 파일보다 기존 원본 갱신을 우선한다.
- 사례는 실제 프로젝트 맥락을 보존하되 활성 GDD를 대체하지 않는다.
- 한 프로젝트에서만 채택된 설계는 바로 `검증된 skill`로 표시하지 않는다.
- 프로젝트가 Base 규칙을 구체화한 내용은 `PROJECT_SKILL_EXTENSION` 또는 적용 기록에 두고 Base 스킬 전문을 복제하지 않는다.
- 파일 삭제는 참조와 대체 경로 확인 뒤 수행하며 일반적으로 Git 이력을 백업으로 사용한다.

## 8. 검증 방법

- 새 작업자가 프로젝트 저장소만 읽고 현재 방향과 Base 적용 차이를 설명하는지 확인한다.
- Base 사례만 읽은 작업자가 프로젝트 고유 수치를 공용 규칙으로 오해하지 않는지 확인한다.
- 프로젝트 작업 종료 시 Base 승격 또는 `승격 없음` 판단이 기록되는지 확인한다.
- Base 갱신 뒤 프로젝트의 로컬 사본·버전 기록 필요 여부가 보고되는지 확인한다.
- 같은 문장이 Base와 프로젝트 여러 파일에 장문 복제되지 않았는지 검색한다.

## 9. 결과

십보강호 작업에서는 Base의 기획서 책임 구조, Active Context, Vertical Slice, 프레젠테이션 상태 비소유, 지식 승격 원칙을 프로젝트 문서에 적용했다. 프로젝트에서는 전투·대회 연출 책임 원본과 Base 적용 기록을 만들었고, 일반화 가능한 교훈은 이 사례와 내부 데이터→세계관 표현 사례로 Base에 분리했다.

실제 게임 구현과 플레이테스트가 필요한 교훈은 skill로 승격하지 않고 사례·방법 보완 수준에 유지했다.

## 10. 재사용 가능한 원칙

- 공용 지식 채택과 프로젝트 지식 승격은 하나의 순환으로 관리한다.
- Base 전체 복제보다 문서 지도 기반 최소 읽기와 책임 원본 연결이 중요하다.
- 프로젝트 적용 기록은 공용 원칙과 프로젝트 전용 값을 구분하는 경계 문서다.
- 실제 결과가 없는 설계는 사례·가설로 남기고 실행 스킬로 과장하지 않는다.
- 인수인계 품질은 양쪽 저장소를 처음 읽는 작업자가 설명 없이 올바른 다음 행동을 선택하는지로 검증한다.
