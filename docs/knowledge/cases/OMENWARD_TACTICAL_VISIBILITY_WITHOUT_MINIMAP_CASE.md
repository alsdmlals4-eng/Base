# 사례 — 전장이 한눈에 보일 때 미니맵을 제거한 결정

- 출처 프로젝트: OMENWARD
- 상태: 채택 / PoC 검증 전
- 주제: 중복 정보 UI 제거와 실제 전장 위 상태 표시

## 1. 문제

전략 게임이라는 이유로 초기 UI에 미니맵을 넣는 안을 검토했지만, 목표 카메라에서는 양측 본진과 세 전선 전체가 기본 화면에 들어온다.

이 조건에서 미니맵은 다음 문제를 만들 수 있었다.

- 실제 전장과 축소 지도의 정보를 중복 표시.
- 작은 HUD 영역을 계속 차지.
- 플레이어 시선을 화면 구석으로 분산.
- 룰렛·건설·전술 패널과 경쟁.
- 성문·거점·접전지 상태를 두 위치에서 동기화해야 함.

## 2. 핵심 질문

미니맵의 목적은 “전략 게임처럼 보이기”가 아니라 멀리 떨어진 중요 상태를 빠르게 파악하게 하는 것이다.

따라서 다음 질문으로 바꿨다.

```text
플레이어가 실제 전장만 보고도 필요한 시간 안에 모든 전선의 중요 상태를 찾을 수 있는가?
```

## 3. 채택한 결정

- 별도 미니맵을 사용하지 않는다.
- 성문 내구도, 중간거점 소유권·점령 진행, 중앙 접전지 상태를 실제 위치 위에 표시한다.
- 화면 구석 HUD는 자원, 웨이브, 룰렛, 건설처럼 전장 위치와 직접 연결되지 않는 정보에 사용한다.
- 카메라 기본 줌에서 전체 전장이 들어오도록 맵 비율과 UI 점유율을 함께 조정한다.

## 4. 장점

- 중복 UI와 동기화 비용 감소.
- 플레이어가 실제 교전과 목표 상태를 같은 시선에서 확인.
- 화면 구석 공간을 핵심 조작 UI에 사용.
- 전장 자체가 정보 구조가 되어 세계와 시스템의 연결이 강화.

## 5. 위험

- 실제 전장 상태 표시가 작거나 겹치면 미니맵보다 읽기 어려울 수 있음.
- 세 라인의 효과와 유닛 수가 많아지면 화면 전체가 복잡해질 수 있음.
- 확대 상태에서 다른 라인의 위험을 놓칠 수 있음.
- 저해상도에서 목표 UI가 겹칠 수 있음.

## 6. 보완 설계

- 중요 상태는 실제 위치에 고정된 간단한 아이콘·게이지로 표시.
- 위험한 성문·거점은 크기, 테두리, 짧은 펄스로 우선순위 표시.
- 확대 중 다른 라인의 긴급 상황은 화면 가장자리 방향 신호로 보조할 수 있음.
- 기본 전략 줌으로 즉시 복귀하는 입력 제공.
- 상태 표시가 유닛과 VFX에 묻히지 않도록 레이어·불투명도 규칙 설정.

## 7. 검증 방법

- 기본 줌에서 3초 안에 세 라인의 성문·거점·접전지 상태를 말할 수 있는지 테스트.
- 1280×720과 1920×1080에서 비교.
- 전투 VFX와 HUD를 모두 켠 상태에서 측정.
- 확대 후 다른 라인 위험을 발견하는 데 걸리는 시간 측정.
- 미니맵이 있는 비교 시안과 정보 탐색 시간·시선 이동·오조작 비교.

## 8. 재사용 가능한 원칙

- 장르 관습보다 정보 문제를 먼저 정의한다.
- 실제 공간이 한눈에 보이면 축소 복제 UI를 자동으로 추가하지 않는다.
- 월드 공간에 자연스럽게 붙는 정보는 실제 위치에 표시하는 편이 이해가 빠를 수 있다.
- UI 제거 결정도 반드시 가독성 테스트를 거친다.
- 미니맵을 없애는 것은 정보 삭제가 아니라 정보 위치를 재설계하는 작업이다.

## 9. 적용하지 말아야 할 조건

- 맵이 화면보다 훨씬 크고 카메라 이동이 핵심인 게임.
- 시야 제한과 탐색 정보가 전략의 핵심인 게임.
- 동시에 관찰해야 할 목표가 화면 밖에 자주 존재.
- 월드 표시가 지형·유닛·VFX에 지속적으로 가려짐.

이 경우에는 미니맵, 전략 지도, 가장자리 알림 중 어떤 방식이 문제를 가장 잘 해결하는지 비교해야 한다.
