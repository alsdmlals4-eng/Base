# 사례 — 상시 호감도 숫자 대신 선택 기억과 후속 반응으로 관계를 표현하기

- 출처 프로젝트: urban-legend / 괴이 기록국
- 상태: 승인 기획·구현 전 검증 필요
- 주제: 캐릭터 관계를 성능 카드나 연애 점수로 축소하지 않고 선택의 기억으로 표현

## 1. 문제

동료 관계를 단일 점수와 보너스로 표현하면 구현과 비교는 쉽지만 다음 위험이 있었다.

- 인물이 최적화 대상과 보상 공급원으로 보임.
- 같은 점수라도 관계의 이유와 성격을 설명하지 못함.
- 신뢰, 경계, 부채, 경쟁처럼 서로 다른 관계가 한 축에 섞임.
- 관계 보상이 핵심 판단과 성능을 대신할 수 있음.

## 2. 관찰한 근거

선택 중심 서사에서는 사용자가 점수보다 말투, 호칭, 정보 공유, 도움 요청, 후속 장면으로 관계 변화를 인식했다. 작은 반응 분기를 여러 번 제공하는 방식은 적은 수의 큰 분기보다 “선택을 기억한다”는 감각을 지속적으로 줄 수 있었다.

## 3. 채택한 결정

- 관계 의미를 신뢰·경계·부채·경쟁·공감·보호·균열 같은 태그로 분리.
- 선택 ID와 대상 인물, 기억할 태도, 발동 가능한 후속 반응을 연결.
- 사용자에게는 말투·정보 공유·행동 순서·거리·후속 이벤트로 표현.
- 대부분은 반응 분기, 중요한 관계만 장면·체인 분기로 확장.
- 관계는 핵심 정답·전투 성능·필수 진행을 소유하지 않음.

## 4. 제외한 접근

- 상시 호감도 게이지.
- 선물 반복과 점수 파밍.
- 높은 관계 수치로 핵심 단서·정답 자동 공개.
- 모든 태도 선택을 장기 분기로 확장.
- 기존 상태를 감사하지 않고 새 관계 저장 구조를 즉시 추가.

## 5. 검증 계획

- 숫자 표시 없이 사용자가 관계가 왜 달라졌는지 설명하는지 확인.
- 같은 선택 반복으로 태그와 장면이 중복 적용되지 않는지 검증.
- 저장·불러오기 뒤 말투와 후속 이벤트 해금이 유지되는지 확인.
- 선택하지 않은 인물이 반응에 등장하지 않는지 확인.
- 관계 변화가 핵심 과제의 정답률과 성능을 직접 높이지 않는지 확인.

## 6. 예상 가치

관계의 방향과 이유를 보존하면서 인물이 시스템 보너스로 축소되는 것을 막을 수 있다. 반응 분기 중심 구조는 콘텐츠 비용을 통제하면서 선택 기억을 자주 보여줄 수 있다.

## 7. 재사용 가능한 원칙

- 내부 계산과 사용자가 느끼는 관계 경험을 분리한다.
- 단일 점수로 설명하기 어려운 관계는 의미 태그와 선택 기억을 사용한다.
- 관계 보상은 성능보다 대사·행동·정보 공유·장면 접근에 우선 배치한다.
- 큰 분기보다 자주 보이는 작은 반응이 선택 기억을 강화할 수 있다.

## 8. 적용 시 주의

- 육성·연애·경영처럼 수치 최적화가 핵심 약속인 장르에는 상시 수치가 필요할 수 있다.
- 태그가 많아지면 충돌 우선순위와 소비·유지 규칙이 필요하다.
- 실제 구현 결과가 아직 없으면 공용 원칙이 아니라 조건부 사례로 유지한다.
