# 벤치마킹 참고 가이드

## 목적

벤치마킹은 다른 제품의 표면을 복제하는 일이 아니라, 사용자가 느끼는 가치와 문제 해결 방식을 현재 프로젝트에 맞게 해석하는 과정이다.

## 기본 기록 형식

벤치마킹 표는 기본적으로 아래 형식을 사용한다.

```md
| # | 게임/사례 | 관찰포인트 | 반응 | 프로젝트 적용(+사유) |
|---:|---|---|---|---|
| 1 |  |  |  |  |
```

| 항목 | 기록할 내용 |
|---|---|
| 게임/사례 | 참고한 게임, 제품, 기능, 장면, 사용자 흐름 |
| 관찰포인트 | 구조, 규칙, UX, 반복 플레이, 보상, 감정 경험 등 확인한 점 |
| 반응 | 플레이어 반응, 평론, 개발자 발언, 전문가 의견, 실제 사용 평가 |
| 프로젝트 적용(+사유) | 그대로 복제하지 않고 현재 프로젝트에 맞게 적용하거나 제외할 이유 |

## 선택 분석 항목

필요하면 기본 표 아래에 다음 항목을 추가한다.

```md
## 추가 분석

- 비용과 위험:
- 그대로 쓰면 안 되는 점:
- 우리 프로젝트만의 차별점:
- 검증 방법:
```

## 적용 기준

- 1차 출처와 실제 사용 흐름을 우선 확인한다.
- 장르나 플랫폼이 다르면 원리와 제약을 분리해 해석한다.
- 핵심 경험, 예산, 일정, 기술 여건에 맞지 않으면 적용하지 않는다.
- 사례는 영감이 아니라 가설의 근거로 사용하고, 실제 사용자 검증으로 결정한다.
- 출처 없는 일반론만으로 판단하지 않는다.
- 다른 작품의 기능을 그대로 복사하지 않는다.

## 사례 수 기준

- 일반 기능 판단: 관련 사례 3~5개.
- 장르 핵심 구조, 시장성 판단, 큰 방향 전환: 10개 이상.
- 이미 Issue에서 구현 방향이 확정된 작업: 새 벤치마킹을 강제하지 않는다.

## 결론 형식

벤치마킹 후에는 반드시 적용 결론을 남긴다.

```md
## 벤치마킹 결론

- 반드시 반영:
- 조건부 반영:
- 제외:
- 위험:
- 검증 방법:
```
