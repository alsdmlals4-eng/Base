# 벤치마킹 참고 가이드

## 목적

벤치마킹은 다른 제품의 표면을 복제하는 일이 아니라, 사용자가 느끼는 가치와 문제 해결 방식을 현재 프로젝트에 맞게 해석하는 과정이다.

GPT는 기획, Issue, Goal, UI/UX, 시스템 설계, 검수 작업에서 항상 작업 성격에 맞는 벤치마킹을 수행한다.

## 기본 원칙

- 벤치마킹은 선택 절차가 아니라 작업 전 기본 절차다.
- 작업 범위가 작으면 1~3개 사례만 압축 확인해도 된다.
- 큰 방향 전환, 시장성, 핵심 시스템, UI/UX 작업은 더 넓게 본다.
- 게임성, 디자인, UX, 아이디어, 시장성, 구현 방식을 구분해 본다.
- 결과는 장문 복사가 아니라 개선점과 수정할 점으로 정리한다.
- Issue/문서에는 필요한 맥락만 반영한다.
- Codex Goal에는 긴 벤치마킹 전문을 넣지 않는다.

## 기본 기록 형식

벤치마킹 표는 기본적으로 아래 형식을 사용한다.

```md
| # | 게임/사례 | 관찰포인트 | 개선점/수정점 | 프로젝트 적용(+사유) |
|---:|---|---|---|---|
| 1 |  |  |  |  |
```

| 항목 | 기록할 내용 |
|---|---|
| 게임/사례 | 참고한 게임, 제품, 기능, 장면, 사용자 흐름 |
| 관찰포인트 | 구조, 규칙, UX, 반복 플레이, 보상, 감정 경험 등 확인한 점 |
| 개선점/수정점 | 현재 프로젝트에서 개선하거나 고칠 점 |
| 프로젝트 적용(+사유) | 그대로 복제하지 않고 현재 프로젝트에 맞게 적용하거나 제외할 이유 |

## 고정 참고 링크

프로젝트가 고정 참고 링크를 지정하면 매 작업에서 먼저 확인한다.

기본 공용 확인 대상 예시:

- Unity Asset Store: 에셋 구성, UI 키트, 장르 템플릿, 판매 페이지 구조, 기능 패키징 방식 참고
- itch.io / Steam 데모 페이지: 짧은 소개 문구, 데모 구성, 스크린샷, 플레이어 반응, 위시리스트 유도, 다운로드/피드백 흐름 참고

프로젝트별 고정 링크는 각 프로젝트의 로컬 문서에 둔다.

## 선택 분석 항목

필요하면 기본 표 아래에 다음 항목을 추가한다.

```md
## 추가 분석

- 비용과 위험:
- 그대로 쓰면 안 되는 점:
- 우리 프로젝트만의 차별점:
- 검증 방법:
```

## 적용 기준

- 1차 출처와 실제 사용 흐름을 우선 확인한다.
- 장르나 플랫폼이 다르면 원리와 제약을 분리해 해석한다.
- 핵심 경험, 예산, 일정, 기술 여건에 맞지 않으면 적용하지 않는다.
- 사례는 영감이 아니라 가설의 근거로 사용하고, 실제 사용자 검증으로 결정한다.
- 출처 없는 일반론만으로 판단하지 않는다.
- 다른 작품의 기능을 그대로 복사하지 않는다.
- 플랫폼 목표가 정해져 있으면 해당 플랫폼의 UX, 입력 장치, 화면비, 배포 페이지를 우선한다.

## 사례 수 기준

- 빠른 수정: 관련 사례 1개 이상 또는 기존 고정 참고 링크 확인.
- 일반 기능 판단: 관련 사례 3~5개.
- 장르 핵심 구조, 시장성 판단, 큰 방향 전환: 10개 이상.
- 이미 Issue에서 구현 방향이 확정된 작업: 고정 참고 링크와 충돌 여부만 압축 확인한다.

## 결론 형식

벤치마킹 후에는 반드시 적용 결론을 남긴다.

```md
## 벤치마킹 결론

- 반드시 반영:
- 조건부 반영:
- 제외:
- 위험:
- 검증 방법:
```
