# 작업 시작·종료 체크리스트

실제 프로젝트 작업에서 사용하는 공용 체크리스트다. 작업 성격에 맞는 항목만 적용하고, 존재하지 않는 도구나 테스트를 실행한 것처럼 표시하지 않는다.

Base는 **[학습형] [공용]** 데이터 원본이고, 프로젝트는 이를 실제 조건에 맞게 분화·적용·검증하는 전용 작업 공간이다.

게임 프로젝트 운영체계의 전면 감사·설치는 `skills/installing-game-project-operating-system/SKILL.md`를 사용한다.

## 1. 작업 유형

| 단계 | 예시 | 경로 |
|---|---|---|
| `L0` | 오탈자, 명확한 문장·형식 변경 | 대상 확인→수정→최소 검증 |
| `L1` | 단일 데이터·작은 UI 수정 | 범위·영향 분야·검증 보완→수정 |
| `L2` | 한 시스템의 설계 선택 | 브레인스토밍→사양→Plan→실행 |
| `L3` | 여러 시스템·핵심 구조·운영체계 | 기획서→Roadmap·Plan→구현 |
| `L4` | 재사용 가능한 방법 | 프로젝트 검증→Base 학습 데이터·case 환류 |

## 2. 시작 전 공용·전용 확인

```md
- [ ] 최신 사용자 지시를 확인했다.
- [ ] 사용자가 Base URL을 명시했다면 Base `START_HERE.md`를 확인했다.
- [ ] 프로젝트 `AGENTS.md`를 확인했다.
- [ ] `BASE_RULES_VERSION.md`와 Base 기준을 확인했다.
- [ ] 관련 Base method·research·skill·template·case를 확인했다.
- [ ] 프로젝트 START_HERE와 Documentation Map을 확인했다.
- [ ] Handoff·Active Context와 프로젝트 방향·관련 본책을 확인했다.
- [ ] Roadmap과 현재 Issue·Goal·Plan을 확인했다.
- [ ] 프로젝트 skill extension을 확인했다.
- [ ] 실제 대상, 연결 파일, 데이터·저장·호환성 영향을 확인했다.
- [ ] 작업 단계를 L0~L4로 분류했다.
- [ ] `archive`·`hold`·`[보류]`를 기본 범위에서 제외했다.
- [ ] 기존 사용자 변경과 저장소 상태를 확인했다.
- [ ] 해결할 문제와 사용자·플레이어 가치를 한 문장으로 설명할 수 있다.
```

`모두 확인`은 저장소 모든 파일을 무작정 읽는다는 뜻이 아니다. Documentation Map과 참조 관계로 현재 작업에 적용되는 공용·전용 현행 책임 원본과 영향 파일을 선별해 빠짐없이 확인한다.

## 3. 영향도 분석

**영향도 분석(Impact Analysis)**: 한 변경이 어떤 분야, 문서, 코드, 데이터, 자산과 테스트에 영향을 주는지 작업 전에 확인하는 절차다.

L1 이상 작업은 다음을 선언한다.

```yaml
primary_discipline:
affected_disciplines:
change_type:
goal:
scope:
out_of_scope:
protected_paths:
required_docs:
asset_impact:
data_impact:
acceptance_criteria:
validation:
```

```md
- [ ] 주 책임 분야를 하나 지정했다.
- [ ] 영향 분야를 빠짐없이 지정했다.
- [ ] 프로젝트 `DOCUMENT_UPDATE_MATRIX.md`에서 필수 갱신 대상을 확인했다.
- [ ] 문서·구현·이미지·사운드·검증·일정 변경 유형을 구분했다.
- [ ] 현재 작업에서 변경하지 않을 보호 경로와 동작을 적었다.
```

## 4. 실행 프롬프트

`skills/transforming-requests-into-prompts/SKILL.md`와 `templates/EXECUTABLE_PROMPT.md`를 사용한다.

```md
- [ ] 목적
- [ ] 맥락
- [ ] 목표 사용자·플레이어 경험
- [ ] 작업 범위
- [ ] 제약·제외 범위
- [ ] 산출물
- [ ] 완료 기준
- [ ] 테스트·검증
- [ ] 먼저 읽을 공용·전용 문서와 파일
- [ ] 위험·의존성
- [ ] 작업 후 문서 최신화와 Base 환류 형식
```

확정되지 않은 내용은 `[권장]` 또는 `[확인 필요]`로 표시한다.

## 5. Definition of Ready

```md
- [ ] 핵심 플레이어 경험과 프로젝트 방향에 기여하는 방식이 설명돼 있다.
- [ ] Roadmap 단계와 선행 조건이 연결돼 있다.
- [ ] 포함 범위와 제외 범위가 분리돼 있다.
- [ ] 기존 동작과 보호할 데이터·인터페이스가 명시돼 있다.
- [ ] 완료 기준이 `조건 → 행동 → 관찰 결과`로 작성돼 있다.
- [ ] 예상 영향 파일과 저장·호환성 위험이 있다.
- [ ] 정상·예외·회귀 검증 방법이 있다.
- [ ] 필요한 Base skill과 Project extension이 연결돼 있다.
- [ ] 필요한 선행 결정과 의존성이 해결됐다.
```

준비되지 않은 요구는 구현자에게 넘기지 않고 질문, 작은 조사, 기획서 보완으로 분리한다.

## 6. 벤치마킹·유저리서치

새로운 방향, 핵심 시스템, 시장·기술·UX 판단에 근거가 필요한 경우에만 수행한다.

```md
- [ ] 결정할 질문이 한 문장인가?
- [ ] 공식·1차 출처와 사용자 반응을 구분했는가?
- [ ] 관찰, 해석, 적용 결론을 분리했는가?
- [ ] 적용·조건부 적용·제외를 기록했는가?
- [ ] 외부 사례의 표면을 복제하지 않았는가?
- [ ] 바뀔 수 있는 사실에 확인 날짜가 있는가?
- [ ] 재사용 가능한 관찰을 research 또는 case로 남겼는가?
```

이미 Issue에서 확정된 작은 구현에 불필요한 신규 벤치마킹을 강제하지 않는다.

## 7. 본책·Roadmap·스킬·Issue

```md
- [ ] 본책만으로 해당 분야의 방향, 플레이어 경험, 품질 기준, 현재 상태와 금지 방향을 이해할 수 있는가?
- [ ] 질문별 현행 책임 문서가 하나인가?
- [ ] 확정, 구현, 검증, 진행 중, 가설, 미확정, 보류, 대체됨과 불일치를 구분했는가?
- [ ] Roadmap에 현재 단계, 우선순위, 선행 조건, 다음 작업과 종료 기준이 있는가?
- [ ] Base skill과 Project extension이 실제 경로·데이터·검증에 연결되는가?
- [ ] Issue는 문제·범위·완료 기준을 보존하는가?
- [ ] Goal·Work Order는 구현에 필요한 결정만 압축했는가?
- [ ] Plan은 실제 파일 조사 결과와 순서를 포함하는가?
- [ ] 같은 설명을 여러 문서에 장문 복사하지 않았는가?
```

## 8. 이미지·자산·사운드

```md
- [ ] 프로젝트 Visual Source와 Asset Manifest를 확인했다.
- [ ] 기존 승인 이미지가 있는 항목에 별도 지시 없이 새 시안을 만들지 않았다.
- [ ] 이미지 전체가 아니라 참고할 요소와 참고하지 않을 요소를 기록했다.
- [ ] 항목별 캐노니컬 경로가 하나인가?
- [ ] `v2`·`final`·`latest` 활성 복제본을 만들지 않았다.
- [ ] REFERENCE·DIRECTION_APPROVED·PRODUCTION_READY·IMPLEMENTED·VISUALLY_VALIDATED를 구분했다.
- [ ] `MIGRATION_PENDING`과 실제 파일 저장 완료를 구분했다.
- [ ] Visual DNA, Import, 피벗, 해상도·프레임·성능 예산 영향을 확인했다.
- [ ] 콘셉트 이미지와 실제 게임 캡처·골든 스크린샷을 구분했다.
- [ ] BGM·SFX·음성 제작과 이벤트 연결·믹싱·검증 상태를 구분했다.
```

## 9. 구현·리팩터링

```md
- [ ] 요청과 직접 관련된 파일만 변경한다.
- [ ] 기존 호출·데이터·저장·UI 흐름을 확인했다.
- [ ] 가장 작은 변경으로 완료 기준을 만족한다.
- [ ] 기능 추가와 리팩터링을 가능한 한 분리했다.
- [ ] 리팩터링 전 현재 동작과 회귀 검증을 확보했다.
- [ ] 공개 인터페이스·저장 형식·사용자 흐름을 승인 없이 바꾸지 않았다.
- [ ] 범위 밖 개선은 별도 제안으로 분리했다.
```

## 10. 파일 변경

```md
- [ ] 생성·수정·삭제·이동·이름 변경 이유가 명확하다.
- [ ] 참조하는 문서·코드·템플릿을 확인했다.
- [ ] 현행 책임 문서가 하나만 남는다.
- [ ] `v2`·`final`·`latest`·날짜별 활성 복제본을 만들지 않았다.
- [ ] 이전 상태는 Git 이력으로 보존했다.
- [ ] 통합이 끝난 중복 파일을 고유 정보와 참조 확인 후 제거했다.
- [ ] Documentation Map·README·Roadmap·skill extension의 경로 영향을 확인했다.
- [ ] 프로젝트 로컬 사본과 Base 버전 동기화 영향을 확인했다.
```

## 11. 검증

```md
- [ ] 포맷·문법·타입·정적 검사 중 존재하는 검증을 실행했다.
- [ ] 관련 자동 테스트를 실행했다.
- [ ] 핵심 정상 경로를 확인했다.
- [ ] 실패·취소·중복 입력·누락 데이터 등 예외를 확인했다.
- [ ] 저장·불러오기·호환성을 확인했다.
- [ ] 실제 화면·플레이·오디오·성능 검증이 필요한지 확인했다.
- [ ] diff가 승인 범위 안인지 확인했다.
- [ ] 사용자가 실행할 수동 검수 순서를 적었다.
- [ ] 실행하지 못한 검증을 미검증으로 표시했다.
```

## 12. GitHub PR·자동 검사

```md
- [ ] Issue·PR 템플릿에 분야·범위·완료 기준·검증이 기록됐다.
- [ ] 문서 governance 검사가 실제 프로젝트 경로에 맞게 설정됐다.
- [ ] 필수 파일·깨진 링크·금지 파일명·Manifest 중복을 검사했다.
- [ ] 코드·자산 변경 시 관련 본책 갱신 규칙을 샘플 PR에서 확인했다.
- [ ] GitHub Actions 파일 존재와 실제 실행 성공을 구분했다.
- [ ] Required Status Check와 CODEOWNERS가 실제 활성화됐는지 확인했다.
```

브랜치 보호를 확인하지 못했다면 `미검증`으로 기록한다.

## 13. 종료·인수인계와 지속성

```md
- [ ] 변경 파일과 이유를 기록했다.
- [ ] 유지한 기존 동작을 기록했다.
- [ ] 검증 명령과 결과를 기록했다.
- [ ] 미검증 항목과 남은 위험을 분리했다.
- [ ] 주 책임 분야와 영향 분야 본책을 갱신했다.
- [ ] Roadmap의 단계·우선순위·선행 조건·다음 작업·종료 기준을 갱신했다.
- [ ] 프로젝트 skill extension을 현재 구조와 맞췄다.
- [ ] Handoff·Active Context와 Documentation Map을 갱신했다.
- [ ] Decision Log·Changelog·Manifest 갱신 필요를 확인했다.
- [ ] Issue·Goal·Plan과 실제 구현 상태의 일치를 확인했다.
- [ ] 다음 작업자가 읽을 시작 문서를 3~6개로 제한했다.
- [ ] 새 작업자가 10분 안에 핵심 방향, 현재 상태, 다음 작업, 금지 범위와 검증 방법을 찾을 수 있다.
```

## 14. Base 학습 환류

`skills/promoting-project-knowledge/SKILL.md`를 사용한다.

```md
- [ ] 프로젝트 고유 내용과 공용 원칙을 분리했다.
- [ ] 검증된 판단법·절차·출력 구조를 기존 Base method·skill·template에 반영했다.
- [ ] 문제·대안·결정·결과·실패·수정·미검증을 case로 작성하거나 갱신했다.
- [ ] 한 번 성공한 방법을 `관찰`·`가설`로 남기고 검증으로 과장하지 않았다.
- [ ] 기존 Base 책임 문서와 중복·충돌을 확인했다.
- [ ] Base Changelog와 문서 라우터를 갱신했다.
- [ ] 프로젝트 로컬 사본과 `BASE_RULES_VERSION.md`의 후속 동기화를 확인했다.
```

공용화할 내용이 없으면 `공용 학습 데이터 없음 — 프로젝트 전용 또는 단발성 작업`으로 보고한다.
