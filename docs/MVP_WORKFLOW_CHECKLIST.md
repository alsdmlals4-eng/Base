# MVP 작업 체크리스트

이 문서는 실제 작업 시작·종료 때 사용하는 공용 체크리스트다.

## 1. 표준 작업 순서

```text
0. 작업 유형 분류
1. 사용자 최신 지시 확인
2. 프로젝트의 Base 로컬 사본 확인
3. 프로젝트 전용 규칙 확인
4. 현재 Issue / MVP 기준선 확인
5. 작업 준비 상태 체크
6. 벤치마킹 수행 및 개선점/수정점 도출
7. 구현 범위 / 제외 범위 확정
8. 위험 분석 / 영향 파일 예상
9. GitHub Issue 작성 또는 수정
10. HTML 전체 흐름 대시보드 갱신
11. Codex Goal 작성
12. 구현
13. 실행 검수
14. 결과 보고 / 회고
15. 공용 규칙 판단과 Base 자동 승격
```

## 2. 작업 유형 분류

| 유형 | 설명 | 경로 |
|---|---|---|
| 빠른 수정 | 오탈자, 단순 문서, 단일 데이터 | 빠른 경로 |
| 기능 작업 | 새 기능, 여러 파일, 저장/불러오기 | 기능 경로 |
| 기획 작업 | 시스템/루프/플레이어 경험 설계 | 기획 경로 |
| 검수 작업 | 구현 결과 확인, 오류 분석 | 검수 경로 |
| 규칙 정리 | 반복 규칙 문서화 | 문서 경로 |

## 3. 작업 시작 체크리스트

```md
- [ ] 사용자의 최신 지시를 확인했는가?
- [ ] 작업 유형을 분류했는가?
- [ ] Base 공용 규칙과 프로젝트 전용 규칙을 구분했는가?
- [ ] 현재 Issue 또는 Goal이 있는가?
- [ ] 이전 작업과 다음 작업이 연결되는가?
- [ ] 벤치마킹을 수행했는가?
- [ ] 벤치마킹에서 개선점과 수정할 점을 도출했는가?
- [ ] 구현 AI와 기획 AI의 역할이 분리되어 있는가?
- [ ] 대시보드/문서/구현 범위가 섞이지 않았는가?
```

## 4. 좋은 프롬프트 변환 체크리스트

사용자의 짧은 요청은 그대로 넘기지 않고 아래 항목을 보강한다.

```md
- [ ] 목표
- [ ] 배경
- [ ] 사용자의 의도
- [ ] 플레이어 경험 또는 사용자 가치
- [ ] 벤치마킹에서 얻은 개선점/수정점
- [ ] 구현 범위
- [ ] 제외 범위
- [ ] 데이터 요구사항
- [ ] 엔진/씬/노드/파일 영향
- [ ] 완료 기준
- [ ] 테스트 방법
- [ ] 결과물 형식
```

## 5. Definition of Ready

구현을 시작하기 전에 다음 조건을 만족해야 한다.

```md
- [ ] 해결할 사용자 문제와 MVP 경계를 한 문장으로 설명할 수 있다.
- [ ] 벤치마킹 근거와 프로젝트 적용/제외 결론이 있다.
- [ ] 완료 기준과 제외 범위가 정리되어 있다.
- [ ] 관련 규칙, Issue, 실제 파일을 확인했다.
- [ ] 기존 변경사항과 저장소 상태를 확인했다.
- [ ] 예상 영향 파일과 저장/호환성 위험을 적었다.
- [ ] 수정 대상과 검증 방법을 작업 전에 공유했다.
```

준비되지 않은 요구사항은 구현자에게 넘기지 말고 질문, Issue 보완, 또는 작은 탐색 작업으로 분리한다.

## 6. 벤치마킹 체크리스트

벤치마킹은 선택 절차가 아니다. GPT는 매 작업에서 작업 성격에 맞는 범위로 벤치마킹을 수행한다.

```md
- [ ] 유사/인접 사례를 확인했는가?
- [ ] 프로젝트 지정 고정 참고 링크를 확인했는가?
- [ ] 게임/사례를 적었는가?
- [ ] 게임성, 디자인, UX, 아이디어 중 관련 항목을 확인했는가?
- [ ] 관찰포인트를 적었는가?
- [ ] 개선점과 수정할 점을 도출했는가?
- [ ] 프로젝트 적용 이유를 적었는가?
- [ ] 반영할 것과 제외할 것을 나눴는가?
- [ ] 출처 없는 일반론으로 판단하지 않았는가?
```

권장 표:

```md
| # | 게임/사례 | 관찰포인트 | 개선점/수정점 | 프로젝트 적용(+사유) |
|---:|---|---|---|---|
```

## 7. Issue 체크리스트

```md
- [ ] 목표
- [ ] 핵심 흐름
- [ ] 벤치마킹 반영/제외 결론
- [ ] 구현 범위
- [ ] 제외 범위
- [ ] 완료 기준
- [ ] 작업 전 확인 파일
- [ ] 공통 규칙 참조
- [ ] 작업 후 보고 형식
```

## 8. Codex Goal 체크리스트

```md
- [ ] Issue 번호가 맞는가?
- [ ] Base 로컬 사본과 프로젝트 전용 규칙을 참조하는가?
- [ ] 벤치마킹 결론이 Issue/문서에 반영되어 있고, Goal에는 실행에 필요한 만큼만 남았는가?
- [ ] 구현 범위만 들어갔는가?
- [ ] 제외 범위가 있는가?
- [ ] 완료 기준이 있는가?
- [ ] 테스트 기준이 있는가?
- [ ] 작업 후 보고 형식이 있는가?
- [ ] 대시보드나 긴 벤치마킹 전문이 빠져 있는가?
```

## 9. 구현 중 체크리스트

```md
- [ ] 요구사항과 직접 관련된 파일만 변경한다.
- [ ] 데이터, 저장, UI, 호출 흐름의 영향을 확인한다.
- [ ] 오류와 예외 상태를 고려한다.
- [ ] 요청 범위 밖 개선은 분리해 제안만 한다.
- [ ] 정상 동작하는 구조를 임의로 바꾸지 않는다.
- [ ] 서사형 안내자의 세계관 역할이 담당 시스템과 일치하는가?
- [ ] 최초 상세 안내와 재방문용 짧은 안내를 분리하고 확인 상태를 저장하는가?
- [ ] 음향 안내와 동등한 시각 또는 문구 안내가 있는가?
- [ ] 안내자가 미획득 단서·정답·숨은 수치를 누설하지 않는가?
```

## 10. 파일 변경 의도 체크리스트

파일을 생성, 수정, 삭제, 이동, 이름 변경할 때는 다음을 확인한다.

```md
- [ ] 변경 유형이 명확한가?
- [ ] 변경 이유가 사용자 요청 또는 Issue로 거슬러 올라가는가?
- [ ] 삭제/이동/이름 변경 시 참조 업데이트가 필요한가?
- [ ] 한 주제의 현행 책임 문서가 하나만 남는가?
- [ ] 확정된 변경을 기존 기준 파일에 직접 반영했는가?
- [ ] 불필요한 v2/final/latest/날짜별 활성 복제본을 만들지 않았는가?
- [ ] 이전 내용은 Git 이력으로 보존하고 불필요한 별도 백업을 만들지 않았는가?
- [ ] 새 기준 문서에 통합된 이전 버전·중복 파일을 참조 확인 뒤 제거했는가?
- [ ] 다른 AI나 다음 작업자가 봐야 할 영향이 있는가?
- [ ] Base 또는 프로젝트 로컬 사본 동기화가 필요한가?
```

권장 기록 형식:

```md
## 파일 변경 의도

- 파일:
- 변경 유형: 생성 / 수정 / 삭제 / 이동 / 이름 변경
- 변경 이유:
- 연결되는 문서 또는 코드:
- 다른 AI 또는 다음 작업자가 알아야 할 영향:
- 후속 동기화 필요 여부:
```

## 11. 결과 검수 체크리스트

```md
- [ ] 변경 파일이 보고되었는가?
- [ ] 변경 이유가 설명되었는가?
- [ ] 구현 내용이 Issue와 일치하는가?
- [ ] 기존 흐름이 깨지지 않았는가?
- [ ] 저장/불러오기를 확인했는가?
- [ ] 사용자가 실행할 확인 순서가 있는가?
- [ ] 남은 위험이 적혀 있는가?
- [ ] 사용 도구와 미사용 도구가 사실대로 적혀 있는가?
```

## 12. 완료 조건

```md
- [ ] 완료 기준을 항목별로 확인했다.
- [ ] 적절한 테스트 또는 실행 검증을 수행했다.
- [ ] 변경 파일과 검증 결과를 기록했다.
- [ ] 사용자가 직접 확인해야 할 항목을 분리했다.
- [ ] 남은 위험이 있다면 현재 해결할 수 없는 이유를 적었다.
```

## 13. 회고와 Base 자동 승격 체크리스트

```md
- [ ] 반복된 혼동이나 실패가 있었는가?
- [ ] 다음 작업에서 줄일 공용 규칙이 있는가?
- [ ] 프로젝트 전용 내용과 공용 규칙을 분리했는가?
- [ ] 공용 규칙이 특정 프로젝트 이름 없이 설명 가능한가?
- [ ] 검증되지 않은 가설을 자동 승격에서 제외했는가?
- [ ] 기존 Base 기준 문서의 중복·충돌을 확인했는가?
- [ ] 안정적이고 일반화 가능한 규칙을 Base 기존 파일에 반영했는가?
- [ ] 프로젝트 로컬 사본 또는 버전 기록의 동기화 필요 여부를 확인했는가?
```

공용화할 내용이 없으면 `Base 자동 승격: 없음 — 프로젝트 전용 또는 단발성 변경`으로 보고한다.
