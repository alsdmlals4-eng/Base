# MVP 작업 체크리스트

이 문서는 실제 작업 시작/종료 때 사용하는 공용 체크리스트다.

## 1. 표준 작업 순서

```text
0. 작업 유형 분류
1. 사용자 최신 지시 확인
2. 프로젝트의 Base 로컬 사본 확인
3. 프로젝트 전용 규칙 확인
4. 현재 Issue / MVP 기준선 확인
5. 작업 준비 상태 체크
6. 벤치마킹 필요 여부 판단
7. 구현 범위 / 제외 범위 확정
8. 위험 분석 / 영향 파일 예상
9. GitHub Issue 작성 또는 수정
10. HTML 전체 흐름 대시보드 갱신
11. Codex Goal 작성
12. 구현
13. 실행 검수
14. 결과 보고 / 회고
15. Base 승격 후보 여부 판단
```

## 2. 작업 유형 분류

| 유형 | 설명 | 경로 |
|---|---|---|
| 빠른 수정 | 오탈자, 단순 문서, 단일 데이터 | 빠른 경로 |
| 기능 작업 | 새 기능, 여러 파일, 저장/불러오기 | 기능 경로 |
| 기획 작업 | 시스템/루프/플레이어 경험 설계 | 기획 경로 |
| 검수 작업 | 구현 결과 확인, 오류 분석 | 검수 경로 |
| 규칙 정리 | 반복 규칙 문서화 | 문서 경로 |

## 3. 작업 시작 체크리스트

```md
- [ ] 사용자의 최신 지시를 확인했는가?
- [ ] 작업 유형을 분류했는가?
- [ ] Base 공용 규칙과 프로젝트 전용 규칙을 구분했는가?
- [ ] 현재 Issue 또는 Goal이 있는가?
- [ ] 이전 작업과 다음 작업이 연결되는가?
- [ ] 벤치마킹이 필요한 작업인가?
- [ ] 구현 AI와 기획 AI의 역할이 분리되어 있는가?
- [ ] 대시보드/문서/구현 범위가 섞이지 않았는가?
```

## 4. 좋은 프롬프트 변환 체크리스트

사용자의 짧은 요청은 그대로 넘기지 않고 아래 항목을 보강한다.

```md
- [ ] 목표
- [ ] 배경
- [ ] 사용자의 의도
- [ ] 플레이어 경험 또는 사용자 가치
- [ ] 구현 범위
- [ ] 제외 범위
- [ ] 데이터 요구사항
- [ ] 엔진/씬/노드/파일 영향
- [ ] 완료 기준
- [ ] 테스트 방법
- [ ] 결과물 형식
```

## 5. Definition of Ready

구현을 시작하기 전에 다음 조건을 만족해야 한다.

```md
- [ ] 해결할 사용자 문제와 MVP 경계를 한 문장으로 설명할 수 있다.
- [ ] 완료 기준과 제외 범위가 정리되어 있다.
- [ ] 관련 규칙, Issue, 실제 파일을 확인했다.
- [ ] 기존 변경사항과 저장소 상태를 확인했다.
- [ ] 예상 영향 파일과 저장/호환성 위험을 적었다.
- [ ] 수정 대상과 검증 방법을 작업 전에 공유했다.
```

준비되지 않은 요구사항은 구현자에게 넘기지 말고 질문, Issue 보완, 또는 작은 탐색 작업으로 분리한다.

## 6. 벤치마킹 체크리스트

```md
- [ ] 유사/인접 사례를 확인했는가?
- [ ] 게임/사례를 적었는가?
- [ ] 관찰포인트를 적었는가?
- [ ] 반응을 적었는가?
- [ ] 프로젝트 적용 이유를 적었는가?
- [ ] 반영할 것과 제외할 것을 나눴는가?
- [ ] 출처 없는 일반론으로 판단하지 않았는가?
```

권장 표:

```md
| # | 게임/사례 | 관찰포인트 | 반응 | 프로젝트 적용(+사유) |
|---:|---|---|---|---|
```

## 7. Issue 체크리스트

```md
- [ ] 목표
- [ ] 핵심 흐름
- [ ] 벤치마킹 반영/제외 결론
- [ ] 구현 범위
- [ ] 제외 범위
- [ ] 완료 기준
- [ ] 작업 전 확인 파일
- [ ] 공통 규칙 참조
- [ ] 작업 후 보고 형식
```

## 8. Codex Goal 체크리스트

```md
- [ ] Issue 번호가 맞는가?
- [ ] Base 로컬 사본과 프로젝트 전용 규칙을 참조하는가?
- [ ] 구현 범위만 들어갔는가?
- [ ] 제외 범위가 있는가?
- [ ] 완료 기준이 있는가?
- [ ] 테스트 기준이 있는가?
- [ ] 작업 후 보고 형식이 있는가?
- [ ] 대시보드나 긴 벤치마킹 전문이 빠져 있는가?
```

## 9. 구현 중 체크리스트

```md
- [ ] 요구사항과 직접 관련된 파일만 변경한다.
- [ ] 데이터, 저장, UI, 호출 흐름의 영향을 확인한다.
- [ ] 오류와 예외 상태를 고려한다.
- [ ] 요청 범위 밖 개선은 분리해 제안만 한다.
- [ ] 정상 동작하는 구조를 임의로 바꾸지 않는다.
```

## 10. 결과 검수 체크리스트

```md
- [ ] 변경 파일이 보고되었는가?
- [ ] 변경 이유가 설명되었는가?
- [ ] 구현 내용이 Issue와 일치하는가?
- [ ] 기존 흐름이 깨지지 않았는가?
- [ ] 저장/불러오기를 확인했는가?
- [ ] 사용자가 실행할 확인 순서가 있는가?
- [ ] 남은 위험이 적혀 있는가?
- [ ] 사용 도구와 미사용 도구가 사실대로 적혀 있는가?
```

## 11. 완료 조건

```md
- [ ] 완료 기준을 항목별로 확인했다.
- [ ] 적절한 테스트 또는 실행 검증을 수행했다.
- [ ] 변경 파일과 검증 결과를 기록했다.
- [ ] 사용자가 직접 확인해야 할 항목을 분리했다.
- [ ] 남은 위험이 있다면 현재 해결할 수 없는 이유를 적었다.
```

## 12. 회고와 Base 승격 체크리스트

```md
- [ ] 반복된 혼동이 있었는가?
- [ ] 다음 작업에서 줄일 규칙이 있는가?
- [ ] 공용 규칙으로 승격할 내용이 있는가?
- [ ] 프로젝트 전용으로 남길 내용이 있는가?
```

공용화할 내용이 있으면 작업 완료 보고에 `Base 승격 후보` 항목을 포함한다.
