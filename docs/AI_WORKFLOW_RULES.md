# 공용 AI 작업 흐름

## 시작 전

1. 프로젝트의 `AGENTS.md`와 로컬 공용 규칙 버전을 확인한다.
2. 현재 Issue, 완료 기준, 제외 범위, 관련 파일을 확인한다.
3. 저장소 상태와 기존 사용자 변경사항을 확인한다.
4. GPT는 작업 성격에 맞는 벤치마킹을 먼저 수행한다.
   - 게임성, 디자인, UX, 아이디어, 시장성, 구현 방식 중 해당 작업에 필요한 항목을 찾는다.
   - 벤치마킹 결과는 장문 복사가 아니라 개선점과 수정할 점으로 압축한다.
   - 적용할 내용과 제외할 내용을 나누고, 간결하고 명확한 컨텍스트로 Issue/Goal/문서에 반영한다.
5. 필요한 스킬을 최소 라우팅한다.
   - `docs/AI_SKILL_ADOPTION_GUIDE.md`의 task signal과 권한 사전점검을 따른다.
   - 외부 스킬 원문을 모두 context에 넣지 않고 현재 작업에 필요한 규칙만 읽는다.
   - 플러그인 미설치나 실패는 프로젝트 작업을 막지 않는다.
6. 수정 대상, 데이터 흐름, 검증 방법을 짧게 정리한다.

## 벤치마킹 기준

- GPT는 기획, Issue, Goal, UI/UX, 시스템 설계, 검수 작업에서 항상 벤치마킹 관점을 먼저 적용한다.
- 벤치마킹은 표면 복제가 아니라 현재 프로젝트에 맞는 개선 가설을 찾는 과정이다.
- 출처 없이 일반론으로 판단하지 않는다.
- 긴 벤치마킹 전문은 Codex Goal에 넣지 않는다.
- Goal에는 최종 반영 기준과 실행 지시만 남긴다.
- 프로젝트에 지정된 고정 참고 링크가 있으면 매 작업에서 먼저 확인한다.

## Context compact

- 긴 조사 결과는 Issue, 계획, active context capsule로 압축한 뒤 실행 단계로 넘긴다.
- 정확한 경로, 명령, 버전, 사용자 결정, 실패 증상, 다음 검증은 보존한다.
- 조사→계획, 계획→구현, 디버깅 종료처럼 논리적 경계에서 compact한다.
- 부분 구현 중에는 context를 줄이지 않는다.
- Goal과 최종 보고에서 같은 설명을 반복하지 않는다.

## 구현 중

- 작업 범위 밖 변경은 승인 없이 하지 않는다.
- 변경 전 관련 호출 위치와 데이터 계약을 확인한다.
- 문서, 코드, 데이터의 설명이 실제 동작과 일치하게 유지한다.
- 불확실하거나 외부 상태에 의존하는 사실은 확인 가능한 근거를 남긴다.

## 종료 전

1. 정적 검사, 포맷, 테스트 또는 실행 가능한 검증을 수행한다.
2. 완료 기준을 항목별로 확인한다.
3. 미검증 항목과 외부 확인이 필요한 위험을 분리한다.
4. 기준 Base 커밋이 바뀌었다면 프로젝트의 버전 기록을 갱신한다.

## Goal 작성 기준

Goal에는 목표, 작업 전 확인 항목, 구현 범위, 제외 범위, 완료 기준, 검증 방법, 작업 후 보고 형식을 포함한다. 완료 기준은 체크리스트로 만들고, 구현 가능한 사실로 쓴다.

단, Issue/로드맵/공통 규칙/벤치마킹 문서에 이미 있는 장문 내용은 Goal에 반복하지 않는다. Goal은 실행에 필요한 압축된 컨텍스트와 지시만 남긴다.
