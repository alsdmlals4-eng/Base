# 공용 기획 시스템

> 문서 위치: `docs/planning/PLANNING_SYSTEM.md` | 콘텐츠 기획 상세: `docs/CONTENT_DESIGN_METHOD.md` | 인수인계 상세: `docs/planning/HANDOFF_CONTEXT_METHOD.md`

## 1. 목적

기획을 아이디어 목록이나 기능 명세의 모음으로 만들지 않고, **사용자 가치 → 경험 → 규칙 → 콘텐츠 → 표현 → 구현 경계 → 검증**이 이어지는 하나의 판단 체계로 만든다.

이 문서는 프로젝트의 모든 기획서를 같은 구조로 정렬하고, 구현 사실·승인 계획·미확정 가설을 혼동하지 않게 하는 공용 기준이다.

## 2. 기획의 여덟 층

```text
1. 문제와 목적
2. 사용자 약속
3. 핵심 행동과 판단
4. 감정과 즉시 반응
5. 규칙과 상태 변화
6. 콘텐츠와 표현 방식
7. 구현 경계와 위험
8. 검증과 학습
```

각 층은 아래 질문에 답한다.

| 층 | 질문 |
|---|---|
| 문제와 목적 | 누구의 어떤 문제를 왜 지금 해결하는가? |
| 사용자 약속 | 사용자가 무엇을 기대하고 실제로 무엇을 받는가? |
| 핵심 행동과 판단 | 사용자가 반복해서 무엇을 보고 선택하거나 조작하는가? |
| 감정과 즉시 반응 | 행동 직후 무엇을 느끼고 어떻게 알아차리는가? |
| 규칙과 상태 변화 | 어떤 조건·입력·판정·결과·기록이 있는가? |
| 콘텐츠와 표현 | 같은 규칙을 어떤 장면·대사·아트·연출로 경험하는가? |
| 구현 경계와 위험 | 무엇을 소유하고 무엇을 침범하지 않는가? |
| 검증과 학습 | 어떤 관찰 결과로 성공·실패·재설계를 판단하는가? |

상위 층이 불명확하면 하위 기능과 아트 제작을 시작하지 않는다.

## 3. 기획 문서 계층

### 프로젝트 상태 문서

현재 구현, 승인 계획, 진행 중, 백업을 구분한다. 모든 기획서보다 먼저 읽는 짧은 문서다.

### 프로젝트 방향 기획서

프로젝트 약속, 핵심 경험, 불변 조건, 금지 방향, 장기 성공 기준을 책임진다.

### 분야별 기획서

- 콘텐츠·서사
- 아트디자인
- 연출·프레젠테이션
- 시스템·경제
- 기술·데이터
- 접근성·운영

분야별 문서는 상위 방향을 해석하지만 다시 정의하지 않는다.

### 실행 로드맵

단계 의존성, 진입 조건, 종료 조건, 검증, 인수인계 산출물을 책임진다.

### 구현 지시서

현재 작업에서 실제로 수정할 파일, 금지 범위, 순서, 테스트, 보고 형식을 적는다. 장기 방향을 다시 설계하지 않는다.

### 사례·연구 자료

관찰 근거와 적용·제외 판단을 보존한다. 현재 설계 원본이 아니라 의사결정 근거다.

## 4. 상태 표기

| 상태 | 의미 | 허용 표현 |
|---|---|---|
| 구현 확인 | 현재 코드·데이터·실행으로 확인됨 | “현재 작동한다” |
| 승인 계획 | 방향과 수용 기준이 승인됐으나 구현 전 | “구현할 계획이다” |
| 진행 중 | 작업 브랜치·부분 구현·미완료 검증 | “일부 구현, 검증 필요” |
| 가설 | 플레이테스트·조사·기술 확인 전 | “검증할 가설이다” |
| 백업 | 현재 작업의 기본 읽기 대상이 아님 | “과거 근거로 보존” |

문서가 있다는 사실을 구현 완료 증거로 사용하지 않는다.

## 5. 한 주제 한 원본

같은 사실을 여러 문서에 장문으로 복사하지 않는다.

- 상태는 상태 문서
- 방향은 프로젝트 방향서
- 상세 시스템은 GDD 또는 시스템 명세
- 분야별 표현은 해당 분야 기획서
- 단계와 순서는 로드맵
- 실제 값과 ID는 코드·데이터
- 완료 근거는 테스트·QA 기록

다른 문서는 원본 링크와 작업에 필요한 차이만 적는다.

## 6. 기획 결정 카드

중요한 결정은 다음 형식으로 남긴다.

```md
## 결정

- 질문:
- 현재 문제:
- 사용자 가치:
- 선택한 방향:
- 대안:
- 적용 이유:
- 제외 이유:
- 영향 범위:
- 위험:
- 검증 방법:
- 상태:
```

결정 카드가 없으면 나중에 같은 논의를 반복하거나, 이유를 잃고 표면만 유지할 가능성이 높다.

## 7. 기획의 입력과 산출물

### 입력

- 사용자 요청과 우선순위
- 현재 구현 상태
- 실제 사용자 행동·피드백
- 데이터·기술·예산·일정 제약
- 기존 사례와 최신 외부 근거
- 접근성·플랫폼·배포 조건

### 산출물

- 한 줄 목표
- 사용자 경험 흐름
- 포함·제외 범위
- 규칙과 상태 변화
- 아트·연출·대사 방향
- 구현 경계와 파일 책임
- 단계별 로드맵
- 테스트 가능한 완료 기준
- 남은 가설과 위험

## 8. 범위 자르기

큰 기능을 메뉴·파일 단위가 아니라 **사용자 경험의 수직 단위**로 자른다.

좋은 수직 단위:

```text
진입 조건
→ 사용자 행동
→ 시스템 반응
→ 결과와 기록
→ 저장·복귀
→ 검증
```

나쁜 분할:

- 데이터 구조만 만들고 화면·결과 없음
- UI만 만들고 실제 상태 연결 없음
- 모든 콘텐츠를 한꺼번에 작성하고 한 장면도 실행되지 않음
- 전체 아트 자산을 만든 뒤 사용 장면을 결정함

## 9. 우선순위 기준

1. 핵심 경험을 이해하거나 공정하게 만드는가?
2. 첫 사용·첫 10분의 혼란을 줄이는가?
3. 사용자의 반복 행동과 즉시 반응을 강화하는가?
4. 기존 시스템을 재사용해 작은 단위로 검증 가능한가?
5. 캐릭터·세계관·브랜드 애착을 강화하는가?
6. 접근성과 제작 효율을 개선하는가?
7. 콘텐츠 양만 늘리고 핵심을 흐리지는 않는가?

## 10. 위험 분류

| 위험 | 예시 | 기획 대응 |
|---|---|---|
| 의미 위험 | 핵심 경험이 다른 장르로 바뀜 | 프로젝트 불변 조건과 금지 방향 확인 |
| 상태 위험 | 저장·진행·경제·엔딩 변경 | 별도 승인, 이관·회귀 계획 |
| 정보 위험 | 정답·비밀·미확정 사실 누설 | 정보 공개 단계와 역할 분리 |
| UI 위험 | 핵심 행동이 가려짐 | 시선 우선순위와 해상도 검증 |
| 생산 위험 | 자산 수가 과도함 | 티어·폴백·재사용 계획 |
| 인수 위험 | 이유와 상태를 새 담당자가 이해 못함 | 책임 원본·위치 표시줄·handoff capsule |
| 연구 위험 | 사례 표면 복제·출처 오해 | 관찰·해석·결정 분리 |

## 11. 기획 검토 순서

```text
목표와 사용자 가치
→ 현재 구현·제약
→ 핵심 행동·질문
→ 정보와 상태 변화
→ 콘텐츠·아트·연출
→ 범위·위험·예외
→ 완료 기준
→ 수직 PoC
→ 플레이테스트
→ 문서 갱신·사례 축적
```

## 12. 완료 기준 작성

기능 이름이 아니라 관찰 가능한 결과로 쓴다.

나쁜 예:

- 표정 시스템 구현
- 관계 시스템 개선
- 연출 강화

좋은 예:

- 존재하지 않는 표정 키를 요청해도 기본 표정으로 진행되며 대사가 중단되지 않는다.
- 선택 후 다음 장면의 말투와 정보 공유가 달라지고 저장 후에도 유지된다.
- 연출 감소 옵션에서 화면 흔들림이 사라져도 경고 의미가 텍스트와 아이콘으로 전달된다.

## 13. 기획 리뷰 체크

- [ ] 문제와 사용자 가치가 한 문장으로 설명된다.
- [ ] 구현 사실과 승인 계획이 구분돼 있다.
- [ ] 사용자가 실제로 하는 행동이 적혀 있다.
- [ ] 행동 직후의 반응과 기록이 있다.
- [ ] 상태 소유자와 표현 계층이 분리돼 있다.
- [ ] 포함·제외 범위가 있다.
- [ ] 실패·취소·복귀·중복 입력·저장 예외가 있다.
- [ ] 가장 작은 수직 PoC가 있다.
- [ ] 완료 기준이 관찰 가능하다.
- [ ] 새 담당자가 참조할 실제 파일 위치가 있다.
