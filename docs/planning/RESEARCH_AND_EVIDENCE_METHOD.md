# 공용 정보수집·근거화 방법

> 문서 위치: `docs/planning/RESEARCH_AND_EVIDENCE_METHOD.md` | 벤치마킹 기록 형식: `docs/BENCHMARKING_REFERENCE_GUIDE.md` | 사례: `docs/planning/CASE_LIBRARY.md`

## 1. 목적

기획 조사와 벤치마킹을 작품 목록 수집이나 취향 확인으로 끝내지 않고, **실제 설계 질문에 답하는 근거 수집 → 관찰 → 해석 → 결정 → 검증** 과정으로 만든다.

조사는 결정을 대신하지 않는다. 사례는 가설의 근거이며 현재 프로젝트의 사용자, 제약, 실제 테스트가 최종 판단 기준이다.

## 2. 조사 전 질문 정의

검색을 시작하기 전에 다음을 적는다.

- 해결할 질문
- 왜 지금 필요한가
- 결정해야 할 선택지
- 이미 알고 있는 사실
- 확인되지 않은 가정
- 필요한 최신성
- 조사 시간과 사례 수 상한
- 결과를 반영할 문서와 검증 방법

나쁜 질문:

> 좋은 UI 사례를 찾아라.

좋은 질문:

> 16:9 대화 화면에서 두 인물과 3개 선택지를 동시에 보여주면서 얼굴과 선택의 가독성을 유지한 사례는 어떤 정보 위계를 사용하는가?

## 3. 조사 유형

| 유형 | 목적 | 권장 근거 |
|---|---|---|
| 개념 조사 | 용어·원리·방법 이해 | 공식 문서, 논문, 전문가 자료 |
| 제품 벤치마킹 | 실제 사용자 흐름·표현 확인 | 공식 페이지, 데모, 매뉴얼, 실제 사용 |
| 시장 조사 | 현재 가격·수요·경쟁·플랫폼 판단 | 최신 상점·공식 발표·시장 데이터 |
| 기술 조사 | API·엔진·포맷·성능 가능성 | 공식 문서, 소스 코드, 릴리스 노트 |
| 사용자 반응 | 반복되는 장점·불편 확인 | 리뷰, 포럼, 플레이테스트 기록 |
| 내부 사례 조사 | 과거 프로젝트 결정 재사용 | Git history, QA, 회고, 사례 카드 |

시간에 따라 변할 수 있는 정보는 반드시 최신 출처를 다시 확인한다.

## 4. 출처 우선순위

1. 공식 문서·원본 데이터·실제 실행
2. 개발자 발표·연구·기술 문서
3. 신뢰할 수 있는 전문 분석
4. 다수의 실제 사용자 반응
5. 커뮤니티 요약과 2차 해석

하위 출처만 있을 때는 불확실성을 명시한다. 사용자 반응은 제품 사실과 분리한다.

## 5. 관찰·해석·결정 분리

조사 기록은 세 칸으로 나눈다.

| 구분 | 질문 | 예시 |
|---|---|---|
| 관찰 | 실제로 무엇을 확인했는가? | 선택지는 대사창 위에 표시되고 현재 화자를 가리지 않는다. |
| 해석 | 왜 효과가 있다고 보는가? | 시선 이동이 짧아 선택 전 맥락을 유지하기 쉽다. |
| 결정 | 우리 프로젝트에 무엇을 적용할 것인가? | 최소 해상도에서 선택지를 대사창 상단에 고정해 PoC한다. |

관찰과 해석을 섞으면 사례의 표면을 잘못 복제할 위험이 커진다.

## 6. 사례 선택

사례 수보다 질문과의 적합성이 중요하다.

- 빠른 수정: 1~3개
- 일반 기능·콘텐츠 판단: 3~5개
- 핵심 경험·시장·큰 방향 전환: 8~12개 이상
- 기술 가능성: 공식 문서와 최소 재현 예제 우선
- 이미 방향이 확정된 구현: 새 조사보다 기존 근거와 충돌 여부 확인

서로 다른 장르와 플랫폼을 비교할 때는 원리와 제약을 분리한다.

## 7. 벤치마킹 카드

```md
## 사례

- 사례명:
- 출처와 확인일:
- 해결하려는 질문:
- 실제 관찰:
- 사용자 반응 또는 검증:
- 적용 가능한 원리:
- 그대로 쓰면 안 되는 점:
- 프로젝트 적용:
- 제외:
- 추가 검증:
```

외형·문구·고유 규칙을 복제하지 않는다.

## 8. 근거 지도

큰 기획서는 결정마다 근거 위치를 연결한다.

```text
결정 ID
→ 사용자 요청
→ 내부 구현 사실
→ 외부 근거
→ 적용 판단
→ PoC 또는 테스트
→ 최종 상태
```

근거 지도는 모든 조사 원문을 기획서에 복사하는 대신 필요한 출처를 찾게 한다.

## 9. 정보의 최신성

다음 정보는 오래된 기억을 사용하지 않고 확인한다.

- 제품·엔진·API 버전
- 가격·라이선스·스토어 정책
- 플랫폼 요구사항
- 현재 운영 중인 기능과 서비스
- 보안·법률·개인정보 기준
- 시장 규모·사용자 수·리뷰 수
- 현재 담당자·회사 상태

문서에는 확인일과 출처를 기록한다.

## 10. 사용자 반응 분석

리뷰 한두 개를 일반화하지 않는다.

반응을 다음으로 분리한다.

- 반복되는 칭찬
- 반복되는 불편
- 특정 사용자군의 요구
- 학습 곡선과 첫 사용 문제
- 장기 사용과 반복 피로
- 기술 문제와 설계 문제
- 기대와 실제 경험의 차이

강한 표현보다 반복성과 맥락을 본다.

## 11. 내부 프로젝트 근거

과거 프로젝트 자료를 읽을 때도 같은 기준을 적용한다.

- 현재 구현과 당시 계획을 구분한다.
- 완료 QA와 미완성 제안을 구분한다.
- 프로젝트 특화 요소를 공용 원칙으로 오인하지 않는다.
- Git history에서 정확한 원문과 변경 이유를 확인한다.
- 이미 해결된 문제를 새 작업처럼 반복하지 않는다.

## 12. 조사 결과의 압축

최종 기획서에는 다음만 남긴다.

```text
관찰
→ 현재 문제의 개선점
→ 적용 또는 제외 이유
→ 위험
→ 검증 방법
```

장문 원문, 모든 링크, 검색 과정은 별도 연구 문서나 사례 라이브러리에 보존한다.

## 13. 조사 중단 기준

다음 중 하나를 충족하면 검색을 멈추고 PoC 또는 결정을 진행한다.

- 서로 다른 3개 이상 사례에서 같은 원리가 반복됨
- 공식 문서로 기술 가능성이 확인됨
- 선택지 간 차이를 판단할 충분한 근거가 있음
- 추가 사례가 기존 결론을 반복함
- 현재 위험은 조사보다 실제 사용자 테스트로만 확인 가능함

조사량을 품질로 착각하지 않는다.

## 14. 실패 패턴

- 작품 이름과 스크린샷만 모음
- 최신성 확인 없이 기억으로 현재 제품을 설명
- 공식 사실과 사용자 의견을 혼합
- 장르·플랫폼 차이를 무시하고 외형 복제
- 조사 원문을 구현 지시서에 그대로 넣음
- 적용하지 않은 이유를 기록하지 않음
- 프로젝트 내부 QA를 구현 완료 근거로 오해
- 출처가 없는 일반론을 권위 있는 규칙처럼 사용

## 15. 검수 체크

- [ ] 조사 질문과 결정 대상이 명확하다.
- [ ] 필요한 최신성과 출처 수준을 정했다.
- [ ] 관찰·해석·결정을 분리했다.
- [ ] 사례의 제약과 우리 프로젝트 제약을 비교했다.
- [ ] 반드시 반영·조건부 반영·제외를 적었다.
- [ ] 적용 후 검증 방법이 있다.
- [ ] 장문 조사 원문과 실행 지시를 분리했다.
- [ ] 시간이 지나면 바뀔 정보에 확인일이 있다.
- [ ] 내부 사례의 구현 상태와 계획 상태를 구분했다.
- [ ] 추가 조사가 실제 결정 품질을 높이는지 판단했다.
