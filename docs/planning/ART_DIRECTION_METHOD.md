# 공용 아트디자인 기획 방법

> 문서 위치: `docs/planning/ART_DIRECTION_METHOD.md` | 연출 설계: `docs/planning/PRESENTATION_DIRECTION_METHOD.md` | 사례: `docs/planning/CASE_LIBRARY.md`

## 1. 목적

아트 방향을 “예쁘게”, “고급스럽게”, “몰입감 있게” 같은 취향 문장으로 남기지 않고, **사용자 경험·브랜드 약속·정보 가독성·제작 제약·자산 운영**으로 분해하는 공용 방법이다.

아트디자인은 독립적인 장식 작업이 아니다. 사용자가 무엇을 먼저 보고, 어떤 감정을 느끼며, 어떤 대상과 상태를 구분하는지를 책임진다.

## 2. 아트 방향의 여섯 질문

1. 이 프로젝트의 한 줄 시각 약속은 무엇인가?
2. 사용자가 화면을 보고 가장 먼저 느껴야 할 감정은 무엇인가?
3. 어떤 정보와 대상이 즉시 구분되어야 하는가?
4. 어떤 스타일 요소를 얼마나 섞을 것인가?
5. 실제 제작·엔진·해상도·일정 제약은 무엇인가?
6. 일관성을 어떤 자산 규격과 QA로 유지할 것인가?

## 3. 시각 약속 문장

좋은 시각 약속은 다음을 포함한다.

```text
대상 또는 세계
+ 핵심 분위기
+ 정보 가독성 또는 사용 상황
+ 차별화되는 재료·형태·색
```

예시 형식:

> 익숙한 도시 공간을 절제된 기록물 질감과 제한된 신호색으로 표현해, 인물의 표정과 비정상적인 변화가 먼저 읽히는 시각 경험.

프로젝트 이름과 고유 설정은 프로젝트 문서에 둔다.

## 4. 스타일 혼합표

여러 스타일을 섞을 때 비율과 역할을 적는다.

| 스타일 요소 | 역할 | 사용 범위 | 피할 과잉 |
|---|---|---|---|
| 기본 렌더링 | 전체 일관성 | 대부분의 캐릭터·배경 | 모든 요소를 같은 밀도로 그림 |
| 질감 요소 | 장르·재료감 | 기록, 오염, 특정 배경 | 화면 전체의 가독성 저하 |
| 강조 스타일 | 능력·감정·브랜드 순간 | 컷인, 핵심 장면 | 상시 반복으로 특별함 소실 |
| 실험 스타일 | 회상·특수 상태 | 제한된 장면 | 기본 화면과 규칙 없는 충돌 |

정확한 비율은 프로젝트에서 정하지만, 각 스타일의 역할을 설명할 수 있어야 한다.

## 5. 시각 축 정의

아트 방향은 최소 네 축으로 기록한다.

### 형태

- 인체 비율
- 실루엣
- 선의 굵기와 성격
- 각진 형태와 둥근 형태의 비율
- 캐릭터·오브젝트·UI의 형태 언어

### 명암과 재료

- 셰이딩 단계
- 광택과 거칠기
- 종이·금속·천·유리 등 재료 구분
- 광원 방향과 대비
- 배경과 캐릭터의 분리 방식

### 색

- 기본 중성색
- 브랜드 강조색
- 위험·성공·상호작용 색
- 캐릭터별 식별색
- 색각 이상에서도 구분할 보조 신호

### 질감과 후처리

- 종이, 필름, 잉크, 노이즈, 픽셀, 붓자국 등
- 기본 화면과 특수 장면의 강도 차이
- 텍스트와 얼굴 가독성을 해치지 않는 상한
- 엔진에서 재현 가능한 방식

## 6. 레퍼런스 보드 작성

레퍼런스는 작품 이름 모음이 아니라 축별 판단 자료다.

```text
형태 레퍼런스
색·조명 레퍼런스
재료·질감 레퍼런스
레이아웃·정보 위계 레퍼런스
캐릭터 태도·포즈 레퍼런스
피해야 할 반례
```

각 이미지에는 다음을 기록한다.

- 관찰할 요소
- 그대로 복제하면 안 되는 요소
- 프로젝트에 적용할 원리
- 실제 제작 제약

## 7. 캐릭터 디자인 카드

캐릭터별로 다음을 정한다.

- 역할과 첫인상
- 실루엣 키워드
- 대표 자세와 무게 중심
- 얼굴·헤어·의상에서 가장 중요한 식별 요소
- 고유 강조색
- 업무·전투·일상 소품
- 감정을 숨기는 방식
- 자주 필요한 표정
- 다른 캐릭터와 겹치면 안 되는 요소
- 자산 티어와 우선순위

외형 설명과 서사 역할을 연결한다. 예를 들어 방어적인 인물이 항상 두꺼운 갑옷을 입어야 하는 것은 아니지만, 공간을 막거나 다른 인물 앞에 서는 자세로 역할을 보여줄 수 있다.

## 8. 자산 티어

모든 자산을 같은 우선순위로 만들지 않는다.

| 티어 | 용도 | 예시 |
|---|---|---|
| T0 필수 | 기능과 식별에 반드시 필요 | 기본 초상, 핵심 배경, 상호작용 아이콘 |
| T1 반복 | 자주 보는 장면의 품질 | 표정 세트, 주요 포즈, 공용 컷인 |
| T2 강조 | 중요 장면의 기억 | 결정적 이벤트 컷, 특수 상태 |
| T3 확장 | 반복 피로 완화·수집 | 대체 의상, 소품 변형, 희귀 연출 |

PoC에서는 T0와 핵심 T1만 만든다.

## 9. 생성형 이미지 사용 원칙

- 이미지 안에 이름·대사·UI 텍스트를 굽지 않는다.
- 캐릭터 정체성과 소품·색·실루엣을 프롬프트보다 별도 카드로 관리한다.
- 레퍼런스 이미지는 정체성·포즈·재료 중 어떤 요소를 참조하는지 구분한다.
- 원본 출력과 후처리 파일을 분리해 보존한다.
- 알파·크기·여백·얼굴 위치·텍스트 없음 여부를 manifest에 기록한다.
- 프롬프트는 화풍 작가명보다 형태·광원·재료·구도·용도를 설명한다.
- 생성 결과의 한글·로고·문양은 신뢰하지 않고 엔진 UI나 수작업 벡터로 분리한다.
- 자산이 일관성 기준을 충족하지 않으면 개별 수정 전에 공통 레퍼런스와 프롬프트 구조를 고친다.

## 10. 아트 브리프 구조

```md
- 자산명과 용도
- 화면에서 보이는 크기
- 사용자에게 먼저 보여야 할 요소
- 캐릭터·오브젝트의 핵심 실루엣
- 색·광원·재료
- 배경과의 대비
- 필요한 알파·해상도·비율
- 금지 요소
- 참조 이미지의 관찰 요소
- 폴백 또는 재사용 자산
- QA 항목
```

## 11. 일관성과 변주

일관성은 모든 자산을 같게 만드는 것이 아니다.

고정할 것:

- 선화와 셰이딩 원칙
- 얼굴 비율과 신체 기준
- 재료 표현의 상한
- 강조색 사용 방식
- 카메라와 여백 규칙
- 텍스트 없는 원본 자산

변주할 것:

- 캐릭터별 실루엣과 무게 중심
- 장소별 조명과 재료
- 특수 상태의 질감
- 감정 강도와 포즈
- 이야기에서 중요한 소품

## 12. 엔진·UI와의 경계

- 이미지 자산은 의미 텍스트를 소유하지 않는다.
- UI가 색·아이콘·문구를 합성한다.
- 캐릭터 아트는 상태를 표현하지만 게임 결과를 결정하지 않는다.
- 해상도와 크롭은 실제 화면 슬롯을 기준으로 검증한다.
- 배경·캐릭터·VFX의 명암이 대사와 선택 가독성을 방해하지 않게 한다.
- 자산 누락 시 기본 이미지 또는 실루엣으로 폴백한다.

## 13. 아트 QA

### 기술

- [ ] 해상도와 화면비가 명세와 일치한다.
- [ ] 필요한 알파가 있고 가장자리 오염이 없다.
- [ ] 파일명·ID·manifest가 일치한다.
- [ ] 엔진에서 실제 로드된다.
- [ ] 이미지 안에 불필요한 텍스트·워터마크·로고가 없다.

### 시각

- [ ] 축소 화면에서도 실루엣과 얼굴이 읽힌다.
- [ ] 캐릭터가 서로 구분되지만 같은 작품에 속해 보인다.
- [ ] 강조색이 정보 색과 충돌하지 않는다.
- [ ] 재료와 광원이 장면마다 무작위로 바뀌지 않는다.
- [ ] 배경이 인물과 현재 행동을 압도하지 않는다.

### 경험

- [ ] 사용자가 자산의 역할을 즉시 이해한다.
- [ ] 중요한 표정·소품·상호작용이 가려지지 않는다.
- [ ] 특수 효과가 반복되어 피로하지 않다.
- [ ] 접근성 옵션과 텍스트 폴백이 있다.

## 14. 실패 패턴

- 레퍼런스 작품 이름만 적고 관찰 요소를 정의하지 않음
- 스타일 키워드를 너무 많이 섞어 기준이 없음
- 캐릭터마다 다른 렌더링 규칙을 사용함
- 고해상도 원화만 보고 실제 게임 크롭을 검증하지 않음
- 생성 이미지 안의 글자를 그대로 사용함
- 전체 자산을 먼저 제작하고 실제 장면 우선순위를 나중에 정함
- 아트가 상태와 기능을 대신 설명하도록 맡김
- 폴백이 없어 한 자산 누락으로 화면이 중단됨

## 15. 완료 기준

아트 방향서는 다음 질문에 답할 수 있어야 한다.

- 어떤 화면을 봐도 같은 프로젝트라고 판단할 근거는 무엇인가?
- 캐릭터와 상태를 어떤 형태·색·재료로 구분하는가?
- 무엇이 필수 자산이고 무엇이 후순위인가?
- 생성·외주·내부 제작에 같은 브리프를 사용할 수 있는가?
- 실제 엔진 화면에서 가독성과 일관성을 어떻게 검증하는가?
- 자산이 없을 때 어떤 폴백으로 기능을 유지하는가?
