# 공용 기획 스킬 라이브러리

> 문서 위치: `docs/planning/PLANNING_SKILL_LIBRARY.md` | 외부 스킬 채택: `docs/AI_SKILL_ADOPTION_GUIDE.md` | 조사 방법: `docs/planning/RESEARCH_AND_EVIDENCE_METHOD.md`

## 1. 목적

기획·아트·연출 작업에 필요한 능력을 도구 이름이 아니라 **언제 사용하고, 어떤 입력을 받아, 어떤 산출물과 검증을 남기는가**로 정의한다.

외부 플러그인·스킬·MCP·에이전트는 아래 능력을 보조하는 선택 수단이다. 설치 여부나 브랜드가 품질을 보장하지 않는다.

## 2. 핵심 스킬 지도

| 스킬 | 사용 시점 | 핵심 산출물 |
|---|---|---|
| 의도 합성 | 사용자 요청이 모호하거나 여러 목표가 섞임 | 문제·목표·사용자 가치·제외 범위 |
| 현재 상태 감사 | 기존 프로젝트를 이어받음 | 구현 사실·승인 계획·차이표 |
| 기획 구조화 | 아이디어를 실행 가능한 계획으로 바꿈 | 방향서·분야별 기획서·로드맵 |
| 콘텐츠 정체성 설계 | 사건·장면·기능을 추가함 | 한 줄 약속·핵심 동사·차별점·PoC |
| 서사·대화 설계 | 캐릭터·선택·관계 콘텐츠 작성 | 장면 비트·목소리 카드·선택 기억 |
| 아트디렉션 | 시각 스타일과 자산 기준 수립 | 시각 약속·스타일 축·자산 티어·QA |
| 연출 설계 | 화면·표정·컷인·음향 구성 | 시선 흐름·연출 비트·폴백·접근성 |
| 벤치마킹 | 새로운 문제 해결 근거 필요 | 관찰·적용·제외·검증 카드 |
| 정보 검증 | 최신 사실·기술·시장 판단 | 출처·확인일·불확실성·결론 |
| 수직 PoC 설계 | 큰 계획의 위험 가정 검증 | 최소 장면·입력·결과·테스트 |
| 수용 기준 작성 | 구현 완료를 판단해야 함 | 관찰 가능한 조건·행동·결과 |
| 인수인계 압축 | 담당자·AI·계정이 바뀜 | 읽기 순서·책임 원본·다음 작업 |
| 회고·지식 승격 | 작업 종료 후 교훈 축적 | 실패 원인·재발 방지·공용 사례 |

## 3. 의도 합성

### 입력

- 사용자 요청
- 문제 배경
- 기존 작업과 결정
- 일정·품질·예산 우선순위

### 수행

- 결과물보다 해결할 문제를 먼저 찾는다.
- 사용자가 원하는 감정·행동·품질 기준을 추출한다.
- 서로 충돌하는 목표를 표시한다.
- 추정한 내용과 확인된 내용을 분리한다.

### 산출물

```text
목표
사용자 가치
현재 문제
포함 범위
제외 범위
결정 필요 항목
```

### 실패 신호

- 요청 문장을 그대로 길게 반복함
- 질문 없이 구현 세부를 임의 결정함
- 결과물 형식만 정하고 성공 기준이 없음

## 4. 현재 상태 감사

### 입력

- 저장소·문서·데이터·실행 화면
- 최근 커밋·Issue·QA
- 전달 패키지

### 수행

- 현재 구현과 승인 계획을 구분한다.
- 실제 파일이 문서와 다른 부분을 기록한다.
- 보호할 ID·저장·상태 소유자를 확인한다.
- 오래된 자료와 현행 원본을 분리한다.

### 산출물

- 기준선
- 차이표
- 위험 경로
- 이번 작업에서 사용할 실제 원본
- 미확인 항목

## 5. 기획 구조화

복잡한 작업은 다음 문서 계층으로 나눈다.

```text
상태 문서
→ 프로젝트 방향
→ 분야별 기획
→ 실행 로드맵
→ 구현 지시
→ 테스트·QA
→ 사례·회고
```

같은 내용을 복사하지 않고 각 문서가 책임질 질문을 정한다.

## 6. 서사·대화 스킬

- 장면의 중심 질문을 하나로 제한
- 행동만으로 장면 흐름을 먼저 작성
- 캐릭터별 판단 기준과 말의 리듬 정의
- 기능 대화와 관계 대화의 밀도 분리
- 선택이 이후 반응·정보 공유·장면 해금으로 남게 설계
- 장문 설명을 갈등·행동·소품으로 변환
- 말투·정보 누설·반복·분량 검수

도구 예시:

- 스프레드시트: 장면·선택·조건·후속 반응 매트릭스
- JSON/YAML: 구조화된 대사·이벤트 레지스트리
- LLM: 초안 변형·말투 비교·반대 검토
- 실제 플레이: 타이밍·피로·선택 이해 검증

LLM의 대사는 캐릭터 원본과 실제 데이터 검수 없이 바로 적용하지 않는다.

## 7. 아트디렉션 스킬

- 시각 약속 문장 작성
- 형태·색·명암·재료·질감 축 분해
- 레퍼런스 보드의 관찰 요소 라벨링
- 캐릭터 실루엣·소품·강조색 차별화
- 자산 T0~T3 우선순위
- 생성형 이미지 프롬프트와 manifest 설계
- 실제 화면 크롭·알파·텍스트 없음 QA

도구 예시:

- 이미지 보드·Figma: 레퍼런스 축과 화면 배치
- 이미지 생성 모델: 텍스트 없는 원본 자산 초안
- 이미지 편집: 알파·크롭·색·일관성 후처리
- 엔진 미리보기: 실제 슬롯과 해상도 검증

## 8. 연출 스킬

- 첫 시선과 중심 질문 정의
- 장면 비트와 카메라 목적 분해
- 표정 의미 키와 폴백 설계
- 컷인 유형·빈도·타이밍 제한
- 음향·색상과 동등한 텍스트 신호 설계
- 모션 감소·점멸 제한·키보드 포커스 검수
- 프레젠테이션 계층과 상태 로직 분리

도구 예시:

- 스토리보드·애니매틱
- 화면 목업
- 타임라인·상태 다이어그램
- 입력·해상도 테스트

## 9. 벤치마킹 스킬

- 질문에 맞는 사례 1~5개 선택
- 실제 흐름과 1차 출처 확인
- 관찰·해석·결정 분리
- 적용·조건부 적용·제외 기록
- 다른 장르·플랫폼의 제약 분리
- 최신성 확인
- PoC와 사용자 테스트로 결론 갱신

검색량보다 결정에 필요한 근거를 우선한다.

## 10. 수직 PoC 스킬

### 최소 단위

```text
진입 조건
→ 사용자 행동 1~3개
→ 성공 결과 1개
→ 실패 결과 1개
→ 기록 또는 상태 변화 1개
→ 복귀·저장 검증
```

### 검증 질문

- 사용자가 핵심 행동을 이해하는가?
- 다시 하고 싶어 하는가?
- 성공·실패 이유를 설명하는가?
- 주변 시스템 없이 핵심 가치가 성립하는가?
- 제작 비용과 위험을 줄였는가?

## 11. 수용 기준 작성 스킬

완료 기준은 `조건 → 행동 → 관찰 결과`로 작성한다.

예시:

```text
자산에 요청한 표정 키가 없을 때
→ 대화 장면을 재생하면
→ 기본 표정으로 폴백하고 입력·대사·저장 상태가 유지된다.
```

테스트할 수 없는 형용사는 구체적 관찰로 바꾼다.

## 12. 인수인계 압축 스킬

- 새 담당자가 먼저 읽을 문서를 3~6개로 제한
- 현재 구현·승인 계획·진행 중·백업 상태 구분
- 분야별 책임 원본과 실제 파일 위치 연결
- 다음 작업의 진입 기준·위험·완료 기준 명시
- 오래된 자료를 기본 읽기에서 제외
- 정확한 과거 원문을 찾는 경로 보존

## 13. 외부 스킬·도구 채택

외부 스킬을 사용하기 전에 다음을 확인한다.

- 실제 필요한 능력과 일치하는가?
- 스크립트·hook·MCP·외부 API가 있는가?
- 저장소·파일·비밀값·브라우저에 어떤 권한을 요구하는가?
- 결과를 독립 검증할 수 있는가?
- 설치 없이 문서 템플릿이나 기존 도구로 해결 가능한가?
- 실패 시 원상복구와 산출물 회수가 가능한가?

자세한 기준은 `docs/AI_SKILL_ADOPTION_GUIDE.md`를 따른다.

## 14. 스킬 기록 카드

```md
## 스킬

- 이름:
- 해결하는 문제:
- 사용 조건:
- 필요한 입력:
- 수행 단계:
- 산출물:
- 검증:
- 권한·비용·위험:
- 사용하지 않을 조건:
- 관련 사례:
- 상태: 가설 / 패턴 / 원칙
```

## 15. 선택 기준

한 작업에서 모든 스킬을 사용하지 않는다.

```text
현재 문제
→ 필요한 능력 1~3개
→ 최소 도구
→ 산출물과 검증
→ 결과 회고
```

도구를 많이 쓰는 것이 아니라, 필요한 판단을 더 정확하고 반복 가능하게 만드는 것이 목표다.
