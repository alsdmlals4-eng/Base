# 공용 기획 인수인계·컨텍스트 설계 방법

> 문서 위치: `docs/planning/HANDOFF_CONTEXT_METHOD.md` | 기획 체계: `docs/planning/PLANNING_SYSTEM.md` | 템플릿: `templates/planning/HANDOFF_CONTEXT.md`

## 1. 목적

새 기획자·AI·구현 담당자가 긴 대화 기록이나 모든 과거 문서를 읽지 않고도 다음을 이해하게 한다.

- 지금 실제로 구현된 것
- 승인됐지만 아직 구현되지 않은 것
- 프로젝트가 지켜야 할 핵심 방향
- 이번 작업에서 먼저 읽을 문서와 실제 파일
- 다음 단계의 진입 조건과 완료 기준
- 바꾸면 안 되는 상태·데이터·사용자 약속

인수인계는 정보를 많이 주는 작업이 아니라, **정확한 원본과 읽기 순서를 설계하는 작업**이다.

## 2. 인수인계의 최소 컨텍스트

```text
1. 프로젝트 한 줄 정의
2. 현재 구현 기준선
3. 승인 계획과 진행 중 작업
4. 핵심 불변 조건
5. 분야별 책임 원본
6. 실제 코드·데이터·에셋 위치
7. 다음 작업의 진입·완료 기준
8. 검증·미검증·위험
9. 과거 자료를 찾는 경로
```

이 항목이 없는 장문 문서보다 위 항목을 갖춘 짧은 문서가 인수인계에 유리하다.

## 3. 권장 문서 세트

| 문서 | 책임 |
|---|---|
| 현재 상태 | 구현·승인 계획·진행 중·백업 구분 |
| 프로젝트 방향 | 사용자 약속·핵심 경험·불변 조건·금지 방향 |
| 분야별 기획서 | 서사, 아트, 연출, 시스템 등 해당 판단 기준 |
| 실행 로드맵 | 단계 의존성·진입·종료·검증 |
| 문서 라우터 | 작업 조건에 따라 읽을 문서 선택 |
| 현재 handoff | 계정·담당자 교대 시 다음 행동 |
| 사례 라이브러리 | 과거 판단 근거와 재사용 원칙 |
| 백업 인덱스 | 과거 원문과 대체 현행 문서 위치 |

한 문서에 모든 내용을 넣지 않고, 첫 문서에서 책임 원본으로 이동하게 한다.

## 4. 읽기 순서

### 새 담당자

```text
프로젝트 AGENTS 또는 작업 규칙
→ 현재 상태
→ 기획 인수인계 인덱스
→ 프로젝트 방향
→ 분야별 기획서 1개
→ 실행 로드맵
→ 실제 대상 파일
```

### 구현 담당자

```text
현재 작업 패키지 README
→ 구현 지시서
→ 현재 상태
→ 분야별 기획서
→ 보호 경로와 대상 파일
→ 테스트 체크리스트
```

### 기획 담당자

```text
현재 상태
→ 프로젝트 방향
→ 분야별 기획서
→ 실제 구현과 사용자 피드백
→ 사례·외부 근거
→ 로드맵·수용 기준 갱신
```

## 5. 상태 표기

인수인계 문서에서 다음 상태를 혼용하지 않는다.

- **구현 확인:** 현재 코드·데이터·실행으로 확인
- **승인 계획:** 방향·수용 기준 승인, 구현 전
- **진행 중:** 부분 구현 또는 미완료 검증
- **가설:** 조사·PoC·플레이테스트 전
- **백업:** 기본 읽기 대상이 아닌 과거 근거

전달 패키지와 기획서가 있다는 사실은 구현 완료 증거가 아니다.

## 6. 책임 원본 표

인수인계 인덱스에는 다음 표를 둔다.

| 질문 | 책임 원본 | 실제 확인 위치 |
|---|---|---|
| 현재 무엇이 작동하는가? | 상태 문서 | 코드·데이터·실행·테스트 |
| 왜 이 방향인가? | 프로젝트 방향서 | 사용자 결정·기획 결정 카드 |
| 어떤 방식으로 표현하는가? | 분야별 기획서 | 화면·에셋·대사·레지스트리 |
| 다음 순서는 무엇인가? | 로드맵 | Issue·Goal·작업 패키지 |
| 완료를 어떻게 판단하는가? | 테스트 체크리스트 | 자동·수동 결과 |
| 과거 판단은 어디에 있는가? | 사례·백업 인덱스 | Git history·연구 원문 |

## 7. 위치 표시줄

현행 문서 첫 부분에 한 줄 위치 표시줄을 둔다.

```text
> 문서 위치: `<현재 경로>` | 책임 원본: `<상위 원본>` | 과거·상세 백업: `<백업 경로>`
```

목적:

- 파일이 복사되거나 패키지에 포함돼도 원래 위치를 알 수 있음
- 현행 원본과 백업을 구분
- 중복 문서가 새 원본으로 오해되는 것을 방지

## 8. 차이 보고

문서와 실제 구현이 다르면 다음 형식으로 보고한다.

```md
| 항목 | 기획 문서 | 실제 구현 | 영향 | 처리 |
|---|---|---|---|---|
|  |  |  |  | 문서 갱신 / 구현 갱신 / 확인 필요 |
```

추정으로 맞추지 않는다. 상태 차이를 먼저 승인받거나, 현재 작업 범위에서 어느 쪽을 원본으로 삼을지 결정한다.

## 9. 구현 패키지 구조

권장 ZIP 구조:

```text
00_README_전달사항.md
01_구현지시서.md
02_기획서/
03_레지스트리_샘플데이터/
04_목업_참고이미지/
05_참고문서/
```

최상단 README에는 다음을 적는다.

- 목표와 사용자 가치
- 현재 구현 기준선
- 읽기 순서
- 수정·금지 범위
- 보호 경로와 고위험 영역
- 구현 순서
- 자동·수동 검증
- 완료 보고 형식
- 패키지 자료와 저장소 실제 파일의 차이 감사 지시

## 10. 인수인계 시 제거할 정보

- 이미 완료된 세부 체크리스트 전문
- 현재 작업과 무관한 과거 Goal
- 같은 사실의 여러 버전
- 장문 대화 기록
- 사용하지 않는 도구 설정
- 적용하지 않은 벤치마킹 원문 전체
- 오래된 다음 작업

삭제 대신 백업 인덱스와 정확한 Git 경로를 남긴다.

## 11. 컨텍스트 캡슐

긴 작업 중 단계가 바뀔 때 다음을 짧게 갱신한다.

```yaml
status:
current_goal:
implemented_baseline:
branch_or_workspace:
completed:
in_progress:
blocked:
protected_contracts:
next_action:
required_documents:
unverified:
```

컨텍스트 캡슐은 상세 문서를 대체하지 않고 현재 위치를 알려준다.

## 12. 완료 뒤 갱신

작업이 실제 완료되면 다음을 심사한다.

1. 현재 상태
2. 로드맵의 완료선과 다음 단계
3. 테스트 체크리스트
4. 분야별 기획서의 상태와 결정
5. handoff 캡슐
6. 문서 라우터
7. 완료 상세의 QA·백업 이동
8. 공용 사례·원칙 승격 후보

## 13. 품질 검수

새 담당자에게 다음 질문을 하고 문서만으로 답할 수 있는지 확인한다.

- 현재 구현 완료선은 무엇인가?
- 다음 작업은 무엇이며 왜 필요한가?
- 어떤 문서를 먼저 읽어야 하는가?
- 실제 파일과 기획 문서가 다르면 무엇을 우선하는가?
- 바꾸면 안 되는 계약은 무엇인가?
- 이번 단계의 완료 기준은 무엇인가?
- 과거 판단 근거는 어디에서 찾는가?

답을 위해 모든 저장소를 검색해야 한다면 인수인계 문서가 부족하다.

## 14. 인수인계 체크

- [ ] 현재 구현과 승인 계획이 분리돼 있다.
- [ ] 읽기 순서가 3~6개 핵심 문서로 제한돼 있다.
- [ ] 분야별 책임 원본이 명확하다.
- [ ] 실제 파일·데이터·에셋 위치가 연결돼 있다.
- [ ] 다음 작업의 진입·완료 기준이 있다.
- [ ] 보호할 상태·ID·사용자 약속이 있다.
- [ ] 미검증·위험·차이가 표시돼 있다.
- [ ] 오래된 정보가 기본 읽기에서 제외돼 있다.
- [ ] 정확한 과거 원문을 찾는 경로가 있다.
- [ ] 작업 완료 뒤 갱신할 문서 목록이 있다.
