# 공용 기획 사례 라이브러리

> 문서 위치: `docs/planning/CASE_LIBRARY.md` | 사례 작성 양식: `templates/planning/CASE_STUDY_CARD.md` | 조사 방법: `docs/planning/RESEARCH_AND_EVIDENCE_METHOD.md`

## 1. 목적

프로젝트와 벤치마킹에서 실제로 사용한 판단을 **문제 → 관찰 → 적용 → 제외 → 검증 → 재사용 원칙**으로 축적한다.

사례는 정답 모음이 아니다. 비슷한 문제에서 검토할 시작점이며, 현재 프로젝트의 사용자·제약·기술 상태를 확인한 뒤 적용한다.

## 2. 상태

| 상태 | 의미 |
|---|---|
| 검증됨 | 실제 구현·사용자 테스트·반복 작업에서 가치가 확인됨 |
| 조건부 | 특정 장르·화면·제약에서 유효함 |
| 가설 | 사례와 논리는 있으나 실제 검증이 부족함 |
| 폐기 | 결과가 나쁘거나 공용성이 없어 재사용하지 않음 |

## 3. CASE-001 — 장면 중심 UI와 정보 패널의 균형

- **출처 유형:** 내부 게임 프로젝트 + 도시 미스터리 벤치마킹
- **문제:** 상태 패널과 메뉴가 많아 배경·인물·현재 선택이 늦게 읽힘.
- **관찰:** 장면 몰입이 강한 작품은 공간·인물·현재 행동을 같은 시선 흐름에 두며, 상세 기록은 필요할 때 열게 한다.
- **적용:** 첫 시선, 중심 질문, 첫 선택을 화면 안에 두고 상세 정보는 보조 패널·기록 화면으로 분리.
- **제외:** 특정 작품의 프레임·폰트·카메라 외형 복제.
- **검증:** 최소 해상도에서 사용자가 장소·주요 대상·현재 행동을 순서대로 설명하는지 확인.
- **재사용 원칙:** 패널 수를 줄이는 것보다 현재 판단에 필요한 정보의 시선 순서를 먼저 정한다.
- **적용하지 않을 조건:** 데이터 분석 도구처럼 여러 수치를 동시에 비교하는 것이 핵심인 화면.
- **상태:** 검증됨.

## 4. CASE-002 — 표시 이름과 내부 식별자 분리

- **출처 유형:** 내부 프로젝트 리네이밍
- **문제:** 캐릭터·기능의 플레이어 노출 이름을 바꾸면서 내부 저장 키와 파일명을 일괄 변경하면 회귀 위험이 큼.
- **관찰:** 사용자 경험의 명칭과 영속 상태의 ID는 수명이 다르다.
- **적용:** UI·대사·문서는 새 명칭을 사용하고, 내부 ID는 호환 계층을 통해 유지.
- **제외:** 코드 전체의 즉시 문자열 치환.
- **검증:** 이전 저장·링크·이벤트가 유지되며 화면에는 구명칭이 노출되지 않는지 확인.
- **재사용 원칙:** 표시 계층과 영속 식별자를 분리하고 이관을 별도 작업으로 다룬다.
- **상태:** 검증됨.

## 5. CASE-003 — 맥락별 대사 밀도 분리

- **출처 유형:** 내부 캐릭터 중심 조사 프로젝트
- **문제:** 캐릭터 매력을 늘리기 위해 모든 장면의 대사를 늘리면 핵심 행동이 느려짐.
- **관찰:** 사용자가 판단 중인 장면은 짧은 기능 대사가 유리하고, 안전한 일상·후일담은 관계와 세계관 대화를 충분히 수용한다.
- **적용:** 기능 장면은 상황·선택·짧은 반응 중심, 관계 장면은 행동·대화·태도 선택 중심으로 대사 예산을 분리.
- **제외:** 모든 장면에 같은 길이·농도의 캐릭터 대화.
- **검증:** 기능 장면의 대사를 가려도 행동 흐름이 이해되고, 관계 장면에서는 말투만으로 인물을 구분하는지 확인.
- **재사용 원칙:** 캐릭터 목소리는 유지하되 콘텐츠 유형마다 대사의 역할과 길이를 다르게 한다.
- **상태:** 검증됨.

## 6. CASE-004 — 숫자형 호감도 대신 선택 기억

- **출처 유형:** 내부 관계 시스템 기획 + 선택 중심 서사 벤치마킹
- **문제:** 관계를 점수와 성능 보너스로만 표현하면 인물이 시스템 카드로 축소됨.
- **관찰:** 사용자는 말투·정보 공유·행동 순서·후속 장면 변화로 관계를 더 자연스럽게 인식한다.
- **적용:** 신뢰·경계·부채·경쟁·공감·보호·균열 같은 태그와 선택 기억을 대사·장면 해금에 사용.
- **제외:** 선물 반복, 상시 점수 팝업, 핵심 정답·전투 성능 보너스.
- **검증:** 숫자 없이도 사용자가 관계가 왜 달라졌는지 설명하는지 확인.
- **재사용 원칙:** 내부 계산과 사용자 경험을 분리하고 관계는 행동으로 보여준다.
- **적용하지 않을 조건:** 수치 최적화 자체가 핵심인 경영·육성 시뮬레이션.
- **상태:** 조건부.

## 7. CASE-005 — 프레젠테이션 계층의 상태 비소유

- **출처 유형:** 내부 UI·연출 아키텍처 기획
- **문제:** 표정·컷인·대화 UI가 별도 분기와 저장 상태를 소유하면 도메인 데이터와 결과가 중복됨.
- **관찰:** 연출은 상태를 읽어 표현할 때 재사용과 폴백이 쉬우며, 결과를 결정하기 시작하면 회귀와 불일치가 늘어난다.
- **적용:** 도메인 로직이 결과와 관계를 결정하고, 프레젠테이션 데이터는 의미 키·컷인 유형·자산 경로만 관리.
- **제외:** UI 스크립트에서 보상·관계·퀘스트 상태 직접 변경.
- **검증:** 연출을 끄거나 자산을 제거해도 동일한 결과와 저장 상태가 유지되는지 확인.
- **재사용 원칙:** 표현 계층은 보여주되 결정하지 않는다.
- **상태:** 검증됨.

## 8. CASE-006 — 생성 이미지와 의미 텍스트 분리

- **출처 유형:** 내부 캐릭터 아트 제작
- **문제:** 생성 이미지 안의 한글·로고·UI는 오탈자, 현지화, 수정, 해상도 대응이 어렵다.
- **관찰:** 이미지 모델은 시각 형태에 강하지만 정확한 의미 텍스트와 인터랙션에 불안정하다.
- **적용:** 텍스트 없는 원본 자산을 만들고 이름·능력·대사·버튼은 엔진·디자인 도구에서 별도 렌더링.
- **제외:** 생성 결과의 글자를 그대로 사용.
- **검증:** 자산에 구운 텍스트가 없고 UI 텍스트를 독립 수정·현지화할 수 있는지 확인.
- **재사용 원칙:** 이미지 모델은 시각 자산, 제품 UI는 의미와 상호작용을 담당한다.
- **상태:** 검증됨.

## 9. CASE-007 — 현행 원본과 백업 분리

- **출처 유형:** 내부 문서 구조 정리
- **문제:** 오래된 로드맵·완료 QA·중복 기획이 기본 읽기에 남아 새 작업마다 불필요한 확인이 반복됨.
- **관찰:** 모든 역사를 삭제할 필요는 없지만, 현재 상태·방향·검증 원본을 짧게 유지해야 인수 비용이 줄어든다.
- **적용:** 한 주제 한 현행 원본, 구현·계획 상태 분리, 조건부 라우터, 날짜별 백업, 기존 링크용 리디렉션.
- **제외:** 과거 자료 완전 삭제와 모든 문서를 기본 읽기에 유지.
- **검증:** 새 담당자가 3~6개 문서로 현재 상태와 다음 작업을 설명하는지 확인.
- **재사용 원칙:** 역사 보존과 기본 읽기 최소화를 동시에 설계한다.
- **상태:** 검증됨.

## 10. CASE-008 — 기록 재열람과 정보 은닉의 균형

- **출처 유형:** 추리 게임 벤치마킹
- **참고 사례:** Return of the Obra Dinn, The Case of the Golden Idol
- **문제:** 사용자가 과거 정보를 재검토해야 하지만 메뉴 탐색과 정답 노출을 피해야 함.
- **관찰:** 확보한 정보의 재열람은 추론 만족을 높이지만, 중요한 기록의 위치가 불명확하거나 힌트가 정답을 말하면 공정성이 무너진다.
- **적용:** 근거·힌트·실패 기록·분위기 정보를 분리하고, 현재 판단과 관련된 기록으로 바로 연결.
- **제외:** 특정 퍼즐 문법과 기록장 외형 복제.
- **검증:** 사용자가 기록 위치를 찾고 두 개 이상의 근거로 결론을 설명하는지 확인.
- **재사용 원칙:** 기록은 숨기는 수집품이 아니라 사용자가 추론을 갱신하는 도구다.
- **상태:** 조건부.

## 11. CASE-009 — 짧은 동료 반응과 실패의 기억

- **출처 유형:** 캐릭터·선택 서사 벤치마킹
- **참고 사례:** Disco Elysium
- **문제:** 선택을 인물 관계에 남기면서 현재 행동을 방해하지 않으려면?
- **관찰:** 짧고 조건부인 동료 반응은 선택을 기억하게 하며, 실패가 의미 있는 정보를 남기면 재시도가 학습으로 바뀐다.
- **적용:** 결과 직후 짧은 반응, 관계 장면에서 후속 대사, 실패 기록에 관찰 가능한 원인 포함.
- **제외:** 장문 독백과 모든 상황의 상시 반응.
- **검증:** 사용자가 수치보다 인물 반응과 실패에서 배운 내용을 기억하는지 확인.
- **재사용 원칙:** 반응의 길이보다 타이밍과 선택 관련성이 중요하다.
- **상태:** 조건부.

## 12. CASE-010 — 규칙 학습과 영구 손실의 분리

- **출처 유형:** 관리·규칙 학습 게임 벤치마킹
- **참고 사례:** Lobotomy Corporation
- **문제:** 대상 규칙과 팀 적합성을 학습하게 하면서 높은 손실 피로를 피하려면?
- **관찰:** 규칙과 적합성 학습은 긴장을 만들지만 영구 사망·반복 손실은 진입 장벽이 될 수 있다.
- **적용:** 실패 이유와 대상 반응을 명확히 남기고 회복·후퇴·재시도 여지를 제공.
- **제외:** 프로젝트 약속과 맞지 않는 영구 손실과 시설 경영 구조.
- **검증:** 사용자가 실패 이유를 설명하고 새로운 대응을 선택하는지 확인.
- **재사용 원칙:** 학습의 긴장과 자산의 영구 손실은 별개의 설계 선택이다.
- **상태:** 조건부.

## 13. CASE-011 — 목업의 역할과 실제 화면 검증

- **출처 유형:** 내부 UI·연출 제작
- **문제:** 목업이 아름답지만 실제 텍스트·데이터·입력에서 깨짐.
- **관찰:** 목업은 시선 우선순위와 미감 탐색에는 유용하지만, 실제 해상도·텍스트 길이·포커스·저장 상태를 대체하지 못한다.
- **적용:** 목업의 역할과 구조를 구현하고 실제 데이터·최소 해상도·키보드·마우스로 재검증.
- **제외:** 픽셀 위치를 그대로 복제하는 구현.
- **검증:** 긴 문장, 최대 항목 수, 자산 누락, 복귀 상태에서 레이아웃과 입력이 유지되는지 확인.
- **재사용 원칙:** 목업은 방향 문서이며 런타임 수용 기준이 아니다.
- **상태:** 검증됨.

## 14. CASE-012 — 최초 상세 안내와 재방문 축약

- **출처 유형:** 공용 튜토리얼 설계
- **문제:** 안내가 부족하면 이해하지 못하고, 매번 길게 반복하면 피로함.
- **관찰:** 첫 사용과 재방문은 필요한 정보량이 다르다.
- **적용:** 최초 상세 안내와 재방문용 짧은 안내를 분리하고 확인 상태를 저장 또는 파생.
- **제외:** 매번 같은 튜토리얼 전문과 완전 자동 생략.
- **검증:** 첫 사용자는 기능을 설명하고, 재방문 사용자는 흐름이 끊기지 않는지 확인.
- **재사용 원칙:** 학습 단계에 따라 안내 밀도를 조정한다.
- **상태:** 검증됨.

## 15. 새 사례 추가 체크

- [ ] 실제 해결하려는 문제를 적었다.
- [ ] 프로젝트·사례의 맥락과 제약을 적었다.
- [ ] 관찰과 해석을 분리했다.
- [ ] 적용한 변경과 제외한 요소를 적었다.
- [ ] 실제 검증 또는 미검증 상태를 적었다.
- [ ] 다른 작업에 재사용 가능한 원칙을 적었다.
- [ ] 적용하면 안 되는 조건을 적었다.
- [ ] 프로젝트 고유 이름·수치·경로를 제거했다.
