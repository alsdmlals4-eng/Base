# 공용 기획 지식 베이스

> 문서 위치: `docs/planning/README.md` | 기존 콘텐츠 기획 기준: `docs/CONTENT_DESIGN_METHOD.md` | 프로젝트 적용 템플릿: `templates/planning/`

이 폴더는 여러 프로젝트에서 재사용할 수 있는 **기획·아트디자인·연출·서사·정보수집·스킬·인수인계 방법**을 종류별로 분리해 누적하는 학습 데이터다.

Base에는 특정 프로젝트의 캐릭터명, 세계관, 수치, 엔진 경로, 실제 저장 구조를 두지 않는다. 프로젝트 사례에서 얻은 교훈은 문제·관찰·적용·검증 구조로 일반화하고, 원본 프로젝트의 고유 요소는 제거한다.

## 1. 사용 목적

- 새 프로젝트가 기획 방향을 빠르게 정한다.
- 새 AI·기획자·아트 담당자가 같은 판단 기준으로 시작한다.
- 다른 작품을 표면적으로 복제하지 않고 문제 해결 원리를 추출한다.
- 아트와 연출을 취향 표현이 아니라 사용자 경험·정보 위계·제작 제약으로 설계한다.
- 프로젝트에서 검증된 방법과 실패 사례를 공용 학습 데이터로 누적한다.
- 인수인계 시 기획서만 읽어도 현재 방향·우선순위·참조 파일을 찾을 수 있게 한다.

## 2. 문서 구조

| 분야 | 문서 | 핵심 질문 |
|---|---|---|
| 기획 체계 | `PLANNING_SYSTEM.md` | 무엇을 어떤 상태와 문서 계층으로 설계할 것인가? |
| 콘텐츠·서사 | `NARRATIVE_CONTENT_METHOD.md` | 장면·대사·선택·관계가 어떻게 의미를 남기는가? |
| 아트디자인 | `ART_DIRECTION_METHOD.md` | 시각 정체성을 어떻게 정의하고 일관되게 생산하는가? |
| 연출·프레젠테이션 | `PRESENTATION_DIRECTION_METHOD.md` | 정보와 감정을 어떤 화면·표정·컷인·음향으로 전달하는가? |
| 정보수집 | `RESEARCH_AND_EVIDENCE_METHOD.md` | 어떤 질문에 어떤 출처와 사례가 필요한가? |
| 스킬·도구 | `PLANNING_SKILL_LIBRARY.md` | 어떤 능력·도구를 언제 쓰고 무엇을 산출해야 하는가? |
| 사례 | `CASE_LIBRARY.md` | 실제 문제에서 무엇을 관찰·적용·제외·검증했는가? |
| 인수인계 | `HANDOFF_CONTEXT_METHOD.md` | 다른 담당자가 기획서만으로 방향과 다음 작업을 이해하는가? |

기존 `docs/CONTENT_DESIGN_METHOD.md`는 핵심 재미, 첫 10분, 콘텐츠 정체성 카드, PoC를 담당한다. 이 폴더는 그 기준을 아트·연출·서사·연구·인수인계까지 확장한다.

## 3. 권장 읽기 순서

### 새 프로젝트 기획

```text
PLANNING_SYSTEM.md
→ CONTENT_DESIGN_METHOD.md
→ 필요한 분야별 방법 1~2개
→ RESEARCH_AND_EVIDENCE_METHOD.md
→ templates/planning/의 해당 양식
```

### 아트·연출 작업

```text
ART_DIRECTION_METHOD.md
→ PRESENTATION_DIRECTION_METHOD.md
→ PLANNING_SKILL_LIBRARY.md의 관련 스킬
→ CASE_LIBRARY.md의 유사 사례
→ 프로젝트 실제 에셋·화면·제약
```

### 서사·캐릭터 작업

```text
NARRATIVE_CONTENT_METHOD.md
→ CONTENT_DESIGN_METHOD.md
→ CASE_LIBRARY.md의 대사·관계 사례
→ 프로젝트 캐릭터·사건 데이터
```

### 인수인계

```text
HANDOFF_CONTEXT_METHOD.md
→ 프로젝트의 현재 상태 문서
→ 프로젝트 방향 기획서
→ 분야별 기획서
→ 실제 파일과 검증 기록
```

## 4. 공용 데이터와 프로젝트 데이터의 경계

### Base에 기록

- 반복 가능한 질문 순서
- 의사결정 프레임
- 문서 구조와 상태 표기
- 아트·연출·서사 품질 기준
- 정보수집과 출처 평가 방법
- 스킬·도구 선택 기준
- 익명화·일반화한 사례 카드
- 실패 패턴과 재발 방지 체크

### 프로젝트에 기록

- 프로젝트 이름과 세계관
- 캐릭터명·대사·사건 규칙
- 특정 수치와 밸런스
- 엔진·코드·데이터 경로
- 실제 에셋 파일명과 라이선스
- 구현 완료 상태와 테스트 결과
- 프로젝트 전용 벤치마킹 결론

## 5. 지식 항목 상태

| 상태 | 의미 |
|---|---|
| 원칙 | 여러 작업에 반복 적용되고 검증된 공용 기준 |
| 패턴 | 특정 문제군에서 유용하지만 조건을 확인해야 하는 방법 |
| 사례 | 실제 프로젝트·벤치마킹에서 관찰한 적용 기록 |
| 가설 | 아직 충분히 검증되지 않은 제안 |
| 폐기 | 적용 결과가 나쁘거나 공용성이 없어 사용하지 않는 방법 |

새 사례를 바로 원칙으로 승격하지 않는다. 최소 두 번 이상의 재사용 또는 다른 프로젝트에서도 동일한 가치가 확인되어야 한다.

## 6. 사례 기록 규칙

사례는 다음 항목을 포함한다.

```text
문제
→ 맥락과 제약
→ 관찰한 근거
→ 적용한 변경
→ 제외한 요소와 이유
→ 검증 결과
→ 재사용 가능한 원칙
→ 적용하면 안 되는 조건
```

작품 이름만 나열하거나 “좋아 보였다”는 감상은 학습 데이터가 아니다.

## 7. 프로젝트 승격 절차

1. 프로젝트에서 문제와 해결 결과를 기록한다.
2. 프로젝트 고유 이름·수치·경로를 제거한다.
3. 관찰 사실과 해석·결정을 분리한다.
4. 적용 조건과 실패 조건을 적는다.
5. Base의 기존 원칙·사례와 중복을 확인한다.
6. 새 사례 또는 패턴으로 추가한다.
7. 반복 검증되면 공용 원칙과 템플릿을 갱신한다.
8. Base 커밋 SHA를 프로젝트의 Base 버전 문서에 기록한다.

## 8. 품질 기준

- 문서만 읽어도 목적·입력·산출물·검증 방법을 알 수 있다.
- “예쁘게”, “몰입감 있게”, “좋은 대사” 같은 모호한 표현을 관찰 가능한 기준으로 바꾼다.
- 사례의 외형을 복제하지 않고 사용자 가치와 구조를 추출한다.
- 정보수집 결과와 프로젝트 결정이 분리돼 있다.
- 스킬·도구는 이름보다 수행할 역할과 검증 결과로 기록한다.
- 템플릿이 문서를 늘리는 목적이 아니라 누락과 오해를 줄이는 데 쓰인다.
- 불필요한 기본 읽기를 늘리지 않고 작업 조건에 따라 필요한 문서만 선택한다.
