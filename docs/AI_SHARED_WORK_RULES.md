# AI Shared Work Rules

이 문서는 특정 엔진이나 장르에 종속되지 않는 공용 AI 작업 규칙이다. 새 프로젝트는 필요한 내용을 로컬 사본으로 동기화하고, 프로젝트의 `AGENTS.md`와 전용 규칙 문서에서 더 구체적인 내용을 덧붙인다.

## 1. 규칙 우선순위

충돌 시 다음 순서로 판단한다.

1. 사용자의 최신 지시
2. 프로젝트 루트의 `AGENTS.md` 또는 `CODEX_RULES.md`
3. 현재 GitHub Issue와 승인된 설계 문서
4. 현재 작업의 Goal 또는 PR 설명
5. 프로젝트에 동기화된 Base 공용 문서의 로컬 사본
6. Base 원격 저장소의 원본
7. 과거 대화와 추측

Base 원격은 일상 작업의 직접 의존성이 아니다. 프로젝트는 `docs/BASE_RULES_VERSION.md`에 기준 커밋을 기록하고, 로컬 사본을 먼저 읽는다. 정상 동작 중인 사용자 변경사항은 되돌리지 않는다.

## 2. 역할 분리

- 기획/조정 AI: 사용자 의도, 플레이어 가치, 벤치마킹, Issue, Goal, 대시보드, 검수 기준을 정리한다.
- 구현 AI: 승인된 Issue/Goal을 기준으로 실제 파일을 읽고 수정하고 검증한다.
- 사용자: 방향을 결정하고, 실제 실행 결과와 플레이 감각을 확인하고 승인한다.

한 작업에서 역할을 섞지 않는다. 구현 AI가 기획을 임의 확장하지 않고, 기획/문서 담당은 구현 완료를 추정하지 않는다.

## 3. 공용 문서와 프로젝트 전용 문서 구분

새 프로젝트는 `공용 규칙 패키지`와 `프로젝트 전용 기억`을 분리한다.

### 공용 규칙 패키지

- `AGENTS.md`: 공용 최상위 원칙
- `docs/AI_SHARED_WORK_RULES.md`: 공용 AI 작업 원칙
- `docs/AI_WORKFLOW_RULES.md`: 작업 순서와 Goal 작성 기준
- `docs/MVP_WORKFLOW_CHECKLIST.md`: MVP 시작과 종료 체크리스트
- `docs/BENCHMARKING_REFERENCE_GUIDE.md`: 벤치마킹 기록과 적용 기준
- `docs/DOCUMENTATION_MAP.md`: 공용/전용 문서의 책임 경계
- `docs/CUSTOM_INSTRUCTIONS_GUIDE.md`: ChatGPT/Codex 맞춤형 지침 작성 기준
- `templates/`: 프로젝트 문서 템플릿

### 프로젝트 전용 기억

- `AGENTS.md`: 프로젝트 최상위 규칙
- `docs/BASE_RULES_VERSION.md`: Base 동기화 기준
- `docs/CODEX_SHARED_WORK_RULES.md` 또는 `docs/CODEX_PROJECT_RULES.md`: 프로젝트용 구현 규칙
- `README.md`: 소개, 실행 방법, 현재 상태
- `PROJECT_BRIEF.md`: 한 줄 설명, 장르, 플레이어 역할, 핵심 경험, 차별점
- `DESIGN_INTENT.md`: 기획 의도, 핵심 루프, 플레이어 감정, 선택지 설계 원칙
- `MVP_ROADMAP.md`: MVP 목록, Issue 번호, 상태, 완료 기준, 다음 작업
- `TEST_CHECKLIST.md`: 실행 검수 순서, MVP별 체크리스트, 오류 기록 방식
- 프로젝트별 설정, 데이터, 코드, 씬, 테스트 자산

## 4. 전용 프로젝트 설계 절차

새 프로젝트를 시작하면 다음 순서로 설계한다.

1. 프로젝트 제목, 장르, 핵심 콘셉트를 정한다.
2. 벤치마킹할 장르와 인접 사례를 정한다.
3. 핵심 플레이어 경험을 한 문장으로 정의한다.
4. 반복 루프를 정의한다.
5. 만들지 않을 것을 먼저 정해 범위 폭발을 막는다.
6. `PROJECT_BRIEF.md`를 작성한다.
7. `DESIGN_INTENT.md`를 작성한다.
8. `MVP_ROADMAP.md`를 작성한다.
9. `TEST_CHECKLIST.md`를 작성한다.
10. 프로젝트용 `AGENTS.md`와 Codex 규칙 문서를 작성한다.
11. HTML 대시보드로 전체 흐름을 시각화한다.
12. 첫 GitHub Issue를 만들고 Goal을 작성한다.

## 5. Definition of Ready

구현을 시작하기 전에 다음 조건을 만족해야 한다.

- [ ] 한 문장 목표와 해결하려는 사용자 문제가 있다.
- [ ] 이전 작업과 다음 작업의 연결이 설명된다.
- [ ] 구현 범위와 제외 범위가 분리되어 있다.
- [ ] 완료 기준이 관찰 가능한 행동 또는 검증 가능한 상태로 작성되어 있다.
- [ ] 예상 영향 파일과 저장/호환성 위험이 있다.
- [ ] 실행·검증 방법과 사용자가 확인할 순서가 있다.
- [ ] 필요한 경우 벤치마킹의 반영/제외 결론이 있다.

준비되지 않은 요구사항은 구현자에게 넘기지 말고 질문, Issue 보완, 또는 작은 탐색 작업으로 분리한다.

## 6. 벤치마킹

- 새 시스템이나 방향 결정은 관련 사례 3~5개를 먼저 조사한다.
- 장르 핵심 구조, 시장성 판단, 큰 방향 전환처럼 표본이 중요한 경우에만 10개 이상으로 확장한다.
- 조사는 시간 제한하고, 사례 수보다 적용 결론을 우선한다.
- 공식 문서, 개발자 발언, 학술/업계 자료, 실제 사용자 반응을 구분한다.
- 출처와 사실을 남기되, 다른 작품의 기능을 그대로 복사하지 않는다.
- 구현이 이미 Issue에서 확정된 경우에는 벤치마킹을 새로 강제하지 않는다.

## 7. Issue, Goal, PR

- Issue는 장기 기준서다: 문제, 목표, 흐름, 범위, 제외, 완료 기준, 위험, 검수 항목을 둔다.
- Goal은 구현 지시서다: Issue와 전용 규칙을 참조하고, 구현 범위와 완료 기준을 짧게 둔다.
- PR은 변경 이유, 영향, 검증 결과, 남은 위험을 기록하는 변경 이력이다.
- 한 Issue는 한 가치 흐름을 설명하고, 관련 없는 개선은 별도 Issue로 분리한다.
- 완료 기준은 기능명보다 `조건 → 행동 → 관찰 결과` 형태로 쓴다.

## 8. 구현 전후 규칙

구현 전에는 현재 브랜치와 작업 상태, 관련 파일, 호출/참조 위치, 저장 포맷을 확인한다.

구현 후에는 다음을 확인한다.

- [ ] 실제 변경 파일과 diff가 Issue 범위 안이다.
- [ ] 문법, 데이터 형식, 핵심 실행 경로를 검증했다.
- [ ] 저장/불러오기와 기존 흐름에 미치는 영향을 확인했다.
- [ ] 테스트하지 못한 항목을 숨기지 않았다.
- [ ] 사용자가 실행할 검수 순서를 적었다.
- [ ] 남은 위험을 수동 확인이 필요한 것과 수정 가능한 것으로 구분했다.

재현 가능한 작은 위험은 보고로 넘기지 말고 현재 작업에서 먼저 줄인다. 범위를 넘는 개선은 별도 제안으로 남긴다.

## 9. 파일 변경 의도와 수명주기 기록

파일을 생성, 수정, 삭제, 이동, 이름 변경할 때는 다른 AI와 다음 작업자가 인지할 수 있게 변경 의도를 남긴다.

특히 삭제, 이동, 이름 변경은 충돌 위험이 크므로 다음을 확인한다.

- 참조하는 문서, 코드, Issue, 템플릿이 있는가?
- 대체 경로가 필요한가?
- 프로젝트 로컬 사본 또는 Base 동기화가 필요한가?
- 다음 작업자가 어떤 파일을 더 이상 보면 안 되는가?

권장 기록 형식:

```md
## 파일 변경 의도

- 파일:
- 변경 유형: 생성 / 수정 / 삭제 / 이동 / 이름 변경
- 변경 이유:
- 연결되는 문서 또는 코드:
- 다른 AI 또는 다음 작업자가 알아야 할 영향:
- 후속 동기화 필요 여부:
```

## 10. Base 승격 운영 방식

새 규칙은 처음부터 Base에 자동 반영하지 않는다.

1. 먼저 프로젝트 전용 규칙 또는 로컬 확장으로 적용한다.
2. 실제 작업에서 반복 사용되거나 여러 프로젝트에 재사용할 가치가 확인되면 Base 승격 후보로 분류한다.
3. 구현 작업자는 Base를 자동 수정하거나 자동 push하지 않는다.
4. 작업 완료 보고에서 `Base 승격 후보`를 별도 항목으로 알린다.
5. 사용자가 먼저 프로젝트 저장소의 변경을 확인한다.
6. 사용자가 승인하면 Base 저장소에 공용 규칙을 별도 커밋으로 반영한다.
7. Base 커밋 SHA가 확정되면 프로젝트의 `docs/BASE_RULES_VERSION.md`를 후속 커밋으로 갱신한다.

### Base 승격 후보 보고 형식

공용화할 만한 내용이 있으면 작업 완료 보고에 아래 항목을 포함한다.

```md
## Base 승격 후보

- 대상:
- 현재 프로젝트에서 확인된 반복 사용 근거:
- 공용 규칙으로 일반화할 내용:
- 프로젝트 전용으로 남겨야 할 내용:
- Base 반영 필요성: 필요 / 불필요 / 추가 검증 필요
- 권장 Base 커밋명:
- Base 반영 후 프로젝트 동기화 필요 여부:
```

공용화할 내용이 없으면 아래처럼 명시한다.

```md
## Base 승격 후보

- 없음: 이번 변경은 프로젝트 전용 규칙 또는 단발성 구현이다.
```

## 11. 회고와 재사용

작업이 끝나면 반복된 혼동 또는 실패, 다음 작업에서 보완할 규칙, 공용 규칙으로 승격할 내용을 기록한다. 공용 규칙에는 안정적인 원칙만 넣고, 엔진·폴더·세계관·일회성 함수명은 프로젝트 전용 문서에 둔다.
