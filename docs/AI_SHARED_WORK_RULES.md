# AI Shared Work Rules

이 문서는 특정 엔진이나 장르에 종속되지 않는 공용 AI 작업 규칙이다.

Base는 프로젝트에서 얻은 방법·실패·검증 결과를 일반화해 누적하는 **[학습형] [공용] 데이터 원본**이다. 새 프로젝트는 필요한 내용을 로컬 사본으로 동기화하고, 프로젝트 `AGENTS.md`와 전용 기획서·스킬에서 세계관, 수치, 엔진, 경로와 실제 검증 방식에 맞게 분화한다.

## 0. 최상위 지속성 규칙

모든 프로젝트는 새 채팅, 새 AI, 새 작업자가 과거 대화나 기존 작업자의 설명 없이도 저장소만으로 현재 작업을 재개할 수 있는 상태를 유지한다.

- 기획서는 핵심 플레이어 경험, 현재 방향, 범위, 불변 조건과 금지 방향을 책임진다.
- Roadmap은 현재 단계, 우선순위, 선행 조건, 다음 작업, 종료 기준과 검증을 책임진다.
- Base skill과 프로젝트 skill extension은 반복 작업의 입력, 절차, 산출물, 검증과 실패 기준을 책임진다.
- Active Context·Handoff는 현재 상태와 읽기 순서를 연결한다.
- Documentation Map은 질문별 현행 책임 원본을 연결한다.
- 방향, 수치, 용어, 범위, 구현 상태 또는 작업 절차가 바뀌면 같은 작업 안에서 영향받는 기획서·Roadmap·스킬·문서 지도·Active Context를 갱신한다.

기획서만 읽어도 작업 방향과 플레이어 가치를 이해할 수 있어야 하고, 세부 구현은 기획서가 가리키는 실제 파일·데이터·테스트 원본에서 확인할 수 있어야 한다.

## 1. 규칙 우선순위

충돌 시 다음 순서로 판단한다.

1. 사용자의 최신 지시
2. 프로젝트 루트 `AGENTS.md`와 보안·엔진·데이터 규칙
3. 프로젝트 Active Context·Handoff
4. 승인된 기획서와 현재 Issue·Goal·Plan
5. 프로젝트에 동기화된 Base 공용 문서의 로컬 사본
6. Base 원격 저장소의 원본
7. 과거 대화와 추측

Base 원격은 일상 작업의 직접 의존성이 아니다. 프로젝트에 `docs/BASE_RULES_VERSION.md`가 있으면 기준 커밋을 기록하고 로컬 사본을 먼저 읽는다. 정상 동작 중인 사용자 변경사항은 되돌리지 않는다.

## 2. 역할 분리

- 기획·조정 AI: 사용자 의도, 플레이어 가치, 벤치마킹, 기획서, Issue·Goal, 검수 기준을 정리한다.
- 구현 AI: 승인된 Issue·Goal을 기준으로 실제 파일을 읽고 수정하고 검증한다.
- 사용자: 방향을 결정하고 실제 실행 결과와 플레이 감각을 확인하고 승인한다.

구현 AI가 기획을 임의 확장하지 않고, 기획·문서 담당은 구현 완료를 추정하지 않는다.

### 2.1 프로젝트 분야별 책임

| 팀 | 책임 | 다른 팀과의 경계 |
|---|---|---|
| 게임 기획 | 규칙·시나리오·UI/UX 행동·밸런스 | 구현 방법, 최종 시각 자산과 사운드 파일을 소유하지 않는다. |
| 프로그래밍 | 엔진에서 실제 동작하는 코드·데이터·AI·물리·네트워크·MVP·로드맵 | 승인된 규칙을 임의로 바꾸지 않는다. |
| 아트 | 2D·3D, 캐릭터·배경·UI 시각, 애니메이션·VFX | 상호작용 규칙과 판정 수치를 소유하지 않는다. |
| 사운드 | BGM·SFX·음성·믹싱과 분위기 | 게임 결과와 판정 타이밍을 단독 변경하지 않는다. |
| QA·PM | 검증·버그·릴리스 게이트, 일정·위험·예산 상태와 소통 | 실패를 숨기거나 미검증 상태를 완료로 바꾸지 않는다. |

프로젝트는 위 다섯 분야의 활성 본책을 유지한다. 한 작업이 여러 분야에 영향을 주면 주 책임 하나와 영향 분야를 모두 기록하고 관련 본책을 같은 작업에서 갱신한다.

## 3. 작업 시작 전 공용·전용 확인

현재 작업에 적용되는 두 집합을 함께 확인한다.

### Base 공용 학습 데이터

- 공용 작업 규칙과 workflow
- 관련 method·research
- 실행 skill과 검수 matrix
- 사용할 template
- 유사 성공·실패·미검증 case

### 프로젝트 전용 데이터

- 프로젝트 `AGENTS.md`, Base 버전과 로컬 사본
- Documentation Map, Active Context·Handoff
- 프로젝트 방향·전체·분야별 기획서
- Roadmap, 현재 Issue·Goal·Plan
- 프로젝트 skill extension
- 실제 수정 대상, 호출·참조, 데이터·저장·호환성 영향

`모두 확인`은 모든 파일을 무작정 읽는다는 뜻이 아니다. Documentation Map과 참조 관계로 현재 작업에 적용되는 공용·전용 현행 책임 원본과 영향 파일의 범위를 정하고 그 범위 안을 빠짐없이 확인한다.

## 4. 공용 데이터와 프로젝트 전용 데이터

### Base — [학습형] [공용]

- `AGENTS.md`: 공용 최상위 원칙
- `docs/AI_SHARED_WORK_RULES.md`: 공용 AI 작업 원칙
- `docs/AI_WORKFLOW_RULES.md`: 작업 순서와 학습 환류
- `docs/knowledge/methods/`: 반복 가능한 판단·기획법
- `docs/knowledge/research/`: 조사·표준·근거
- `docs/knowledge/skills/`: 분야별 역량·검수 계약
- `skills/`: 직접 실행 가능한 작업 절차
- `templates/`: 프로젝트에 분화할 출력 구조
- `docs/knowledge/cases/`: 일반화된 성공·실패·미검증 사례

### 프로젝트 — [전용] [분화·적용·검증]

- 프로젝트 루트 `AGENTS.md`
- `docs/BASE_RULES_VERSION.md`
- README, Documentation Map, Active Context·Handoff
- 프로젝트 방향서, 실제 GDD, 분야별 책임 기획서
- Roadmap, Issue·Goal·Plan, 테스트 기록
- 프로젝트 설정, 데이터, 코드, 씬, 자산과 파일 경로
- 실제 모델·비용·승인 프롬프트
- Base skill의 프로젝트 extension
- 공용화 전 관찰·가설·실험 결과

프로젝트의 구체 해답을 Base에 복사하지 않는다. 반복 가능한 문제 해결 원리, 적용 조건, 실패 조건과 검증 방법을 일반화하고 실제 사례는 case로 남긴다.

## 5. 전용 프로젝트 설계 절차

1. 프로젝트 제목, 장르, 핵심 콘셉트를 정한다.
2. 핵심 플레이어 경험과 차별점을 한 문장으로 정의한다.
3. 반복 루프와 핵심 선택을 정의한다.
4. 만들지 않을 것을 먼저 정해 범위 폭발을 막는다.
5. 방향서, 전체·분야별 기획서, Roadmap, 테스트 기준을 작성한다.
6. 프로젝트 `AGENTS.md`, Documentation Map, Active Context·Handoff를 작성한다.
7. 필요한 Base skill과 프로젝트 skill extension을 연결한다.
8. 첫 GitHub Issue와 Goal을 Roadmap 단계에 연결한다.
9. 새 작업자가 콜드 스타트 질문에 답할 수 있는지 확인한다.

## 6. Definition of Ready

구현을 시작하기 전에 다음 조건을 만족해야 한다.

- [ ] 한 문장 목표와 해결하려는 사용자 문제가 있다.
- [ ] 핵심 플레이어 경험과 프로젝트 방향에 어떻게 기여하는지 설명된다.
- [ ] 이전 작업, Roadmap 단계와 다음 작업의 연결이 설명된다.
- [ ] 구현 범위와 제외 범위가 분리되어 있다.
- [ ] 완료 기준이 관찰 가능한 행동 또는 검증 가능한 상태로 작성되어 있다.
- [ ] 예상 영향 파일과 저장·호환성 위험이 있다.
- [ ] 실행·검증 방법과 사용자가 확인할 순서가 있다.
- [ ] 관련 Base method·skill과 프로젝트 extension을 확인했다.
- [ ] 필요한 경우 벤치마킹의 반영·제외 결론이 있다.

준비되지 않은 요구사항은 구현자에게 넘기지 말고 질문, Issue 보완 또는 작은 탐색 작업으로 분리한다.

## 7. 벤치마킹

- 새 시스템이나 방향 결정은 관련 사례 3~5개를 먼저 조사한다.
- 장르 핵심 구조, 시장성 판단, 큰 방향 전환처럼 표본이 중요한 경우에만 10개 이상으로 확장한다.
- 조사는 시간 제한하고 사례 수보다 적용·제외 결론을 우선한다.
- 공식 문서, 개발자 발언, 학술·업계 자료, 실제 사용자 반응을 구분한다.
- 출처와 사실을 남기되 다른 작품의 기능·문구·아트를 그대로 복사하지 않는다.
- 구현이 이미 Issue에서 확정된 경우에는 벤치마킹을 새로 강제하지 않는다.
- 재사용 가능한 관찰은 research 또는 case로 기록한다.

## 8. Issue, Goal, Plan, PR

- Issue는 장기 기준서다: 문제, 목표, 흐름, 범위, 제외, 완료 기준, 위험, 검수 항목을 둔다.
- Goal은 구현 지시서다: Issue와 전용 규칙을 참조하고 구현 범위와 완료 기준을 짧게 둔다.
- Plan은 실제 파일 조사 결과를 반영한 작업 순서다.
- PR은 변경 이유, 영향, 검증 결과, 남은 위험과 문서 최신화를 기록한다.
- 한 Issue는 한 가치 흐름을 설명하고 관련 없는 개선은 별도 Issue로 분리한다.
- 완료 기준은 기능명보다 `조건 → 행동 → 관찰 결과` 형태로 쓴다.

## 9. 구현 전후 규칙

구현 전에는 현재 브랜치와 작업 상태, 관련 공용·전용 책임 문서, 실제 파일, 호출·참조 위치, 저장 포맷을 확인한다.

구현 후에는 다음을 확인한다.

- [ ] 실제 변경 파일과 diff가 Issue 범위 안이다.
- [ ] 문법, 데이터 형식, 핵심 실행 경로를 검증했다.
- [ ] 저장·불러오기와 기존 흐름에 미치는 영향을 확인했다.
- [ ] 테스트하지 못한 항목을 숨기지 않았다.
- [ ] 사용자가 실행할 검수 순서를 적었다.
- [ ] 남은 위험을 수동 확인이 필요한 것과 수정 가능한 것으로 구분했다.
- [ ] 기획서·Roadmap·스킬·Active Context의 영향이 반영됐다.
- [ ] 영향받는 5개 분야 본책의 방향·현재 상태·최신 자료·검증 결과가 반영됐다.
- [ ] 시각자료가 바뀌었다면 최신 승인 이미지 인덱스와 아트 본책이 함께 갱신됐다.

재현 가능한 작은 위험은 보고로 넘기지 말고 현재 작업에서 먼저 줄인다. 범위를 넘는 개선은 별도 제안으로 남긴다.

## 10. 기준 문서와 파일 수명주기

- 한 질문에는 현행 책임 문서 하나만 유지한다.
- 확정된 변경은 기존 기준 파일을 직접 갱신한다.
- `*_v2`, `*_final`, `*_latest`, 날짜별 활성 복제본을 만들지 않는다.
- 이전 내용은 기본적으로 Git 이력으로 보존한다.
- 별도 백업은 사용자의 명시적 요구, 법무·감사, 외부 링크 보존 사유가 있을 때만 만든다.
- archive·`[백업]`, hold·`[보류]`는 재개 결정 전까지 활성 기획·구현·테스트의 기본 읽기 대상에서 제외한다.
- 새 기준 문서에 내용이 통합된 뒤 남은 이전 버전·중복 파일은 참조 경로를 확인하고 제거한다.

파일을 생성, 수정, 삭제, 이동, 이름 변경할 때 다음을 확인한다.

- 참조하는 문서, 코드, Issue, 템플릿이 있는가?
- 대체 경로가 필요한가?
- Documentation Map·README·Roadmap·skill extension을 갱신해야 하는가?
- 프로젝트 로컬 사본 또는 Base 동기화가 필요한가?
- 다음 작업자가 어떤 파일을 더 이상 보면 안 되는가?

## 11. 작업 종료·인수인계와 Base 학습 환류

1. 프로젝트의 확정 결정, 수치, 구현·테스트 상태를 현행 기획서와 데이터·테스트 문서에 반영한다.
2. Roadmap의 현재 단계, 우선순위, 선행 조건, 다음 작업과 종료 기준을 갱신한다.
3. 프로젝트 skill extension과 실제 파일·데이터·검증의 일치를 확인한다.
4. Active Context·Handoff와 Documentation Map을 최신화한다.
5. 결과를 프로젝트 전용 데이터와 Base 공용 학습 데이터로 분리한다.
6. 검증된 공용 판단법·절차·출력 구조는 기존 method·skill·template에 반영한다.
7. 문제, 대안, 결정, 실제 결과, 실패, 수정 과정, 미검증 항목은 case로 기록하거나 상태를 갱신한다.
8. 한 번 성공했거나 검증이 부족한 내용은 `관찰`·`가설` case로 남기고 확정 rule·skill로 표시하지 않는다.
9. Base 변경 뒤 Changelog와 프로젝트 `BASE_RULES_VERSION.md`·로컬 사본의 동기화를 확인한다.
10. 새 작업자가 10분 안에 핵심 방향, 현재 상태, 다음 작업, 금지 범위, 책임 문서와 검증 경로를 찾을 수 있는지 확인한다.

공용화할 내용이 없으면 `공용 학습 데이터 없음 — 프로젝트 전용 또는 단발성 작업`으로 보고한다.

## 12. 서사형 안내자와 튜토리얼 정보 경계

- 서사형 튜토리얼 안내자는 담당 시스템과 자연스럽게 연결되는 세계관 내 역할을 가진다.
- 최초 상세 안내와 재방문용 짧은 안내를 분리하고 확인 상태를 저장해 같은 전체 연출을 반복하지 않는다.
- 음향 신호는 음소거 상태에서도 이해할 수 있는 동등한 시각 또는 문구 신호와 함께 제공한다.
