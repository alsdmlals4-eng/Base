# 공용 콘텐츠 기획 방법

## 목적

이 문서는 게임과 인터랙티브 콘텐츠를 설계할 때 기능 목록보다 먼저 **의도 → 플레이 경험 → 규칙 → 흐름 → 검증**을 정리하기 위한 공용 기준이다.

특정 장르, 엔진, 세계관, 수치, 캐릭터 이름은 프로젝트 저장소에서 정의한다. Base에는 여러 프로젝트에서 재사용할 수 있는 판단 순서와 템플릿만 둔다.

## 1. 기획의 기본 순서

기획은 다음 순서로 작성한다.

```text
의도
→ 플레이어가 하게 될 행동
→ 유도할 판단과 감정
→ 행동을 가능하게 하는 규칙
→ 화면과 콘텐츠의 진행 흐름
→ 성공·실패·예외 결과
→ PoC와 플레이테스트 검증
```

기능 이름만 적지 않는다. 각 기능은 최소한 다음 질문에 답해야 한다.

1. 왜 필요한가?
2. 없으면 어떤 플레이 문제가 생기는가?
3. 플레이어에게 어떤 행동과 고민을 유도하는가?
4. 어떤 조건에서 어떻게 작동하는가?
5. 결과를 플레이어가 어떻게 알아차리는가?
6. 기존 콘텐츠와 무엇이 다른가?
7. 예상 위험과 축소안은 무엇인가?

## 2. 핵심 재미와 뾰족한 경험

핵심 재미는 기능의 개수가 아니라 플레이어가 반복하고 싶어 하는 **행동과 감정의 결합**이다.

좋은 핵심 재미 문장은 다음을 포함한다.

- 플레이어 행동
- 판단 대상
- 즉시 반응
- 성공했을 때의 감정

예시 형식:

> 플레이어는 ___을 읽거나 조작해 ___을 판단하고, 그 판단이 즉시 ___으로 나타날 때 ___을 느낀다.

모든 주요 콘텐츠는 다음 중 하나를 강화해야 한다.

- 핵심 행동
- 핵심 판단
- 핵심 감정
- 반복 피로 완화
- 캐릭터·세계관 애착

어느 역할도 하지 않는 콘텐츠는 추가보다 삭제·통합을 먼저 검토한다.

## 3. Dopamine Driven Development 적용 원칙

간단하고 빠른 만족을 중시하되, 보상 팝업이나 수치 폭발을 무조건 늘리는 방식으로 해석하지 않는다.

### 첫 10분 계약

첫 10분 안에 플레이어가 다음을 경험해야 한다.

1. 게임의 매력적인 상황 또는 세계관을 본다.
2. 핵심 행동을 직접 수행한다.
3. 선택이나 조작에 대한 즉시 반응을 받는다.
4. 핵심 시스템이 왜 재미있는지 한 번 체감한다.
5. 다음에 무엇을 해보고 싶은지 알게 된다.

### 적합한 즉시 보상

- 예상이 맞았다는 확인
- 서로 떨어진 정보가 연결되는 순간
- 세계가 선택에 즉시 반응하는 연출
- 새로운 행동 가능성의 해금
- 캐릭터의 짧은 인정이나 반응
- 기록, 도감, 보고서가 플레이 결과로 채워지는 변화

### 피해야 할 방식

- 의미 없는 재화·점수 팝업 반복
- 장르와 맞지 않는 과도한 성공 연출
- 설명을 읽기 전에 여러 시스템을 동시에 개방
- 핵심 판단 대신 자동 정답이나 수치 성장으로 해결
- 첫 10분에 장기 경제·세부 능력치·모든 메뉴를 설명

## 4. 콘텐츠 정체성 카드

사건, 스테이지, 캐릭터 에피소드, 퀘스트 등 주요 콘텐츠는 제작 전에 정체성 카드를 작성한다.

### 필수 항목

- 콘텐츠명
- 한 줄 약속
- 플레이어가 반복할 핵심 동사
- 주요 질문 또는 판단
- 유도할 감정
- 정보 또는 자원의 형태
- 성공 경험
- 실패 시 남는 것
- 다른 콘텐츠와의 차별점
- 시작 조건
- 종료 또는 합류 조건
- 최소 구현 단위
- 저장·UI·데이터 영향
- 제외할 요소

### 사건·추리 콘텐츠 추가 항목

- 발생 배경
- 관측 가능한 전조
- 규칙을 입증하는 핵심 근거
- 탐색 방향을 주는 힌트
- 잘못된 대응에서 얻는 위험 기록
- 최종 대응과 근거의 연결
- 완료 후 남는 기록

정체성 카드에 없는 사실을 대본이나 구현 단계에서 임의로 추가하지 않는다.

## 5. 단서·힌트·실패 정보의 분리

정보형 콘텐츠는 역할을 구분한다.

- **근거:** 규칙이나 판단을 입증하는 확보 정보
- **힌트:** 아직 확인하지 않은 장소·대상·비교 방향
- **실패 기록:** 잘못된 행동과 관측 결과를 다음 판단에 쓸 수 있게 남긴 정보
- **분위기 기록:** 세계관과 몰입을 보강하지만 핵심 판정에는 사용하지 않는 정보

분위기 기록을 근거 수집률에 포함하면 플레이어가 중요도를 오해할 수 있다. 힌트가 정답 행동을 직접 지시하면 추리의 공정성이 무너진다.

실패 결과는 가능한 한 다음 순서로 작성한다.

```text
선택한 행동
→ 관측된 전조 또는 변화
→ 발생한 결과
→ 새로 확인된 위험 조건
```

정답을 직접 알려주지 않되 같은 실수를 반복하지 않을 근거는 남긴다.

## 6. 시스템 규칙 작성 형식

각 규칙은 다음 항목으로 분해한다.

- 상황
- 적용 범위
- 필요한 정보
- 플레이어 입력
- 내부 처리
- 판정 기준
- 플레이어에게 보이는 출력
- 실패 결과
- 예외 처리

기본 구조:

```text
조건 → 입력 → 처리 → 판단 → 결과 → 기록
```

반드시 확인할 예외:

- 자원·조건 부족
- 중복 입력과 빠른 연타
- 이미 변경된 상태에 대한 입력
- 저장·불러오기 중간 상태
- 재시도와 보상 중복
- 취소·이탈·중단 후 복귀
- UI 표시와 실제 상태의 불일치

## 7. 콘텐츠 흐름 설계

한 장면에서는 가능한 한 한 가지 질문만 전면에 둔다.

권장 흐름:

```text
상황 제시
→ 의미가 다른 선택 또는 행동
→ 짧은 결과
→ 상태 변화와 기록
→ 새로운 질문
```

화면에 표시할 정보는 중요도보다 **현재 판단에 필요한 정도**로 정한다.

- 자주 쓰는 기능은 1~2단계 안에 둔다.
- 상세 수치와 이전 기록은 필요할 때 연다.
- 같은 기능은 같은 위치와 표현을 사용한다.
- 색과 아이콘은 한 의미로 통일한다.
- 선택 후 발생할 일을 문구로 예측할 수 있게 한다.

## 8. PoC와 기획 재조정

PoC는 전체 분량이나 모든 시스템을 만드는 단계가 아니다. 가장 위험한 가정을 최소 단위로 검증한다.

### PoC 질문

- 플레이어가 핵심 행동을 이해하는가?
- 그 행동을 다시 하고 싶어 하는가?
- 성공 이유와 실패 이유를 설명할 수 있는가?
- 세계관과 시스템의 이유가 연결되는가?
- 주변 시스템 없이도 핵심 재미가 성립하는가?
- 첫 10분에 불필요한 설명이나 메뉴가 있는가?

### 검증 단위

- 핵심 장면 1개
- 선택 2~3개
- 성공 결과 1개
- 실패 결과 1개
- 기록 또는 보상 1개

### 재조정 원칙

PoC 결과가 약할 때 콘텐츠 양을 먼저 늘리지 않는다.

1. 핵심 동사가 불분명한지 확인한다.
2. 선택 간 차이가 실제 결과로 보이는지 확인한다.
3. 즉시 반응이 늦거나 약한지 확인한다.
4. 설명과 시스템 수를 줄인다.
5. 그래도 약하면 핵심 규칙을 다시 설계한다.

## 9. 콘텐츠 추가 우선순위

새 콘텐츠는 다음 순서로 심사한다.

1. 핵심 경험의 공정성 또는 이해를 개선하는가?
2. 첫 10분과 핵심 루프를 강화하는가?
3. 기존 콘텐츠의 반복 피로를 줄이는가?
4. 캐릭터와 세계관을 보강하는가?
5. 기존 규칙과 데이터로 최소 구현 가능한가?
6. 장기 분량만 늘리고 핵심을 흐리지는 않는가?

권장 우선순위 표기:

- `P0`: 새 분량보다 먼저 해결해야 하는 핵심 경험 문제
- `P1`: P0 검증 뒤 추가할 보강 콘텐츠
- `P2`: 실제 플레이타임과 반복 피로가 확인된 뒤 검토할 확장

## 10. 추가하지 않는 판단

다음 경우에는 콘텐츠를 추가하지 않는 편이 낫다.

- 기존 콘텐츠와 핵심 동사가 같다.
- 숫자나 수집량만 늘어난다.
- 필수 판단을 장비·성장·자동화가 대신한다.
- 첫 10분의 정보량을 늘린다.
- 저장·경제·진행 구조를 목적 없이 복잡하게 만든다.
- 실패를 학습이 아니라 시간 손실로만 만든다.
- 플레이테스트 없이 분량 부족을 가정한다.

## 11. 사례 — 선택적 하이라이트를 정상 완주 경로와 분리하기

대표 데모에는 기억에 남는 상위 능력·특수 장면·반전·보너스 결말이 필요할 수 있다. 그러나 이를 반드시 획득해야 최종 구간에 진입하도록 만들면 기본 루프의 자립성을 검증하지 못한다.

권장 구조:

```text
공통 핵심 루프
→ 하이라이트 획득 여부
   ├─ 보유: 새 선택·연출·의미를 추가
   └─ 미보유: 기존 판단과 상태로 정상 진행
→ 두 경로 모두 완전한 결과와 다음 목표 제공
```

세 가지를 구분한다.

- **감정적 가치:** 획득 순간과 사용 순간이 제품의 상위 약속을 보여준다.
- **정상 경로:** 미획득 상태에서도 최종 구간의 진입·공략·결말이 완전해야 한다.
- **검증 지표:** 보유·미보유 완주율, 실패 원인, 실제 사용률, 핵심 루프 설명 가능성을 따로 기록한다.

하이라이트는 핵심 판단을 자동 정답으로 대체하지 않고 선택과 표현을 확장해야 한다. 해당 기능 자체가 제품의 핵심 조작이거나 반드시 전달해야 하는 사실이면 선택 보상으로 두지 않는다.

## 12. Codex 전달용 사양

구현 AI에게 전달할 문서는 다음을 포함한다.

1. 작업명
2. 목표와 해결할 문제
3. 의도한 플레이어 경험
4. 현재 확인된 구현 사실
5. 승인된 변경 사양
6. 미확정·가정 항목
7. 포함 범위
8. 제외 범위
9. 예상 영향 파일과 보호 경로
10. 저장·UI·데이터 위험
11. 완료 기준
12. 테스트 체크리스트
13. 최종 보고 형식

구현 완료 전에는 설계 문서에 `구현 완료`로 적지 않는다. 실제 코드·데이터·테스트와 충돌하면 차이를 보고하고 사용자 결정을 받는다.

## 13. 최종 검수 질문

- 핵심 재미를 한 문장으로 설명할 수 있는가?
- 첫 10분에 핵심 행동과 즉시 보상이 존재하는가?
- 선택지는 실제 정보·위험·접근 방식이 다른가?
- 실패가 다음 판단에 쓸 근거를 남기는가?
- 각 콘텐츠가 다른 핵심 동사를 가지는가?
- UI는 현재 판단에 필요한 정보부터 보여주는가?
- 새 시스템 없이 문구·배치·피드백으로 해결할 수 있는가?
- 추가하지 않는 편이 나은 항목을 명시했는가?
- 저장·경제·진행을 불필요하게 복잡하게 만들지 않았는가?
- 구현 사실, 승인 설계, 계획, 미확정을 구분했는가?

프로젝트별 작성에는 `templates/CONTENT_DESIGN_BRIEF.md`를 사용한다.
