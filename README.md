# Base

여러 프로젝트가 함께 사용하는 AI 협업 규칙의 원본 저장소입니다.

Base에는 특정 게임, 엔진, 세계관, 제품에 종속되지 않는 안정적인 작업 원칙과 템플릿만 둡니다. 각 프로젝트는 필요한 Base 문서를 저장소 내부에 **로컬 사본**으로 동기화하고, 프로젝트 전용 규칙은 해당 프로젝트 저장소에 별도로 둡니다.

## 목적

- 새 프로젝트와 새 AI가 같은 작업 기준으로 시작한다.
- 공용 규칙과 프로젝트 전용 규칙을 섞지 않는다.
- Base 원격 링크 변화가 진행 중인 프로젝트 규칙을 갑자기 바꾸지 않도록, 프로젝트별 기준 커밋과 로컬 사본을 남긴다.
- 반복 검증된 원칙만 Base로 승격한다.

## 공용 원본 문서

| 파일 | 역할 |
|---|---|
| `AGENTS.md` | 여러 프로젝트에 공통인 최소 AI 작업 원칙 |
| `docs/AI_SHARED_WORK_RULES.md` | 역할 분리, 범위 관리, 품질 원칙, Base 승격 운영 방식 |
| `docs/AI_WORKFLOW_RULES.md` | 공통 작업 순서와 Goal 작성 기준 |
| `docs/MVP_WORKFLOW_CHECKLIST.md` | 실제 작업 시작/종료 체크리스트 |
| `docs/BENCHMARKING_REFERENCE_GUIDE.md` | 벤치마킹 표 형식과 적용 기준 |
| `docs/DOCUMENTATION_MAP.md` | Base와 프로젝트 문서의 책임 경계 |
| `docs/CHANGELOG.md` | Base 규칙 변경 기록 |
| `templates/` | 새 프로젝트에 복사할 문서 템플릿 |

## 사용 방식

1. Base에서 재사용 가능한 공용 원칙을 관리한다.
2. 각 프로젝트는 필요한 문서를 저장소 내부에 로컬 사본으로 동기화한다.
3. 일상적인 프로젝트 작업에서는 **프로젝트의 로컬 사본을 먼저 읽는다.**
4. 프로젝트의 `docs/BASE_RULES_VERSION.md`에 기준 Base 커밋 SHA와 동기화 날짜를 기록한다.
5. Base 원격 저장소는 동기화, 기준 확인, 공용 규칙 개선 때만 직접 확인한다.
6. 프로젝트 전용 `AGENTS.md`, 엔진 규칙, Issue 요구사항이 공용 문서보다 구체적이면 그것을 우선한다.

## Base 승격 방식

새 규칙은 처음부터 Base에 자동 반영하지 않습니다.

1. 먼저 프로젝트 전용 규칙 또는 로컬 확장으로 적용한다.
2. 실제 작업에서 반복 사용되거나 여러 프로젝트에 재사용할 가치가 확인되면 Base 승격 후보로 분류한다.
3. 구현 작업자는 Base를 자동 수정하거나 자동 push하지 않는다.
4. 작업 완료 보고에서 `Base 승격 후보`를 별도 항목으로 알린다.
5. 사용자가 승인하면 Base 저장소에 공용 규칙을 별도 커밋으로 반영한다.
6. Base 커밋 SHA가 확정되면 프로젝트의 `docs/BASE_RULES_VERSION.md`를 후속 커밋으로 갱신한다.

## 경계

Base에는 엔진 코드, 게임 데이터, 특정 제품의 설계, 프로젝트별 테스트 결과를 두지 않습니다. 그런 내용은 각 프로젝트의 `AGENTS.md`, 전용 규칙, README, 데이터와 코드에 둡니다.

프로젝트가 늘어나 동기화 비용이 커질 때에만 스크립트나 GitHub Action을 추가합니다. 그 전에는 기준 커밋을 기록하는 명시적 동기화 방식을 사용합니다.
