## 항상 기록하는 원칙

- 모든 의미 있는 스킬 호출은 한 줄 이상의 실행 결과를 남긴다.
- 스킬 본문을 매번 무조건 수정하지 않는다.
- 변경할 근거가 없으면 `스킬 변경 없음`과 이유를 기록한다.
- 실패·부분 성공·미검증도 삭제하지 않는다.
- 프로젝트 고유 정보와 공용화 가능한 원리를 분리한다.
- 한 번의 성공은 관찰 또는 가설이며 반복 검증 전에는 강제 규칙으로 승격하지 않는다.

## 지식 상태 정의

- `관찰`: 한 번 발견한 현상
- `가설`: 추가 검증이 필요한 해결 방법
- `패턴`: 프로젝트 안에서 반복된 경향
- `검증`: 여러 조건에서 재현된 안정적 방법
- `승격 후보`: Base 공용 원리로 검토할 가치가 있음

## 검토 트리거

- 동일 실패가 두 번 이상 반복됨
- 사용자 피드백이 완료 기준과 다름
- 책임 본책·파일·도구·검증 경로가 변경됨
- 새 예외·폴백이 필요함
- 새 작업 유형에 대응할 스킬이 없음
- 여러 스킬이 같은 절차를 중복함
- 새 채팅이 필요한 스킬을 찾지 못함
- 90일 이상 사용됐지만 검토 기록이 없음

## Base 환류 검수

- [ ] 프로젝트 고유 세계관·수치·경로·승인 자산을 제거했다.
- [ ] 여러 조건 또는 프로젝트에서 재사용 가치가 있다.
- [ ] 기존 Base Method·Skill·Template·Case·Test와 중복을 확인했다.
- [ ] 적용 조건과 사용하지 않는 조건이 있다.
- [ ] 실패 조건과 검증 방법이 있다.
- [ ] 한 번의 성공을 검증된 공용 규칙으로 과장하지 않았다.
