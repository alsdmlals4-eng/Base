# 프로젝트 기획 문서 체계

> 목적: 새 채팅, 새 AI, 새 작업자가 과거 대화 없이도 기획서만으로 프로젝트 방향을 이해하고, 연결된 로드맵·스킬·실제 파일을 따라 작업을 재개하게 한다.

## 0. 콜드 스타트 계약

- 기획서는 프로젝트의 핵심 경험, 방향, 범위, 불변 조건과 금지 방향을 책임진다.
- 로드맵은 현재 단계, 우선순위, 선행 조건, 다음 작업, 종료 기준과 검증을 책임진다.
- Documentation Map은 질문별 현행 책임 원본과 최초 읽기 순서를 책임진다.
- Active Context·Handoff는 현재 상태와 최근 결정, 다음 실행 순서를 연결한다.
- Base skill과 프로젝트 skill extension은 반복 작업의 입력, 절차, 산출물, 검증과 실패 기준을 책임진다.
- 실제 구현 상태, 데이터와 파일 경로는 해당 기술·데이터 책임 문서와 실제 파일이 책임진다.

방향, 수치, 용어, 범위, 구현 상태, 우선순위 또는 작업 절차가 바뀌면 같은 작업 안에서 영향받는 책임 원본을 갱신한다.

## 5. 로드맵 연결

| 단계 | 사용자 가치 | 선행 조건 | 핵심 산출물 | 관련 기획서·스킬 | 종료 기준 | 검증 | 상태 |
|---|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |  |

로드맵 규칙:

- 각 단계는 기능 목록이 아니라 플레이어 가치 또는 검증할 가설을 가진다.
- 현재 Issue·Goal은 반드시 로드맵의 단계와 연결한다.
- 선행 조건이 충족되지 않은 작업은 다음 우선순위로 올리지 않는다.
- 종료 기준은 `조건 → 행동 → 관찰 결과`로 작성한다.
- 완료된 단계도 구현과 검증을 구분한다.
- 우선순위나 범위가 바뀌면 같은 작업 안에서 로드맵과 Active Context를 갱신한다.

|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |

- Base skill 전체를 복제하지 않는다.
- 프로젝트 추가·대체 규칙, 실제 경로, 데이터 계약, 완료 기준과 테스트만 extension에 기록한다.
- 작업 절차가 바뀌면 프로젝트 extension을 갱신하고, 공용화 가능한 차이는 Base 환류 대상으로 분리한다.

## 8. 작업 중 갱신 매트릭스

| 변경 내용 | 필수 갱신 대상 |
|---|---|
| 프로젝트 방향·플레이어 경험 | 방향서·전체 기획서·Active Context |
| 시스템 규칙·수치·용어 | 분야별 기획서·데이터 문서·Issue |
| 우선순위·선행 조건 | Roadmap·Active Context |
| 반복 작업 절차·검증 | Project skill extension·필요 시 Base skill |
| 구현·테스트 상태 | Active Context·Handoff·테스트 기록 |
| 책임 문서·파일 경로 | Documentation Map·README·연결 문서 |
| 공용 교훈·실패 방지 | Base method·skill·template·case |
| 최신 승인 이미지 교체 | 아트 본책·시각자료 인덱스·해당 화면의 게임 기획·QA |
| BGM·SFX·음성 변경 | 사운드 본책·이벤트 계약·QA·PM |

작업 종료 시 관련 본책을 갱신한다. 갱신할 내용이 없으면 PR에 `문서 영향 없음`과 근거를 기록한다.

### 최신 이미지 계약

- 최신 승인 이미지는 버전 접미사가 없는 안정 경로 하나를 사용한다.
- 이전 이미지는 Git 이력으로 보존하고, 비교 보존 가치가 있을 때만 명시적 아카이브로 이동한다.
- 이미지 인덱스는 상태, 출처, 저장 경로, 참고할 것, 참고하지 않을 것, 현재 결정과 다른 것을 기록한다.
- 이미지가 암시하는 시스템은 게임 기획과 데이터 책임 문서를 덮어쓰지 않는다.

## 9. 문서 수명주기

- 한 질문에 현행 원본 하나를 유지한다.
- 새 결정을 기존 원본에 반영한다.
- 중복 문서는 참조 확인 후 통합·삭제한다.
- 과거 상태는 Git 이력으로 보존한다.
- 경로 변경 시 README·Documentation Map·Issue·Goal·Plan을 갱신한다.
- Active Context에 전문을 복제하지 않고 책임 원본을 연결한다.
- 작업 완료 전에 로드맵·기획서·스킬·Active Context의 불일치를 검색한다.

## 10. 콜드 스타트 검수

새 작업자가 10분 안에 다음 질문에 답할 수 있어야 한다.

1. 핵심 플레이어 경험과 프로젝트 차별점은 무엇인가?
2. 현재 단계와 최우선 작업은 무엇인가?
3. 다음 작업의 선행 조건과 종료 기준은 무엇인가?
4. 변경하면 안 되는 방향과 범위는 무엇인가?
5. 각 주제의 현행 책임 문서는 어디인가?
6. 어떤 Base skill과 프로젝트 extension을 사용하는가?
7. 실제 구현·데이터·테스트는 어디에서 확인하는가?
8. 무엇이 승인, 구현, 검증, 미확정 상태인가?
9. 작업 종료 시 어떤 공용 학습 데이터를 Base로 환류해야 하는가?

답하지 못하면 기획서의 맥락, 책임 문서 링크, 로드맵, 스킬 연결 또는 상태 표기를 보강한다.
