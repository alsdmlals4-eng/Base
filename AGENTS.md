# Base 공용 AI 작업 규칙

이 파일은 여러 프로젝트에 복사할 수 있는 공용 AI 작업 원칙이다. 프로젝트 저장소에서는 엔진, 폴더 구조, 실행 방법, 금지 범위, 검증 방법을 프로젝트 전용 문서로 추가한다.

## 적용 범위와 우선순위

- 최신 사용자 지시가 가장 우선한다.
- 프로젝트의 `AGENTS.md`, 보안 규칙, 엔진 규칙, 현재 Issue가 더 구체적이면 이 문서보다 우선한다.
- Base는 공용 원칙의 원본이다. 실제 작업에서는 프로젝트에 동기화된 로컬 사본을 먼저 읽고, Base 원격 저장소는 동기화와 기준 확인에만 사용한다.
- 충돌하는 규칙이나 실제 파일 상태는 추정하지 말고 확인한 뒤 보고한다.

## 역할 분리

- 기획/조정 AI: 사용자 의도, 사용자 가치, 벤치마킹, Issue, Goal, 검수 기준을 정리한다.
- 구현 AI: 승인된 Issue 또는 Goal을 기준으로 실제 파일을 읽고, 수정하고, 검증한다.
- 사용자: 방향을 결정하고 실제 실행 결과를 확인한다.

구현 AI는 기획을 임의로 확장하지 않고, 기획 또는 문서 담당은 구현 완료를 추정하지 않는다.

## 기본 원칙

- 실제 파일과 현재 작업 상태를 확인하기 전에는 구현 상태를 가정하지 않는다.
- 정상 동작 중인 사용자 변경사항을 되돌리거나 덮어쓰지 않는다.
- 승인 없이 요청 범위 밖 리팩터링이나 기능 추가를 하지 않는다.
- 완료 기준은 기능명이 아니라 관찰 가능한 상태로 작성한다.
- 테스트하지 못한 항목은 테스트하지 못했다고 명확히 보고한다.
- 문서에는 현재 구현되었거나 실제로 확인한 사실만 기록한다.

## 최소 작업 순서

1. 사용자 최신 지시와 프로젝트의 `AGENTS.md`를 확인한다.
2. 프로젝트의 Base 버전 기록과 로컬 공용 사본을 읽는다.
3. 프로젝트 전용 규칙, 현재 Issue 또는 Goal, 관련 실제 파일을 확인한다.
4. 수정 대상과 검증 계획을 짧게 공유한다.
5. 가장 작은 변경으로 요구사항을 구현한다.
6. 포맷, 테스트, 실행 가능한 동작 검증을 수행한다.
7. 변경 파일, 구현 내용, 검증 결과, 사용자 확인 항목, 남은 위험을 보고한다.

## 작업 경로

### 빠른 경로

오탈자, 단순 문서, 이미 결정된 단일 데이터 수정:

```text
최신 지시 확인 → 실제 파일 확인 → 범위 확인 → 수정 → 최소 검증 → 보고
```

### 기능 경로

새 시스템, 여러 파일, 저장/불러오기, UI/UX, 게임 루프 변경:

```text
의도 분석 → 기준 문서/Issue 확인 → 준비 상태 확인 → 필요한 근거 조사 → 범위·위험 확정 → Goal → 구현 → 검증 → 회고
```
