# Base 공용 AI 작업 규칙

Base는 여러 프로젝트가 공유하는 **[학습형] [공용] 데이터 원본**이다. 프로젝트에서 반복해 얻은 방법·실패·검증 결과를 규칙, method, research, skill, template, case로 누적하고 갱신한다.

각 프로젝트 저장소는 Base를 복제해 별도 공용 원본으로 만드는 곳이 아니다. Base 공용 지식을 프로젝트의 세계관, 수치, 엔진, 파일 경로와 현재 구현 상태에 맞게 **분화·적용·검증하는 전용 작업 공간**이다.

## 1. 우선순위

충돌 시 다음 순서를 따른다.

1. 사용자의 최신 지시
2. 프로젝트 `AGENTS.md`와 보안·엔진 규칙
3. 프로젝트 `Active Context` 또는 Handoff 기준 문서
4. 승인된 기획서, 현재 Issue·Goal·Plan
5. 프로젝트에 동기화된 Base 로컬 사본
6. Base 원격 원본
7. 과거 대화와 추정

실제 파일과 실행 결과가 문서와 다르면 차이를 보고한다. 정상 동작 중인 사용자 변경을 임의로 되돌리지 않는다.

## 2. 역할

- 기획·조정 AI: 의도, 플레이어 가치, 대안, 기획서, Issue·Goal, 검수 기준을 정리한다.
- 구현 AI: 승인된 범위에서 실제 파일을 수정하고 검증한다.
- 사용자: 방향과 우선순위를 결정하고 실제 결과를 확인한다.

기획 담당은 구현 완료를 추정하지 않고, 구현 담당은 제품 방향을 임의로 확장하지 않는다.

## 3. 작업 전 공용·전용 컨텍스트 확인

작업을 시작하기 전에 **현재 작업에 적용되는 Base 공용 기준 전체**와 **프로젝트의 현행 전용 기준 전체**를 함께 확인한다.

여기서 `전체`는 저장소의 모든 파일을 무조건 읽는다는 뜻이 아니다. `DOCUMENTATION_MAP.md`와 참조 관계를 이용해 다음 두 집합에서 현재 작업에 적용되는 현행 책임 원본과 영향 파일을 빠짐없이 확인한다.

- 공용 집합: Base 로컬 사본의 작업 규칙, 관련 method·research·skill·template·case
- 전용 집합: 프로젝트 `AGENTS.md`, Active Context·Handoff, 기획서, 로드맵, Issue·Goal·Plan, 실제 대상·연결 파일

권장 읽기 순서:

```text
최신 사용자 지시
→ 프로젝트 AGENTS.md
→ BASE_RULES_VERSION.md와 Base 로컬 사본
→ 프로젝트 Documentation Map
→ Active Context·Handoff
→ 프로젝트 방향과 관련 분야 책임 문서
→ 현재 Issue·Goal·Plan
→ 실제 수정 대상과 연결 파일
```

- `archive`, `[백업]`, `hold`, `[보류]`, `deferred`, `parking-lot`은 사용자가 재개를 요청하지 않는 한 기본 읽기·구현 대상에서 제외한다.
- Active Context가 실제 파일이나 분야별 기준 문서와 충돌하면 차이를 보고하고 같은 작업 안에서 정리한다.
- 과거 대화만을 프로젝트의 유일한 현재 상태로 사용하지 않는다.

## 4. 요청을 작업 프롬프트로 변환

짧은 요청을 곧바로 구현 지시로 간주하지 않는다. 작업 영향에 따라 다음처럼 처리한다.

- `L0`: 오탈자·명확한 형식 변경 — 바로 처리
- `L1`: 범위가 명확한 작은 수정 — 범위와 검증을 보완해 처리
- `L2`: 선택지가 있거나 시스템에 영향 — 브레인스토밍 후 사양 확정
- `L3`: 핵심 구조·여러 시스템·장기 방향 — 기획서와 실행 계획 작성
- `L4`: 여러 프로젝트에서 재사용할 방법 — 프로젝트 검증 후 Base 학습 데이터로 반영

실행 프롬프트는 다음 8요소를 포함한다.

```text
목적 → 맥락 → 경험 → 범위 → 제약 → 산출물 → 완료 기준 → 검증
```

- 실행 스킬: `skills/transforming-requests-into-prompts/SKILL.md`
- 템플릿: `templates/EXECUTABLE_PROMPT.md`

## 5. 기본 작업 흐름

```text
의도 확인
→ 공용 기준과 프로젝트 기준 확인
→ 대안 비교와 사양 확정
→ 실행 프롬프트·기획서·Goal
→ 가장 작은 안전한 변경
→ 테스트·검수
→ 프로젝트 기준 문서와 Active Context 최신화
→ 공용 학습 데이터 추출·사례 기록·Base 반영
```

이미 확정된 내용을 다시 질문하지 않는다. 불확실한 내용은 `확인 필요`, 제안은 `권장`, 범위 밖 내용은 `제외` 또는 `보류`로 구분한다.

## 6. 파일과 문서 수명주기

- 한 주제에는 현행 책임 문서 하나만 둔다.
- 확정된 변경은 기존 기준 문서를 갱신한다.
- `v2`, `final`, `latest`, 날짜별 활성 복제본을 만들지 않는다.
- 이전 상태는 기본적으로 Git 이력으로 보존한다.
- 통합이 끝난 중복·구버전 파일은 참조를 확인한 뒤 삭제한다.
- 생성·삭제·이동·이름 변경 시 연결 문서, 코드, 템플릿, 로컬 사본의 후속 동기화를 확인한다.
- Active Context는 분야별 기준 문서의 대체본이 아니라 현재 방향·상태·미확정 범위·다음 작업을 연결하는 인수 문서다.

## 7. 작업 종료·인수인계와 학습 환류

기획, 문서, 구현, 검수 작업은 다음 정비가 끝나야 완료로 보고할 수 있다.

1. 확정된 프로젝트 결정, 수치, 구현 상태를 프로젝트의 현행 기획서·테스트·로드맵에 반영한다.
2. 프로젝트 Active Context 또는 Handoff를 최신화한다.
3. 결과를 `프로젝트 전용 데이터`와 `공용으로 재사용 가능한 학습 데이터`로 분리한다.
4. 공용 데이터는 기존 Base 책임 문서와 중복·충돌을 확인한 뒤 다음 위치에 반영한다.
   - 판단법·기획법: `docs/knowledge/methods/`
   - 조사·근거: `docs/knowledge/research/`
   - 실행 절차: `skills/`
   - 역량·검수 지도: `docs/knowledge/skills/`
   - 복사 양식: `templates/`
5. 작업에서 얻은 문제, 선택, 결과, 실패, 미검증 항목을 `docs/knowledge/cases/` 형식으로 기록한다.
6. 한 번의 성공이나 미검증 가설은 확정 규칙·스킬로 승격하지 않고 먼저 `관찰`·`가설` 사례로 남긴다.
7. 문서 지도, README, Issue·Goal·Plan과 참조 경로를 확인한다.
8. 폐기된 표현·수치·구조를 검색하고 중복·충돌·낡은 컨텍스트를 제거한다.
9. Base를 변경했다면 프로젝트 로컬 사본과 `BASE_RULES_VERSION.md`의 후속 동기화를 기록한다.
10. 실제 구현·테스트 상태와 문서가 일치하는지 확인하고, 검증하지 못한 상태는 완료로 쓰지 않는다.

단순 대화라도 제품 방향, 수치, 용어, 범위, 완료 기준 또는 재사용 가능한 작업 방법이 확정되었다면 문서 정비와 학습 환류 대상이다. 다음 작업자는 과거 대화 없이 프로젝트 저장소와 Base만으로 작업을 재개할 수 있어야 한다.

## 8. 리팩터링

리팩터링은 외부 동작을 유지하면서 구조를 개선하는 작업이다.

- 기능 추가와 구조 변경을 가능한 한 분리한다.
- 현재 동작과 회귀 검증 방법을 먼저 확보한다.
- 긴 함수, 중복, 복잡한 조건, 책임 혼합을 작은 단위로 개선한다.
- 저장 형식, 공개 인터페이스, 게임 규칙과 사용자 흐름은 별도 승인 없이 바꾸지 않는다.
- 각 단계에서 테스트하고 되돌릴 수 있는 변경 단위를 유지한다.

## 9. Base와 프로젝트 경계

### Base — [학습형] [공용]

- 여러 프로젝트에서 재사용 가능한 규칙, method, research, skill, template
- 프로젝트 사례에서 일반화한 성공·실패·검증 교훈
- 검증 상태와 적용 조건이 기록된 체크리스트와 사례
- 프로젝트가 분화해 사용할 공용 원본

### 프로젝트 — [전용] [분화·검증]

- 세계관, 캐릭터, 밸런스, 실제 수치, 파일 경로
- 현재 구현 상태, 로드맵, Issue·Goal, 테스트 결과
- Base 스킬을 프로젝트 상황에 맞게 확장한 전용 규칙
- 실제 사용한 모델·자산·프롬프트·실행 결과
- 공용화 전의 관찰·가설·실험 기록

프로젝트의 구체 해답을 Base에 복사하지 않는다. 프로젝트에서 확인한 문제 해결 원리, 적용 조건, 실패 조건과 검증 방법을 일반화해 반영한다. 상세 절차는 `skills/promoting-project-knowledge/SKILL.md`를 따른다.

## 10. 로컬과 GitHub

로컬 저장소와 GitHub는 자동 양방향 동기화가 아니다.

- 작업 전 원격 변경과 로컬 상태를 확인한다.
- 로컬 변경은 검증 후 commit·push해야 원격에 반영된다.
- 원격 변경은 fetch·pull해야 로컬에 반영된다.
- 자동화는 테스트와 상태 경고에 사용하고, 미검증 변경의 자동 push·충돌 자동 해결은 피한다.

## 11. 완료 조건

완료 보고에는 다음을 포함한다.

- 변경 파일과 이유
- 유지한 기존 동작
- 실행한 검증과 결과
- 미검증 항목과 남은 위험
- Active Context와 연결 문서의 최신화 결과
- 제거한 낡은 컨텍스트와 중복
- 프로젝트 전용으로 남긴 내용
- Base 기획서·스킬·템플릿에 반영한 공용 학습 데이터
- 작성·갱신한 사례와 지식 상태
- 후속 동기화가 필요한 문서
- 다음 작업자가 먼저 확인할 항목

실행하지 않은 테스트나 확인하지 않은 구현을 완료로 보고하지 않는다.
