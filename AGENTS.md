# Base 공용 AI 작업 규칙

Base는 여러 프로젝트가 공유하는 작업 원칙과 재사용 가능한 지식의 원본이다. 프로젝트별 세계관, 수치, 엔진 경로, 구현 상태는 각 프로젝트 저장소에서 관리한다.

## 1. 우선순위

충돌 시 다음 순서를 따른다.

1. 사용자의 최신 지시
2. 프로젝트 `AGENTS.md`와 보안·엔진 규칙
3. 프로젝트 `Active Context` 또는 Handoff 기준 문서
4. 승인된 기획서, 현재 Issue·Goal·Plan
5. 프로젝트에 동기화된 Base 로컬 사본
6. Base 원격 원본
7. 과거 대화와 추정

실제 파일과 실행 결과가 문서와 다르면 차이를 보고한다. 정상 동작 중인 사용자 변경을 임의로 되돌리지 않는다.

## 2. 역할

- 기획·조정 AI: 의도, 플레이어 가치, 대안, 기획서, Issue·Goal, 검수 기준을 정리한다.
- 구현 AI: 승인된 범위에서 실제 파일을 수정하고 검증한다.
- 사용자: 방향과 우선순위를 결정하고 실제 결과를 확인한다.

기획 담당은 구현 완료를 추정하지 않고, 구현 담당은 제품 방향을 임의로 확장하지 않는다.

## 3. 작업 전 컨텍스트

저장소 전체를 무조건 읽지 않는다. `DOCUMENTATION_MAP.md`와 참조 관계를 이용해 현재 작업에 필요한 기준 문서와 영향 파일만 확인한다.

권장 읽기 순서:

```text
최신 사용자 지시
→ 프로젝트 AGENTS.md
→ Active Context·Handoff
→ BASE_RULES_VERSION.md와 Base 로컬 사본
→ 현재 Issue·Goal·Plan
→ 수정 대상과 연결 파일
```

- `archive`, `[백업]`, `hold`, `[보류]`, `deferred`, `parking-lot`은 사용자가 재개를 요청하지 않는 한 기본 읽기·구현 대상에서 제외한다.
- Active Context가 실제 파일이나 분야별 기준 문서와 충돌하면 차이를 보고하고 같은 작업 안에서 정리한다.
- 과거 대화만을 프로젝트의 유일한 현재 상태로 사용하지 않는다.

## 4. 요청을 작업 프롬프트로 변환

짧은 요청을 곧바로 구현 지시로 간주하지 않는다. 작업 영향에 따라 다음처럼 처리한다.

- `L0`: 오탈자·명확한 형식 변경 — 바로 처리
- `L1`: 범위가 명확한 작은 수정 — 범위와 검증을 보완해 처리
- `L2`: 선택지가 있거나 시스템에 영향 — 브레인스토밍 후 사양 확정
- `L3`: 핵심 구조·여러 시스템·장기 방향 — 기획서와 실행 계획 작성
- `L4`: 여러 프로젝트에서 재사용할 방법 — 프로젝트 검증 후 Base 승격

실행 프롬프트는 다음 8요소를 포함한다.

```text
목적 → 맥락 → 경험 → 범위 → 제약 → 산출물 → 완료 기준 → 검증
```

- 실행 스킬: `skills/transforming-requests-into-prompts/SKILL.md`
- 템플릿: `templates/EXECUTABLE_PROMPT.md`

## 5. 기본 작업 흐름

```text
의도 확인
→ 필요한 컨텍스트 확인
→ 대안 비교와 사양 확정
→ 실행 프롬프트·기획서·Goal
→ 가장 작은 안전한 변경
→ 테스트·검수
→ 기준 문서와 Active Context 최신화
→ 프로젝트 전용 내용과 Base 공용 지식 분리
```

이미 확정된 내용을 다시 질문하지 않는다. 불확실한 내용은 `확인 필요`, 제안은 `권장`, 범위 밖 내용은 `제외` 또는 `보류`로 구분한다.

## 6. 파일과 문서 수명주기

- 한 주제에는 현행 책임 문서 하나만 둔다.
- 확정된 변경은 기존 기준 문서를 갱신한다.
- `v2`, `final`, `latest`, 날짜별 활성 복제본을 만들지 않는다.
- 이전 상태는 기본적으로 Git 이력으로 보존한다.
- 통합이 끝난 중복·구버전 파일은 참조를 확인한 뒤 삭제한다.
- 생성·삭제·이동·이름 변경 시 연결 문서, 코드, 템플릿, 로컬 사본의 후속 동기화를 확인한다.
- Active Context는 분야별 기준 문서의 대체본이 아니라 현재 방향·상태·미확정 범위·다음 작업을 연결하는 인수 문서다.

## 7. 작업 종료 컨텍스트 정비

기획, 문서, 구현, 검수 작업은 다음 정비가 끝나야 완료로 보고할 수 있다.

1. 이번 작업에서 확정된 결정과 변경 수치를 관련 기준 문서에 반영한다.
2. 프로젝트의 Active Context 또는 Handoff 문서를 최신화한다.
3. 문서 지도, README, 테스트, Issue·Goal·Plan의 연결 영향을 확인한다.
4. 폐기된 표현·수치·구조를 검색하고 중복·충돌·낡은 컨텍스트를 제거한다.
5. 보류 항목을 활성 구현과 분리하고 재개 조건을 명시한다.
6. 실제 구현·테스트 상태와 문서가 일치하는지 확인한다.
7. 검증하지 못한 상태는 완료로 쓰지 않는다.
8. 프로젝트 전용 지식과 Base 승격 대상을 구분한다.
9. GitHub 또는 사용 중인 기준 저장소에 반영한 결과와 남은 위험을 보고한다.

단순 대화라도 제품 방향, 수치, 용어, 범위 또는 완료 기준이 확정되었다면 문서 정비 대상이다. 작업자가 바뀌어도 과거 대화 없이 저장소만으로 현재 작업을 재개할 수 있어야 한다.

## 8. 리팩터링

리팩터링은 외부 동작을 유지하면서 구조를 개선하는 작업이다.

- 기능 추가와 구조 변경을 가능한 한 분리한다.
- 현재 동작과 회귀 검증 방법을 먼저 확보한다.
- 긴 함수, 중복, 복잡한 조건, 책임 혼합을 작은 단위로 개선한다.
- 저장 형식, 공개 인터페이스, 게임 규칙과 사용자 흐름은 별도 승인 없이 바꾸지 않는다.
- 각 단계에서 테스트하고 되돌릴 수 있는 변경 단위를 유지한다.

## 9. Base와 프로젝트 경계

Base에 둔다:

- 여러 프로젝트에서 재사용 가능한 규칙, method, research, skill, template.
- 검증된 체크리스트와 일반화된 사례.

프로젝트에 둔다:

- 세계관, 캐릭터, 밸런스, 실제 수치, 파일 경로.
- 현재 구현 상태, 로드맵, Issue·Goal, 테스트 결과.
- Base 스킬을 확장하는 프로젝트 전용 규칙.

승격 절차는 `skills/promoting-project-knowledge/SKILL.md`를 따른다.

## 10. 로컬과 GitHub

로컬 저장소와 GitHub는 자동 양방향 동기화가 아니다.

- 작업 전 원격 변경과 로컬 상태를 확인한다.
- 로컬 변경은 검증 후 commit·push해야 원격에 반영된다.
- 원격 변경은 fetch·pull해야 로컬에 반영된다.
- 자동화는 테스트와 상태 경고에 사용하고, 미검증 변경의 자동 push·충돌 자동 해결은 피한다.

## 11. 완료 조건

완료 보고에는 다음을 포함한다.

- 변경 파일과 이유.
- 유지한 기존 동작.
- 실행한 검증과 결과.
- 미검증 항목과 남은 위험.
- Active Context와 연결 문서의 최신화 결과.
- 제거한 낡은 컨텍스트와 중복.
- 프로젝트 전용으로 남긴 내용.
- Base에 반영한 공용 내용.
- 후속 동기화가 필요한 문서.
- 다음 작업자가 먼저 확인할 항목.

실행하지 않은 테스트나 확인하지 않은 구현을 완료로 보고하지 않는다.
